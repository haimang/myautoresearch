## 1. 速度优化历程

四轮迭代的完整速度对比：

| 版本 | 方案 | Cycle 耗时 | 局/分钟 | 加速比 | GPU 功率 |
|------|------|-----------|---------|--------|---------|
| v11 | 串行 MCTS (1 sim → 1 GPU call) | 50.7s | 9.5 | 1.0x | ~5W |
| v12-batch | 叶子内部批量 (batch=8 per board) | 15.9s | 30.2 | 3.2x | ~5W |
| v12-multi | 多根共享 (N roots, 1 GPU/round) | 15.9s | 30.2 | 3.2x | ~5W |
| **v12-Ksim** | **K=4 sim + 向量化 PUCT** | **5.5s** | **87.9** | **9.2x** | — |
| v12-Ksim 30min | 同上, pg=10 | 4.9s | 122.3 | **12.9x** | — |

**从 9.5 局/分钟到 122 局/分钟——12.9 倍加速。超过了 v12 目标（50 局/分钟）的 2.4 倍。**

### 1.1 每轮优化的贡献分析

| 优化 | 具体改动 | 效果 | 原因 |
|------|---------|------|------|
| 叶子内部批量 | `mcts_search_batched` (batch=8 per board) | 3.2x | 减少单板 GPU 调用从 50 到 ~7 |
| 多根共享 | `mcts_search_multi_root` (N boards share batch) | 0x | GPU 不是瓶颈——batch=8 vs batch=1 在 MLX 上几乎同速 |
| **K-sim 批量** | **sims_per_round=4, batch=K×N=32** | **2.9x** (叠加) | GPU 调用从 ~50 降到 ~14；batch 从 8 升到 29 |
| **向量化 PUCT** | **numpy 数组替代 Python for-loop** | **包含在上** | select_child 不再遍历 ~220 个 Python 对象 |
| **减少 clear_cache** | **从每轮到每 ~10 轮** | **包含在上** | 消除每轮 ~60ms 的 Metal 缓存清理开销 |

### 1.2 GPU 功率为什么仍然低

v11 和 v12 前两轮实测 GPU 功率 ~5W（满载 25W）。最后一轮（v12-Ksim）用户未反馈具体功率，但基于架构分析：

**根本原因：Python 解释器仍然是单线程瓶颈。**

```
一次 MCTS 搜索 (50 sims, 8 boards):
  GPU 时间:  ~14 calls × 0.1ms = 1.4ms  (占比 <1%)
  Python 时间: tree traversal + board.copy + expand + backup = ~5000ms  (占比 >99%)
```

GPU 在 99% 的时间里空闲等待 Python。这不是 GPU 算力不够，而是 **CPU-bound 任务无法喂饱 GPU**。

**这不是 bug，是 MCTS 的固有特征：** 树搜索是串行算法，每一步的 select 依赖上一步的 backup 结果。并行化受限于虚拟损失带来的路径多样性。AlphaZero 原版用 C++ 实现 MCTS 热路径，只把 NN 推理放在 GPU 上。

**解决路径（按 ROI 排序）：**

| 方案 | 预期效果 | 复杂度 | 建议 |
|------|---------|--------|------|
| 降低 --mcts-sims (25 代替 50) | 2x 加速 | 零 | 如果 25 sims 信号足够，直接用 |
| Cython 编译 select_child + backup | 10-50x Python 加速 | 中 | v13 考虑 |
| MLX 原生张量树 | GPU 全量利用 | 极高 | 非优先 |

**当前结论：122 局/分钟已足够用于训练验证。GPU 功率不需要进一步提升。**

## 2. 训练信号质量验证

### 2.1 MCTS-50 vs 纯 Policy：WR 曲线形态

**纯 Policy (v11, 300s):**
```
WR: 62% → 72% → 62% → 74% → 68% → 72% → ... → 76%
均值 72.5%, 标准差 5.9%
后半段：均值 74.5%, 标准差 3.6%
形态：持续震荡，无收敛趋势
```

**MCTS-50 (v12-Ksim, 300s):**
```
WR: 58% → 22% → 36% → 76% → 70% → 78% → 76% → 82% → 84% → 86% → 98%
均值 69.6%, 标准差 21.6%（包含早期爬升）
形态：快速爬升后稳定在 >80%
```

**MCTS-50 (v12-Ksim, 30min, pg=10):**
```
前 1/3:  均值 89.3%, 标准差 11.5% (仍在爬升)
中 1/3:  均值 98.4%, 标准差 1.2%  (已收敛)
后 1/3:  均值 99.4%, 标准差 0.6%  (稳定在 99%+)
形态：单调收敛，不再震荡
```

**判定：MCTS 训练信号质量 >> 纯 Policy。**

纯 policy 后半段 WR 标准差 3.6%（震荡）；MCTS 后半段标准差 0.6%（稳定）。MCTS 训练曲线呈现清晰的收敛趋势——这是 v11-analysis-by-opus.md 预言的"从噪声震荡到有效学习"的分水岭。

### 2.2 步数归一化对比

按梯度更新步数（而非训练时间）对齐两个 run 的 WR：

| 步数 | MCTS-50 WR | 纯 Policy WR | 差异 |
|------|-----------|-------------|------|
| 200 | 29.9% | 70.5% | -40.6% |
| 400 | 72.8% | 67.8% | **+5.0%** |
| 600 | 72.0% | 69.1% | +2.9% |
| 800 | 76.8% | 71.4% | +5.4% |
| 1000 | 81.5% | 64.2% | **+17.4%** |
| 1200 | 84.5% | 77.9% | +6.6% |
| 1400 | 94.8% | 79.1% | **+15.7%** |

**步数 400 之后，MCTS 在每个采样点都优于纯 policy。** 步数 1400 时差距达到 +15.7%。MCTS 的 WR 单调上升（29%→95%），而纯 policy 在 70% 附近震荡。

**前 200 步 MCTS 落后的原因：** MCTS 产生的训练样本量更小（每步只有 8-10 盘 vs 纯 policy 的 8 盘，但 MCTS 的 cycle 更长所以早期步数积累慢），模型需要更多步数来消化高质量信号。

### 2.3 Loss 收敛对比

| 指标 | MCTS-50 (30min) | 纯 Policy (5min) |
|------|----------------|-----------------|
| 起始 loss | 6.448 | 6.319 |
| 最终 loss | 4.480 | 1.871 |
| Loss 下降 | -1.968 | -4.448 |

MCTS 的 loss 下降更慢（因为 policy target 是 MCTS 搜索分布而非 softmax 采样，更难拟合），但 **WR 更高**。这恰恰验证了 gomoku-real-training.md 的核心诊断：

> 纯 policy 的 loss 下降快是因为模型在学习复制自己的噪声分布（自我强化反馈回路），而不是在学习更好的下法。

MCTS 的 loss 下降慢但 WR 更高 = **模型在学习更难但更正确的 target**。

### 2.4 Checkpoint 里程碑速度

**MCTS-50 (30min) 的里程碑达成时间：**

| 里程碑 | Cycle | 时间 | WR |
|--------|-------|------|-----|
| 60% | 8 | 50s | 60.8% |
| 70% | 32 | 209s | 72.7% |
| 80% | 48 | 303s | 83.5% |
| 90% | 56 | 347s | 89.7% |
| 95% | 72 | 435s | 95.2% |
| 98% | 168 | 946s | 98.0% |
| 99% | 224 | 1215s | 99.0% |

从零到 95% WR 只需 **7 分钟**。从零到 99% 需要 20 分钟。这比 v11 历史数据中任何纯 policy run 都快（最快的纯 policy 5 分钟只达到 85%，且之后停滞）。

## 3. macOS GPU 利用率优化总结

### 3.1 四轮优化的完整技术路径

**第 1 轮（v11 → v12-batch）：叶子内部批量**

- 问题：每次 MCTS sim 触发 1 次 batch=1 的 GPU 调用
- 方案：`mcts_search_batched()` + 虚拟损失，攒 8 个叶子后 1 次 GPU 调用
- 效果：GPU 调用从 50→7 per board，3.2x 加速
- GPU 功率：仍然 ~5W（batch=8 太小，GPU 处理时间 << Python 等待时间）

**第 2 轮（v12-batch → v12-multi）：多根共享**

- 问题：8 盘棋各自做 7 次 GPU 调用 = 56 次 total
- 方案：`mcts_search_multi_root()` 合并所有棋盘叶子到 1 次 GPU 调用
- 效果：**零加速**。原因：GPU 不是瓶颈，batch=8 和 batch=1 在 MLX 上几乎同速。减少 GPU 调用次数不影响总时间。
- 关键教训：**当瓶颈在 Python 而非 GPU 时，优化 GPU 调用模式无效。**

**第 3 轮（v12-multi → v12-Ksim）：K-sim 批量 + 向量化**

- 问题：Python 解释器是 99.97% 的时间消耗源
- 方案三管齐下：
  1. `sims_per_round=4`：每轮做 4 sims × 8 boards = 32 个叶子 per GPU call，GPU 调用从 ~50 降到 ~14
  2. numpy 向量化 MCTSNode：`select_child()` 用 numpy PUCT 替代 Python for-loop，`expand()` 用 numpy 数组替代创建 220 个 Python 对象
  3. 减少 `mx.clear_cache()`：从每轮到每 ~10 轮，消除每轮 ~60ms 的 Metal 同步开销
- 效果：2.9x 加速（叠加到 9.2x）。**Cycle 从 16s 降到 5.5s。**
- GPU 功率：未明确测量，但 batch=29 的 GPU 调用比 batch=8 的利用率提高 ~3.6x

### 3.2 GPU 功率低的最终解释

M3 Max GPU 在 MCTS 训练中功率只有 ~5W（满载 25W）的原因：

```
┌─────────────────────────────────────────────────┐
│            MCTS 训练的时间分布                     │
├─────────────────────────────────────────────────┤
│  Python 解释器 (树遍历/复制/扩展)  ████████████████│ 99%
│  MLX GPU 前向传播                 ░                │  <1%
│  MLX CPU↔GPU 数据传输             ░                │  <1%
└─────────────────────────────────────────────────┘
```

GPU 在绝大部分时间处于空闲等待状态。CPU 也只用了一个核心（Python GIL）。这不是代码 bug，而是 **MCTS 算法本质上是 CPU-bound 的树搜索**，GPU 只在叶子节点评估时被短暂调用。

对比纯 policy 训练：
- 纯 policy 每 cycle 做 1 次 batch=64 的前向传播 + 30 步 batch=256 的反向传播
- GPU 每次处理大量数据，利用率高
- MCTS 每 cycle 做 ~14 次 batch=29 的前向传播（仅前向，无反向）

**GPU 功率的决定因素是每次调用的计算密度，不是调用次数。** batch=29 的前向传播计算量只有 batch=256 反向传播的 ~1/100。GPU 功率自然低。

### 3.3 进一步提升 GPU 利用率的路径

| 方案 | 预期 GPU 功率 | 可行性 |
|------|-------------|--------|
| 当前 v12-Ksim | ~5-8W | 已实现 |
| 增加 parallel-games 到 32+ | ~10W | 立即可试，增大 batch |
| Cython MCTS 热路径 | ~15W | 减少 Python 等待，GPU 被调用更频繁 |
| MLX 原生张量树 | ~20W+ | 整个搜索在 GPU 上，但实现极复杂 |

**推荐：** 先尝试 `--parallel-games 32`（增大 GPU batch 到 ~128），看功率是否上升。如果仍然低，说明 Python 是唯一瓶颈，需要 Cython。

## 7. 对 v13 的建议

### 7.1 已解决的问题

- MCTS 训练信号：有效，可收敛，优于纯 policy ✓
- MCTS 速度：122 局/分钟，足够用于长时间训练 ✓
- 框架能力：stagnation detection + Pareto + step-normalized compare ✓

### 7.2 v13 应聚焦

1. **对手阶段晋升：** MCTS-50 已达到 vs L0 random 99%+ WR。下一步训练应对阵 minimax-L1 (depth 2)、L2 (depth 4)、L3 (depth 6)。需要验证 MCTS 训练能否打败非随机对手。

2. **Pareto 集成到 report：** 现在有真实的 MCTS 数据了，可以将 `--pareto` 输出嵌入 `--report --format json`，让 agent 在每次迭代前看到当前前沿。

3. **早停机制验证：** `--auto-stop-stagnation` 已实现但未在 MCTS 场景验证。30min MCTS run 没有停滞（WR 持续上升），需要在更长训练中观察。

4. **entropy 持久化到 DB：** TUI 显示 entropy 但不写入 cycle_metrics。如果需要跨 run 分析 MCTS 搜索质量变化，应将 mcts_stats 持久化。

### 7.3 不建议在 v13 做的事

- **Cython MCTS 热路径：** 122 局/分已足够，ROI 不高
- **MLX 原生张量树：** 工程复杂度极高，当前速度够用
- **增大 parallel-games 到 32+：** 可能导致内存压力，当前 8-10 已够

# v23.1 Update — Framework Boundary Repair / Megafile Split / Internal Reorganization 执行计划

> 2026-04-25  
> 前置：`updates/v23-update.md`、`updates/v23-findings.md`  
> 定位：不再继续堆新 domain 功能，而是专门解决 v23 已经暴露出来的 **framework / domain 边界错位、巨石文件增长、目录组织失衡** 三个问题，为下一阶段继续扩展 domain、Bayesian、adapter、可视化和 benchmark 做结构收敛。

---

## 1. v23.1 一句话目标

> **把 v23 暂时可用但结构上已经开始失衡的实现，收敛为一个边界更清晰、目录更稳定、巨石被拆散、旧入口仍兼容、并且经过完整回归验证的 framework 结构。**

v23.1 不做：

1. 不增加新的研究 domain。
2. 不新增新的业务功能面。
3. 不接入 live conversion。
4. 不把重构伪装成“顺手美化目录”。
5. 不做一次性大爆炸式路径搬迁。

v23.1 要做：

1. 把 **Bayesian generic machinery** 从 `domains/spot_trader/` 抽回 framework。
2. 把 `framework/core/db.py` 从单文件拆成 package。
3. 把 `framework/analyze.py` 从“总控脚本”拆成 facade + services。
4. 把 execution / research orchestration 从顶层平铺脚本中抽成服务层。
5. 建立 `facade/ + services/ + 收缩后的 core/` 目录骨架。
6. 保持现有 CLI 路径和关键 import 的兼容。
7. 在拆分完成后做完整测试、命令验证、artifact 验证，并把结果回填到本文底部。

---

## 2. 为什么 v23.1 必须先做结构收敛

v23 已经证明了：

1. `spot_trader` 可以承载 constrained Bayesian-style frontier refinement。
2. run-scoped workspace、可读 Pareto、constraint activation、mock uplift surface 都成立。
3. benchmark / replay / frontier plot / DB evidence 这条链路已经跑通。

但 v23 findings 也明确暴露出 3 个新的结构性问题：

1. **Bayesian loop 放在 `domains/spot_trader/` 中，作为 spike 可以接受，但长期已经越界。**
2. **`framework/analyze.py`、`framework/core/db.py` 等文件已经巨石化。**
3. **`framework/` 顶层继续平铺增长，而 `core/` 已经不足以承接所有内部抽象。**

如果不先做 v23.1，后面继续增加：

1. 新 domain
2. 新 adapter
3. 新 benchmark
4. 新 visualization layer
5. 更强的 Bayesian / posterior machinery

系统会先在 **组织层** 失控，而不是先在算法层失控。

因此 v23.1 的任务不是“顺手整理一下”，而是：

> **在功能继续扩张之前，先把 framework 的边界和组织形态固定下来。**

## 4. v23.1 的结构设计原则

v23.1 必须遵守下面 6 条原则：

1. **先抽取，再替换，不做大爆炸重排**
   - 先建立新模块位置，再让旧入口转调。

2. **旧命令先不消失**
   - 继续保留：
     - `framework/analyze.py`
     - `framework/sweep.py`
     - `framework/branch.py`
     - `framework/promote.py`
     - `framework/selector.py`
   - 但让它们逐步退化为 facade / compatibility shim。

3. **先拆 framework，总后再拆 domain**
   - 优先拆：
     - `core/db.py`
     - `analyze.py`
   - `domains/gomoku/train.py` 这种大文件不是 v23.1 第一刀。

4. **generic 留在 framework，domain 只保留 adapter / evaluator**
   - Bayesian loop、frontier benchmark、surrogate / acquisition 必须属于 framework。

5. **每一 phase 都必须可回归**
   - 每一轮拆分之后都要：
     - 通过测试
     - 跑旧命令
     - 核对 artifact 和 DB schema

6. **目录重组必须服务于下一阶段扩展**
   - 不是为了“目录好看”，而是为了：
     - 新 domain
     - 新 Bayesian capability
     - 新 visualization
     - 新 adapter
     有稳定落点。

## 8. Phase 1 — 拆分 `framework/core/db.py`

### 8.1 为什么先拆它

`core.db` 是当前依赖中心。

被以下位置广泛依赖：

1. framework command / service
2. domains/gomoku
3. domains/spot_trader
4. tests

如果它不先拆，后面的服务化会一直被它牵制。

### 8.2 拆分目标

把：

```text
framework/core/db.py
```

改成：

```text
framework/core/db/
  __init__.py
  connection.py
  migrations.py
  schema_base.py
  schema_campaigns.py
  schema_recommendations.py
  schema_fx.py
  repo_runs.py
  repo_campaigns.py
  repo_recommendations.py
  repo_fx.py
  repo_frontier.py
```

### 8.3 推荐拆分顺序

1. **connection**
   - `_connect`
   - `init_db`
   - `DB_PATH`

2. **schema / migrations**
   - `_SCHEMA_SQL`
   - `ALTER TABLE ...`
   - feature-version migrations

3. **repo_runs**
   - `create_run`
   - `finish_run`
   - `find_run_by_sweep_tag`
   - `list_run_metrics`

4. **repo_campaigns**
   - `save_search_space`
   - `save_objective_profile`
   - `get_or_create_campaign`
   - `link_run_to_campaign_v20`
   - `save_campaign_stage`

5. **repo_recommendations**
   - recommendation / batch / outcomes / surrogate snapshot

6. **repo_fx**
   - `save_quote_window`
   - `save_fx_quote`
   - `save_fx_route_leg`
   - `save_experiment_run`

7. **repo_frontier**
   - frontier snapshot / frontier-related persistence

### 8.4 兼容要求

`framework/core/db/__init__.py` 必须继续支持旧 import：

```python
from core.db import init_db, create_run, finish_run
```

这意味着：

1. 旧调用方无需立刻改完
2. 迁移可以分轮完成
3. 测试可以逐步切换

### 8.5 这一 phase 的测试

必须回归：

1. 所有 `core.db` 相关单测

### 18.6 兼容 shim 说明

本轮刻意保留了旧入口，不做一次性路径爆炸：

1. `framework/analyze.py`
2. `framework/sweep.py`
3. `framework/branch.py`
4. `framework/promote.py`
5. `framework/selector.py`

其中：

1. 前 4 个现在都是 **compatibility wrapper**
2. `framework/selector.py` 是 **compatibility shim**
3. `core.db` 则通过 package `__init__.py` 做 **兼容 re-export**

这保证了：

1. 旧脚本路径还可以直接运行
2. 旧测试中的 `from core.db import ...`
3. 旧测试中的 `from selector import ...`

都没有因为结构迁移而直接失效。

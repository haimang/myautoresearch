# v24.1 Findings — Floorplan-Checker 数据切分净洁度诊断与修复

> 2026-04-26  
> 前置：`updates/v24.1-update.md`

---

## 1. 结论先行

本轮发现的核心问题不是：

1. 训练时随机取 batch 取坏了
2. manifest 大面积指向了不存在的文件
3. 原始图片本体明显损坏

而是：

> **`floorplan_checker` 当前数据包的 `train / eval / test` 切分协议不干净：同一个 `listingId` 被同时放进了多个 split。**

这意味着 benchmark 有**题目泄漏（data leakage）**：

- 模型可能已经在 `train` 见过某个 listing 的样本
- 却又在 `eval` / `test` 再次被要求回答同一 listing
- 从而导致评估结果偏乐观，Pareto / recommendation 也会被污染

因此，“数据切分的净洁度”指的不是图片像素本身是否脏，而是：

> **benchmark 的样本边界是否隔离得足够干净。**

---

## 2. 我们到底检查了什么

本轮检查分成了四层：

1. **manifest 字段完整性**
   - `output_path`
   - `bedroom_head`
   - `bathroom_head`
   - `parking_head`
   - `listingId`

2. **文件路径可达性**
   - `output_path` 指向的图片是否真实存在

3. **split 计数一致性**
   - manifest 记录数与 `summary.json` 是否一致

4. **split leakage**
   - 同一个 `listingId` 是否跨 `train / eval / test`

---

## 3. 诊断结果

### 3.1 不是路径损坏问题

在修复前的抽样检查中：

1. `output_path` 可解析
2. 文件大面积存在
3. label 字段与当前 label map 基本对齐

修复后做的 **full strict contract check** 进一步证明：

1. `405,431` 条路径全部可达
2. 缺文件数 = `0`
3. 缺字段数 = `0`

所以问题**不是 manifest 链接大量失效**。

### 3.2 也不是训练时取样逻辑把数据混在一起

leakage 在训练开始前就已经存在于 manifest 中。  
也就是说：

1. 还没进入 train loop
2. 还没开始随机 batch
3. benchmark 就已经被污染了

所以问题**不是 loader 在运行时组合错了数据**。

### 3.3 真正的问题是 split 协议错误

修复前的统计结果：

1. 总记录数：`405,431`
2. 唯一 `listingId` 数：`394,802`
3. 没有缺失 `listingId` 的记录

但其中出现了跨 split 的 listing：

| overlap 类型 | listing 数 |
|---|---:|
| `train ∩ eval` | 1,597 |
| `train ∩ test` | 954 |
| `eval ∩ test` | 89 |
| **总 overlap listing** | **2,640** |

更具体地说，修复前的唯一 `listingId` 分布模式是：

| 模式 | listing 数 |
|---|---:|
| 只在 `train` | 337,754 |
| 只在 `eval` | 34,565 |
| 只在 `test` | 19,843 |
| 同时在 `train + eval` | 1,597 |
| 同时在 `train + test` | 954 |
| 同时在 `eval + test` | 89 |

这说明：

> **问题出在 split 生成阶段，而不是 train 阶段。**

从症状看，旧包更像是按图片或局部样本随机切分，而不是按 `listingId` 整组切分。

---

## 4. 对“原始数据是否有问题”的判断

当前更合理的判断是：

1. **原始图片不一定有问题**
2. **manifest 也不一定是“坏文件”**
3. **真正有问题的是：manifest 所表达的 benchmark 切分规则**

换句话说：

> 更像是“原始数据可用，但切分协议错误”，而不是“原始资产本身坏掉”。

所以优先修复点不是重做模型，而是：

1. 重建 clean split
2. 让同一个 `listingId` 只属于一个 split
3. 然后再重新恢复 Pareto / benchmark 推理

---

## 5. 可选修复路径

理论上有三种修法：

### 5.1 临时缓解法

把所有跨 split 的 `listingId` 从 `eval` / `test` 剔除。

优点：

1. 实现快
2. benchmark 立刻不再漏题

缺点：

1. 浪费样本
2. split 比例会漂
3. 不够优雅，也不够可复现

### 5.2 旧 split 最小改动法

对每个 `listingId`，把它全部归到“原来占比最多”的 split。

优点：

1. 改动少
2. 与旧分布更接近

缺点：

1. 结果依赖旧包状态
2. 不够稳定，也不够容易重建

### 5.3 推荐修法：按 `listingId` 稳定哈希重切分

对每个 `listingId` 做稳定 hash，再按目标比例分到：

1. `train = 0.86`
2. `eval = 0.09`
3. `test = 0.05`

优点：

1. 完全可复现
2. 与旧 split 偶然状态解耦
3. 以后重新打包时也能稳定得到同一结果
4. 可以从协议层彻底避免同一 listing 跨 split

缺点：

1. 会有一批记录从旧 split“逻辑迁移”到新 split
2. 需要重写 manifest

本轮选择了 **5.3**。

---

## 6. 本轮实际执行的修复

### 6.1 修复策略

新增脚本：

```text
domains/floorplan_checker/rebuild_manifests.py
```

执行逻辑：

1. 先备份旧 manifest：
   - `domains/floorplan_checker/dataset/manifests/backup-v24_1-pre-clean-split/`
2. 读取旧的：
   - `train.jsonl`
   - `eval.jsonl`
   - `test.jsonl`
3. 以 `listingId` 为唯一切分单元
4. 按稳定 SHA1 hash 将 listing 分配到新 split
5. 重写新的：
   - `train.jsonl`
   - `eval.jsonl`
   - `test.jsonl`
   - `summary.json`

### 6.2 为什么没有搬动全部图片文件

本轮没有去复制/搬运整套 `16GB+` 图片资产，而是：

> **保留物理文件原地不动，只重建 benchmark membership 的 manifest。**

原因是：

1. loader 实际上只依赖 `output_path` 是否可达
2. benchmark 的真正规则定义在 manifest，而不是目录名
3. 这样可以避免一次额外的大规模文件搬运

因此，修复后：

1. **物理文件仍在原位置**
2. **但 `train / eval / test` 的合法成员关系已经重新定义**

---

## 8. 最终判断

本次问题的本质是：

> **manifest/split 协议问题，而不是模型问题。**

更具体地说：

1. 不是 loader 在训练时把数据混坏了
2. 不是大量 `output_path` 失效
3. 不是当前已知证据下的“原始图像本体坏掉”
4. 是旧数据包把同一 `listingId` 分配到了多个 split

因此，这类问题是**可以修复的**，而且已经修复：

1. 通过 `listingId` 稳定哈希重切分
2. 通过 manifest 重建 clean benchmark
3. 通过 strict contract + smoke + Pareto 验证闭环确认

---

## 9. 后续建议

虽然本轮已经把 benchmark 修干净了，但还有两条后续建议值得保留：

1. **以后所有 floorplan 数据包都应把 split 规则写死在打包协议中**
   - 明确规定：split 的最小隔离单元必须是 `listingId`

2. **后续可以考虑再加一层近重复图检查**
   - 如果未来怀疑不同 `listingId` 其实还是近重复图片，可再引入：
     - 文件 hash
     - 感知哈希
     - listing 级去重规则

但这已经是“更高阶的数据去重”问题，不再是本轮的 benchmark 泄漏主问题。

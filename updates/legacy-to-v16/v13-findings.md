## 3. Run 2：8x128 vs S0（NN 对手晋升首测）

### 3.1 这是什么

这是项目历史上第一次 **NN vs NN 的 MCTS 训练**：
- 训练模型：8x128（2.49M 参数，是 S0 的 3.5 倍）
- 对手：S0（8x64, 713K，已达到 vs L0 99.2%）
- 搜索：MCTS-80（比 S0 训练时的 MCTS-50 更强）

### 3.2 速度

| 指标 | 8x64 pg=32 MCTS-50 | 8x128 pg=32 MCTS-80 | 比值 |
|------|--------------------|--------------------|------|
| Cycle 耗时 | 13.7s | 52.0s | 3.8x |
| 局/分钟 | 141 | 37 | 0.26x |

速度大幅下降的原因：
1. **模型 3.5x 大**：8x128 (2.49M) vs 8x64 (713K)，每次前向传播更慢
2. **MCTS-80 vs MCTS-50**：搜索量多 60%
3. **对手推理开销**：vs S0 的 probe eval 需要加载对手模型做推理

**37 局/分钟对于 2.5M 参数模型 + MCTS-80 是合理的。** 问题不在速度，在于训练量是否足够。

### 3.3 WR 曲线（核心发现）

```
cycle  5:   1.3%  ← 新模型完全不会下棋，被 S0 碾压
cycle 10:   4.0%
cycle 15:  29.3%  ← 开始学会基本下法
cycle 20:  32.0%
cycle 25:  46.7%  ← 追上中位线
cycle 30:  57.3%
cycle 35:  50.7%  ← 回落（震荡开始）
cycle 40:  34.7%  ← 大幅回落
cycle 45:  50.7%
cycle 50:  66.7%  ← 新高
cycle 55:  48.7%  ← 又回落
cycle 60:  85.3%  ← 突破！
cycle 65:  61.3%  ← 但没稳住
```

**WR 曲线呈上升趋势但剧烈震荡（std 22.7%）。** 前半段均值 28.4%，后半段均值 56.9%，趋势 +28.4%。这和 v12 分析中纯 policy vs L0 的模式有相似之处（震荡），但有关键区别：

| 特征 | 纯 Policy vs L0 (v12) | 8x128 MCTS-80 vs S0 |
|------|----------------------|---------------------|
| 总体趋势 | 平坦（72.5% 均值不变） | **上升（28% → 57%）** |
| 震荡幅度 | std 5.9% | std 22.7% |
| 最终 WR | 76% | 82% (checkpoint WR) |

**关键判断：上升趋势是真实的（28% 的均值提升不是噪声），但震荡是因为训练量不够。** 69 个 cycle、2208 盘对于 2.49M 参数的大模型来说太少了——模型需要更多数据来稳定。

### 3.4 Loss 分析

Loss 从 6.24 降到 0.97。最终 loss 0.97 远低于 S0 训练时的 4.97——说明 8x128 模型容量足够大，能快速拟合 MCTS 搜索分布。**但 loss 下降快而 WR 震荡大，可能意味着模型在过拟合 replay buffer 中的近期数据而非学到稳定策略。**

## 5. GPU 利用率观察

用户反馈 GPU 功率仍然在 ~5W 左右。基于 v12-findings.md 的分析，这是 MCTS 算法的固有特征：

```
时间分布（估算）:
  Python MCTS 树搜索:  ~95%   ← CPU 单线程
  numpy board.copy():   ~3%   ← CPU
  MLX GPU 前向传播:     ~1%   ← GPU
  MLX CPU↔GPU 同步:     ~1%   ← 等待
```

**pg=32 增大了 GPU batch（~256 vs 之前的 ~29），但 GPU 调用次数没有减少（仍然是每 sim round 1 次调用）。** GPU 每次处理更多数据（batch=256 更高效），但被调用的频率没有变——Python 仍然是瓶颈。

**这解释了为什么 pg=32 的 GPU 功率没有明显提升：** batch 从 29 升到 256 让每次调用更高效，但 GPU 仍然 99% 时间在等 Python。功率提升需要减少 GPU 等待时间，而这需要 Python 加速（多进程或 C 扩展）。

## 7. 对 8x128 vs S0 训练的建议

### 7.1 当前问题

WR 震荡大（std 22.7%）的可能原因：

1. **训练量不足**：2,208 盘 / 2.49M 参数 = 0.89 盘/K参数。vs L0 训练时是 2,112 / 713K = 2.96 盘/K参数。**大模型需要更多数据。**
2. **MCTS-80 可能过强**：80 sims 的搜索比 50 sims 更聚焦，但 policy target 可能与当前模型能力差距太大，导致训练不稳定。
3. **Replay buffer 过小**：默认 50K 的 buffer 对 2.49M 模型可能不够——模型在反复拟合近期数据。

### 7.2 下一步建议

```bash
# 方案 A：更长训练（2 小时），让 8x128 有足够数据
uv run python domains/gomoku/train.py \
  --mcts-sims 50 --parallel-games 32 --mcts-batch 8 \
  --num-blocks 8 --num-filters 128 \
  --time-budget 7200 \
  --eval-opponent S0 --eval-level 1 \
  --eval-interval 5 --probe-games 50 \
  --buffer-size 100000 \
  --auto-stop-stagnation --seed 42

# 方案 B：降回 8x64，先验证 vs S0 能否稳定收敛
uv run python domains/gomoku/train.py \
  --mcts-sims 50 --parallel-games 32 --mcts-batch 8 \
  --num-blocks 8 --num-filters 64 \
  --time-budget 3600 \
  --eval-opponent S0 --eval-level 1 \
  --eval-interval 5 --probe-games 50 \
  --auto-stop-stagnation --seed 42
```

**推荐方案 B：** 先用 8x64（与 S0 同架构）验证 MCTS vs NN 训练能否稳定收敛。如果 8x64 vs S0 能稳定到 80%+，再考虑升级到 8x128。

## 1. v24.1 一句话目标

> **以 Gomoku 阶段积累的实验工程经验为约束，对 `floorplan_checker` 做一次“研究系统化”升级：补齐数据落位、评估协议、策略文件、测试覆盖、数据吞吐、CV 语义 continuation 与诊断指标，使其真正具备长期 Campaign / Bayesian Frontier Search / 人类可解释决策的基础。**

v24.1 不做：

1. 不引入图像分割、目标检测、OCR 等新任务形态。
2. 不引入非 Apple Silicon 的训练支持。
3. 不为 `floorplan_checker` 在 framework 内部新增图像专属硬编码。
4. 不急于追求最强 SOTA 模型，而忽略评估协议和实验可解释性。
5. 不在 benchmark 尚未固定前就贸然把 BO 搜索预算扩大到大规模长跑。

v24.1 要做：

1. **完成数据落位与数据契约固化**：把 root tar 中的数据解到 domain 期望位置，并明确 manifest/label/路径约束。
2. **完成 v24 初版缺口修复**：修 manifest 路径残留、README 文档漂移、废弃 MLX API、policy schema 草稿化问题。
3. **建立 `floorplan_checker` 的测试与冒烟基线**：至少具备 manifest load、训练 smoke、framework 默认策略路径回归。
4. **把 Gomoku 的关键经验翻译到 CV 领域**：先固定 benchmark，再扩大 search；先保吞吐，再扩模型；先要诊断指标，再谈 Pareto 决策。
5. **形成可执行的 v24.1 Phase 方案**：为后续真正进入长期 CV Campaign 做结构化铺垫。

## 3. 为什么 v24.1 是必要的

`floorplan_checker` 当前虽然已经从“设计稿”进入“可执行代码”，但仍明显处在：

```text
能跑
≠
可研究
≠
可复现
≠
可做长期 BO / Pareto 决策
```

这与 Gomoku 早期阶段非常相似。

Gomoku 阶段最重要的经验并不是“怎么把模型训起来”，而是：

1. **先固定 benchmark，否则结论会整体漂移**
2. **先保证吞吐，否则研究节奏会被基础设施拖死**
3. **先产生高质量诊断指标，否则 Pareto 图只是漂亮图片**
4. **先分阶段淘汰坏点，否则算力会浪费在无意义区域**
5. **先把 domain-specific continuation reason 讲清楚，否则 branch 只是形式化存在**

`floorplan_checker` 现在正好处于：

> **必须把这些经验正式翻译进 CV 领域工程的时间点。**

## 5. v24.1 结构原则

v24.1 必须遵守下面 8 条原则：

1. **Benchmark-first**
   - 所有更大的搜索、branch、Bayesian refine，都必须建立在固定的 eval/test 协议之上。

2. **Data-contract-first**
   - dataset 不是“有文件就能训”，而是一个要被 framework 长期依赖的协议。

3. **Throughput matters**
   - 对 40.5 万个小文件的 CV 任务，吞吐不是优化项，而是研究节奏的决定因素。

4. **Legacy compatibility stays, but decision surface must move to `run_metrics`**
   - `final_win_rate` 仍可保留作兼容代理值
   - 但真正的 Pareto / selector / acquisition 决策，应逐步转向真实 `run_metrics`

5. **CV 特定逻辑只留在 `domains/floorplan_checker/`**
   - framework 仍然保持 domain-agnostic

6. **Multi-head diagnosis must be first-class**
   - 不能只看平均 acc
   - 必须能知道哪个 head 变好了、哪个 head 在拖后腿

7. **Stage / branch / BO 是一套联动系统**
   - stage policy 不是装饰
   - branch policy 不是占位
   - acquisition 不是孤立 reranker

8. **每一步都要保持快速冒烟可用**
   - 任何新能力都必须有小样本、小预算、短时 smoke 路径

   - `samples_per_second`
   - `loader_wait_ratio`（或近似等待指标）

4. 支持图像尺寸成为真实执行轴：
   - `image_resolution`

**验收标准**：

1. 数据吞吐高于 v24 初版 loader
2. 单轮 smoke 不再频繁新建线程池

## 7. Gomoku 经验到 Floorplan-Checker 的明确映射

这是 v24.1 最重要的思想来源。

### 7.1 经验一：Benchmark 一定要先固定

Gomoku 的早期问题证明：

> **如果 benchmark 本身漂移，后面所有胜率 / frontier / report 都会被污染。**

映射到 `floorplan_checker`：

1. `eval` 和 `test` 必须固定
2. split leakage 必须显式检查
3. head 指标与 class bucket 不能临时改口径

### 7.2 经验二：吞吐决定研究速度

Gomoku 中 self-play 吞吐的改进，直接决定了研究迭代速度。

映射到 `floorplan_checker`：

1. loader 不是次要问题
2. 40.5 万小文件场景下，I/O 组织就是核心研究基础设施
3. 没有 samples/s 与 step time 指标，就无法判断是模型慢还是 loader 慢

### 7.3 经验三：不要过早相信单一总分

Gomoku 虽然最终常看 `win_rate`，但真正决策依赖很多细粒度信号。

映射到 `floorplan_checker`：

1. 平均 acc 只能做兼容代理值
2. 真正决策要看三个 head 与成本轴共同构成的 Pareto

### 7.4 经验四：多阶段 fidelity 是节省预算的关键

Gomoku 中：

1. A/B/C/D 的意义是先筛坏点，再确认好点

映射到 `floorplan_checker`：

1. 先 smoke 排除 shape/data/OOM
2. 再短预算看收敛趋势
3. 最后再进入长预算验证

### 7.5 经验五：branch 必须是“可解释的后续实验”

Gomoku 的 continuation 真正有价值，是因为它回答：

> 为什么要从这个点继续，而不是从别的点继续？

映射到 `floorplan_checker`：

1. branch reason 必须回答：
   - 是不是某个 head 弱？
   - 是不是容量不够？
   - 是不是 learning rate 太高？
   - 是不是 resolution 不够？

## 8. v24.1 收口判断

当 v24.1 完成时，系统至少应能可靠回答下面这些问题：

1. **“在固定 eval/test 协议下，哪个 head 最难学？”**

3. **“哪些点位于真正的多头 Pareto frontier，而不是平均 acc 假象？”**

5. **“如果显存预算固定，我们应该优先加深网络、加宽网络，还是调整 loss 权重 / 分辨率？”**

v24.1 的完成标准不是“又多写了几个 JSON”，而是：

> **`floorplan_checker` 必须从一个新加入的 domain，升级为一个可被 framework 长期驱动、可被人类可靠理解、可被 Bayesian / Pareto 机制持续探索的研究域。**

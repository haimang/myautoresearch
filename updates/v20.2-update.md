# v20.2 Update — Multi-Fidelity Promotion Engine

> 2026-04-20 | 前置：`updates/v20-roadmap.md` §6.3、`updates/v20.1-update.md`（v20.1 已完成 search-space schema + campaign ledger）

---

## 1. 版本目标（一句话）

> **让 v20.1 已经 formalize 的 campaign，不再只是“同一轮研究里的平面 runs”，而是升级成“短预算筛查 → 中预算复筛 → 长预算晋升”的分层实验系统；也就是：把 campaign ledger 继续推进成真正的 multi-fidelity promotion engine。**

v20.1 已经完成的事实清单（必须承认）：

- `search_space.json` / `framework/search_space.py` 已落地
- `campaigns` / `campaign_runs` / `search_spaces` 已落库
- `sweep.py --campaign --search-space` 已落地
- `analyze.py --campaign-summary` / `--pareto --campaign` 已落地
- 至少 1 条 gomoku smoke campaign 已真实写入 `tracker.db`

但 v20.1 仍然缺少一层关键能力：

> **系统已经知道“这是一轮什么研究”，却还不知道“哪些点只该短跑筛查，哪些点值得被追加预算，哪些点必须先做 seed 复验才能继续推进”。**

v20.2 就是补这一层。

---

## 2. v20.2 的直接任务

v20.2 只做三件事：

1. **Stage policy 化**
   - campaign 不再只有一个统一预算
   - 而是明确分成 Stage A / B / C / D
   - 每个 stage 有自己的预算、seed 要求、晋升名额、晋升门槛

2. **Promotion ledger 化**
   - “某个点为什么从 A 晋升到 B、为什么另一个点被淘汰”
   - 不再只存在于人类笔记里
   - 而是正式写进数据库，带有 reason / metric / rank / seed coverage

3. **Seed 复验纪律化**
   - Stage A 的胜利不能直接当成长周期结论
   - 晋升必须经过 policy 指定的 seed 扩充与聚合统计
   - 系统必须区分“单 seed 偶然领先”和“多 seed 仍稳定领先”

如果用最短的话概括：

> **v20.1 解决“这是一轮什么研究”，v20.2 解决“这轮研究该如何分层推进，以及谁有资格拿到更多预算”。**

---

## 3. v20.2 与相邻节点的明确边界

这一节是 v20.2 最重要的边界说明。后续所有开发都必须按这个表控 scope。

| 节点 | in-scope | out-of-scope |
|---|---|---|
| **v20.1** | search-space schema、campaign ledger、protocol lock、campaign summary | stage policy、promotion rule、seed variance governance |
| **v20.2** | Stage A/B/C/D policy、promotion decision ledger、stage-aware execution、seed revalidation、stage summary | checkpoint continuation、trajectory report、next-point recommendation、Bayesian acquisition |
| **v20.3** | checkpoint continuation / branch ledger、trajectory report、branch reason、parent-child 对比 | surrogate ranking、Bayesian MOBO |
| **v21** | surrogate-guided next-point / next-branch ranking | 标准意义上的 BO / MOBO |
| **v21.1** | Bayesian multi-objective frontier exploration | 更后面的 dashboard / telemetry orchestration 扩展 |

**v20.2 的铁边界：**

1. **不**做 checkpoint branching
2. **不**做 trajectory report
3. **不**做 next-point / next-branch recommendation
4. **不**做 Bayesian optimization

只要需求跨过这四条，统一判到 v20.3+。

---

## 4. 为什么 v20.2 必须紧跟在 v20.1 之后

v20.1 解决的是研究的“归属”和“可比性”；  
v20.2 要解决的是研究的“节奏”和“预算分配”。

如果 v20.2 不做，系统仍然会有三个根本问题：

1. **所有 runs 仍然被视为同一层 fidelity**
   - 短预算和长预算混在一起看 front
   - 会把早期噪声误当长期结论

2. **晋升逻辑仍然靠人脑**
   - 人类需要自己从 campaign summary 里判断谁该继续跑
   - 框架无法沉淀这个判断过程

3. **seed 纪律仍然没有框架约束**
   - 单 seed 优势随时可能是偶然
   - 但系统目前还不会强迫补 seed 再决定

所以 v20.2 的角色并不是“更聪明地选点”，而是先把：

> **“先短跑筛掉明显差点，再在更稳定的统计条件下分配更贵预算”**

这件事正式制度化。

严格列表形式。每一项都是独立可验收的工作包。

### 6.3 C 系 — Sweep / Promote 的 stage-aware 执行

| # | 工作包 | 文件 | 产出 / 可验收 | 预期收益 |
|---|---|---|---|---|
| **C1** | `sweep.py` 新增 `--stage-policy` / `--stage` | `framework/sweep.py` | 指定 campaign 当前运行的 stage；stage 预算优先取 policy，而不是手填 CLI | 每轮 run 明确知道自己处于 A/B/C 哪一层 |
| **C2** | 新建 `promote.py --plan` | `framework/promote.py`（新） | 基于 `campaign` + `from_stage` + `to_stage` 读取 policy，按 candidate 聚合并输出 promotion plan | 人类第一次能直接看到“谁会晋升、依据是什么” |

## 7. v20.2 的 in-scope / out-of-scope

### 7.1 In-scope

下面这些内容属于 v20.2 正式范围：

1. Stage policy 的**格式定义**
2. Stage policy 的**加载、校验、摘要输出**
3. `campaign_stages` / `promotion_decisions` 的正式入库
4. `candidate_key` 与按 seed 聚合的 promotion 纪律
5. `sweep.py` 的 stage-aware budget / stage 标注
6. `promote.py` 的 deterministic promotion plan / execute
7. `analyze.py` 的 stage summary / promotion log / stage filter
8. 一条最小 gomoku A→B→C smoke chain 作为真实证据

### 7.2 Out-of-scope

下面这些事情**明确不在 v20.2**：

1. **checkpoint continuation / branch 执行** → v20.3
2. **trajectory report / branch tree / parent-child 对比** → v20.3
3. **自动判断下一条最值得探索的新点** → v21
4. **surrogate model / recommend-next / active ranking** → v21
5. **Bayesian / MOBO / EHVI / acquisition function** → v21.1
6. **自动切换 eval benchmark（L0 → L1 → L2）** → 后续节点 / findings
7. **长期 gomoku 最优超参结论** → findings 文档，不是 v20.2 发布门槛

### 7.3 一个特别重要的边界

v20.2 可以：

- 识别“哪些候选值得继续”
- 把这些候选推进到更高预算的 Stage B / C
- 为未来 Stage D 预留 handoff 语义

但 **v20.2 不能真的把某个 checkpoint 分叉成 continuation branch**。

也就是说：

> **v20.2 可以知道“这个候选已经值得进入 Stage D”，但不能在本节点里真正执行 checkpoint branching。**

这条边界必须守住。

## 8. Stage 语义建议（v20.2 内部标准）

为避免实现时每个人理解不同，先把 v20.2 默认 stage 语义写死在 update 中：

| Stage | 定位 | 默认用途 | 默认 seed 纪律 | 输出给下一阶段的东西 |
|---|---|---|---|---|
| **A** | 短预算筛查 | 快速排除明显差点 | 允许 1 seed | 候选列表 |
| **B** | 中预算复筛 | 对 A 胜出点做 seed 复验 + 重新排序 | 至少 2 seeds | 更稳的候选列表 |
| **C** | 长预算晋升 | 对少数候选做更贵预算确认 | 至少 2-3 seeds | 值得长期跑或值得进 D 的候选 |
| **D** | handoff slot | 留给 v20.3 的 continuation / branch 验证 | 不在 v20.2 执行 | 未来 trajectory 输入 |

**重要说明：**

1. v20.2 的正式收口只要求 **A→B→C** 真正跑通
2. Stage D 必须在 policy 中被明确定义
3. 但 Stage D 只作为 **handoff target**，不作为 v20.2 的实装闭环

## 9. 测试要求（强化版）

这一版的测试要求比 v20.1 更偏“决策正确性”。  
因为 v20.2 一旦把 promotion rule 或 seed discipline 写错，后续所有 trajectory 和 selector 都会建立在错误的预算分配之上。

### 9.1 自动化测试必须新增三组

| 测试文件 | 重点 | 最低覆盖要求 |
|---|---|---|
| `tests/test_stage_policy.py` | policy 解析、校验、负例 | 至少覆盖 8 个 case |
| `tests/test_promotion_db.py` | stage / decision ledger、幂等、聚合 JSON | 至少覆盖 8 个 case |
| `tests/test_promotion_cli.py` | `promote.py` / `sweep.py` / `analyze.py` 的 stage 行为 | 至少覆盖 10 个 case |

### 9.2 `test_stage_policy.py` 必测项

1. 合法 gomoku stage policy 可加载
2. 缺 `stages` / `time_budget` / `seed_count` 任一字段时报错
3. stage 顺序非法时报错（例如缺 B 直接跳 C）
4. `promote_top_k < 0` 报错
5. Stage D 缺失时报错或明确拒绝
6. `describe_stage_policy()` 输出稳定且包含 budgets / seeds / top-k
7. 相同 policy hash 稳定一致
8. `search_space_ref` 与 domain 不一致时报错

### 9.3 `test_promotion_db.py` 必测项

1. DB 初始化后 `campaign_stages` / `promotion_decisions` 存在
2. 重复初始化 DB 不报错
3. `candidate_key` 对相同 axis values 稳定一致
4. promotion decision 写入成功且 JSON round-trip 正确
5. 同一 decision 的重复写入具备幂等或显式唯一约束
6. `campaign_stage` 生命周期（open → closed）可追踪
7. 聚合指标 `mean/std/run_count` 按预期保存
8. 同一 candidate 跨 seed / 跨 stage 关联正确

### 9.4 `test_promotion_cli.py` 必测项

1. `sweep.py --stage-policy ... --stage A --dry-run` 正确打印 stage budget
2. `promote.py --plan` 正确按 candidate 聚合、排序并输出 decision
3. `promote.py --execute` 只补缺失 seeds，不重复已有 runs
4. `promote.py --plan` 在 seed 不足时输出 hold / need-more-seeds
5. `analyze.py --stage-summary <campaign>` 输出 stage run count / candidate count
6. `analyze.py --promotion-log <campaign>` 输出 decision/reason/rank
7. `--pareto --campaign --stage X` 只分析该 stage runs
8. 不存在的 campaign / stage 给出友好提示
9. policy 与 campaign/search-space 不兼容时拒绝执行
10. Stage D execute 尝试被明确拒绝并提示转交 v20.3

### 9.5 现有测试的基线要求

v20.2 开工前，必须先跑当前已有测试作为 baseline。  
v20.2 完成后，必须重新跑**全量**测试：

```bash
python3 -m unittest discover tests -v
```

要求：

1. v20.1 既有测试全部继续通过
2. 新增三组测试全部通过
3. 不允许只跑 promotion 相关测试就宣布完成

### 17.4 关键问题详解

#### 问题 1：`promote.py` 真正执行的晋升规则，比设计稿更弱

`framework/promote.py` 的 `plan_promotion()` 当前读取了：

- `promote_top_k`
- `seed_target`
- `metric`

但它**没有读取也没有使用** `min_runs`。  
实际判断条件是：

```python
elif rank <= promote_top_k:
    if seed_count >= seed_target:
        decision = "promote"
```

而 `framework/stage_policy.py` 里另一套 `plan_promotions()` 实现，却明确要求：

```python
if cand["seed_count"] >= seed_target and cand["run_count"] >= min_runs:
    decision = "promote"
```

这说明当前仓库里存在**两套 promotion planner 实现**，并且已经发生了行为漂移。  
这不是纯粹的代码整洁问题，而是会直接影响候选是否被错误晋升的正确性问题。

#### 问题 2：policy compatibility guard 没有落到“search-space 级”

v20.2 文档要求：

> policy 与 campaign/search-space 不兼容时，应拒绝执行

但 `promote.py` 当前只检查：

```python
if policy["domain"] != campaign["domain"]:
    sys.exit(1)
```

也就是说：

- `domain` 错了会拒绝
- 但 `search_space_ref.name/version` 错了不会拒绝
- `stage_policy` 与 campaign 绑定的 search-space/profile 不一致也不会拒绝

我已经做了实际复现：把 `search_space_ref.name` 改成错误值后，`promote.py --plan` 仍然返回 0 并输出完整计划。  
因此这条 guard 当前只能算 **半实现**。

#### 问题 3：Stage D 边界是“文档边界”，还不是“代码边界”

文档多处强调：

- Stage D 必须存在
- Stage D 是 v20.3 的 handoff slot
- v20.2 只能识别“值得进入 D”，但不能真的执行 branching

其中“执行 D 被阻止”这一点代码已经做到：`promote.py` 在 `to_stage == "D"` 时会直接退出。  
但“Stage D 必须存在”这一点，`validate_stage_policy()` 并没有强制。A/B/C-only policy 当前会被接受。

这意味着 v20.2 与 v20.3 的边界，目前**不是由 validator 保证的**，而是主要靠文档约束和人工自觉。

#### 问题 4：真实 smoke chain 还没有证明正式的 seed discipline

v20.2 的工作目标不是单纯引入 stage 字段，而是把：

> “短预算筛查 → seed 复验 → 更贵预算晋升”

这套纪律正式落地。

但当前真实链路使用的是：

- `output/stage_policy_smoke.json`
- Stage A/B/C 全部 `seed_target = 1`

因此它证明了：

- stage ledger 能工作
- promotion_decisions 能落库
- A→B→C 的执行链能跑通

但它**没有证明**：

- Stage B/Stage C 在正式 3-seed policy 下真的会补种子
- `promote.py --execute` 真能把“缺 seed → hold → 补 seed → promote”这条链路跑通

也就是说，当前 smoke 证明了 **multi-stage execution**，但还没有完全证明 **seed revalidation discipline**。

## 1. 为什么要写这份讨论记录

在 `v25-findings-v4` 与 `v25.4-update` 之后，我们又进行了一轮非常关键的讨论。  
这轮讨论的重点不是补更多实验，而是重新追问：

1. `floorplan_checker` 真正的优化目标到底是什么。
2. 笛卡尔积平面的横轴、纵轴到底应该是什么。
3. Pareto 前沿上的斜率到底有没有真实世界意义。
4. 哪些指标应该进入权衡，哪些指标只是诊断项。
5. 哪些参数应该进入主搜索，哪些参数只是噪声、约束或次级放大轴。

这份文档的目的，是把这轮讨论单独记录下来，避免后续又退回到：

1. 用诊断指标代替最终目标。
2. 在没有定义清楚平面语义前就开始做新实验。
3. 在没有现实意义的平面上谈 Pareto、slope、knee。

## 2. 用户的核心论述

用户在这轮讨论中提出的关键观点，可以概括为下面 5 条。

### 2.1 floorplan_checker 的目标不是 bathroom

用户明确指出：

1. 在 `spot_trader` 中，追求的是保值、增值与流动性的权衡。
2. 在 `gomoku` 中，追求的是单位时间内 WR 上升的速度。
3. 在 `floorplan_checker` 中，追求的也应该是：
   - **综合 eval 精度 / 正确率的提升**
   - 以及这种提升与成本之间的权衡
4. `bathroom` 不是最终目标之一，也不是用户关心的权衡目标之一。

换句话说，用户强调的是：

> **我们最终要优化的是模型整体在 eval 上的表现，而不是某个单头指标。**

### 2.2 必须先定义“牺牲了什么，换来了什么”

用户强调，真正的 tradeoff 问题必须回答：

1. 做了选择 `A`
2. 牺牲了什么 `B`
3. 换来了什么 `C`

如果这一点说不清楚，那么所谓的平面、前沿、knee 都没有意义。

用户真正关心的是：

1. 模型能不能稳定训练。
2. 应该怎么训练。
3. 哪些参数是需要权衡的。
4. 哪些参数只是横扫、噪声、无效参数，不应该进入主决策平面。

### 2.3 必须先定义笛卡尔积平面的横轴、纵轴

用户明确指出，之前实验里最根本的问题不是“点不够多”，而是：

1. 横轴没有被定义成一个有现实意义的量。
2. 纵轴没有被定义成一个有现实意义的量。
3. 于是画出来的 Pareto front 本身就没有现实解释。

用户的批评非常明确：

> **如果没有先定义平面，任何新的测试都是毫无意义的。**

### 2.4 如果斜率没有意义，Pareto 就没有意义

用户进一步指出：

1. 即使前沿曲线存在
2. 如果曲线上的斜率没有真实世界解释
3. 那么这条曲线本身就没有决策意义

这意味着，后续实验不能只追求“画出 front”，而必须回答：

> **前沿上每一段的 slope，到底在现实里代表什么代价/收益交换。**

### 2.5 诊断指标不能直接进入最终权衡

用户明确反对把 `bathroom` 作为目标中心，并指出：

1. 自己不想要 bathroom 本身作为目标
2. 也不关心 bathroom 本身动不动
3. 真正关心的是：
   - 模型整体能不能训练
   - 应该怎么训练
   - 哪些参数值得权衡
   - 哪些参数应从主搜索中剔除

这实际上是在强调：

> **诊断项与决策项必须分层。**

## 3. 我之前的错误与需要纠正的地方

这轮讨论里，我前面的主要错误有 4 个。

### 3.1 把 bathroom 这个“瓶颈诊断项”说成了“决策中心”

我之前不断强调 bathroom，是因为：

1. v2 严格评估下，bathroom 是当前最弱头。
2. 它拖住了 `min_head` 与 `head_gap`。

但问题在于：

1. 这只能说明 bathroom 是一个**诊断信号**
2. 不能说明 bathroom 是**最终优化目标**

也就是说，我把：

1. “当前最显著的故障指示器”

误说成了：

2. “当前应该围绕它做决策的目标”

这是不对的。

### 3.2 把诊断平面误当成最终决策平面

我曾把：

1. `val_loss_bathroom`
2. `val_acc_min_head`
3. `val_head_acc_gap`

放进中心讨论中，并据此设计局部平面。

这在“诊断某类失败机理”时是有用的，但问题在于：

1. 它们不是最终产品目标
2. 也不是用户真正想权衡的东西

因此，这种平面最多只能回答：

> **为什么模型涨不动。**

不能回答：

> **什么样的模型值得选、值得继续投入预算。**

### 3.3 在没有定义平面语义前就谈 Pareto / knee / Bayes

我前面的实验虽然做了不少点，但从用户这轮追问来看，一个核心缺陷非常明显：

1. 我没有先定义清楚：
   - 横轴是什么
   - 纵轴是什么
   - slope 是什么
2. 却已经开始讨论：
   - Pareto front
   - knee
   - Bayes refine

这导致实验在形式上像是在探索 frontier，但在意义上并没有落到真实业务权衡上。

### 3.4 没有先把“决策层 / 诊断层 / 可行性层”分开

之前的思路里，下面这些层次没有被严格分开：

1. **可行性层**
   - 能不能训
   - 会不会 NaN / OOM / 数据泄漏

2. **决策层**
   - 综合 eval utility
   - wall time / latency / memory 的代价

3. **诊断层**
   - bathroom
   - head gap
   - per-class recall
   - count MAE

这使得很多指标虽然“有信息”，但不应该直接进入权衡平面。

## 4. 我对用户论述的修正性回应

在这轮讨论后，我对前面思路的修正，可以概括成下面 6 条。

### 4.1 bathroom 不是最终目标，只是诊断信号

更准确的说法应该是：

1. bathroom 是当前实验中最显眼的瓶颈诊断项
2. 但它不是用户真正要优化的目标
3. 它不能进入最终决策平面做纵轴

正确的位置应该是：

> **bathroom 属于诊断层，而不是决策层。**

### 4.2 floorplan_checker 的主目标应回到“综合 eval utility”

像 `gomoku` 看 `WR / 时间`、`spot_trader` 看收益/保值/流动性一样，`floorplan_checker` 的主问题也应该写成：

> **综合 eval 精度 / 正确率，与成本之间的权衡。**

也就是说，真正的目标不是某个头，而是：

1. 模型整体 eval 表现
2. 这种表现的提升速度
3. 提升所需付出的成本

### 4.3 必须先定义真实世界有意义的笛卡尔积平面

后续所有 Pareto / knee / Bayes 之前，必须先把平面定义清楚。

我在回应中给出的修正是：

#### 主研究平面

1. `x = wall_time_s`
2. `y = overall eval utility`

这张图的含义是：

> **多花训练成本，换来多少整体 eval 精度。**

#### 部署平面 1

1. `x = inference_latency_ms`
2. `y = overall eval utility`

含义：

> **多付多少推理延迟，换来多少整体精度。**

#### 部署平面 2

1. `x = peak_memory_mb`
2. `y = overall eval utility`

含义：

> **多付多少内存，换来多少整体精度。**

### 4.4 纵轴必须是单一综合指标

我在回应中承认，纵轴不能再是：

1. bathroom
2. min-head
3. bathroom loss
4. head gap

这些都应退到诊断层。

真正可进入纵轴的候选，只能是一个**单一综合 utility**。  
回应里提出的候选包括：

1. `overall eval exact-match rate`
2. `overall eval aggregate accuracy`

其中更接近真实世界整体效用的，是：

> **整体 eval exact-match rate**  
> 即：一张图上的全部目标都正确，才算整张图正确。

如果业务上不需要 exact-match，那么也至少要先正式定义一个统一的 aggregate score。

### 4.5 slope 只有在平面定义正确时才有意义

我在回应中接受了用户关于斜率的批评，并给出修正定义：

如果：

1. `x = cost`
2. `y = overall eval utility`

那么前沿上相邻点的 slope：

1. `Δutility / Δcost`

表示：

> **每多付出 1 单位成本，能换来多少综合精度。**

反过来看：

1. `Δcost / Δutility`

表示：

> **为了多拿 1 单位综合精度，需要多花多少成本。**

这时，slope 才有真实世界意义。

而且 knee 才能被解释为：

> **从某一点往后，单位成本换来的综合精度边际收益明显变差。**

### 4.6 应把参数分成“主搜索参数 / 次级放大参数 / 冻结参数”

我在回应中接受了用户对“哪些参数应该进入主搜索”的要求，并给出修正分层：

#### 主搜索参数

只保留那些最可能真实改变“整体精度 vs 成本”关系的参数：

1. `loss_family`
2. `sampler`
3. `head_arch`

这些参数回答的是：

> **模型应该怎么训练，整体效用才会真正上升。**

#### 次级放大参数

在已经找到有效训练方法之后，再重新打开：

1. `num_res_blocks`
2. `image_resolution`

它们回答的是：

> **在方法已经有效的前提下，扩大容量或分辨率是否值得。**

#### 冻结参数

在当前阶段先固定：

1. `batch_size = 16`
2. `loss_weight_bed = loss_weight_bath = loss_weight_park = 1.0`
3. `seed = 42`
4. 显式 sample caps

因为这些参数当前更像：

1. 协议约束
2. 稳定性设置
3. 已经被证明信息密度不高的旋钮

## 5. 经过讨论后，应该如何重写 floorplan_checker 的问题定义

讨论之后，更合理的问题定义应是：

> **在确保训练可行、评估协议稳定、数据契约正确的前提下，寻找那些能够改变“整体 eval utility vs 成本”关系的训练方法参数；其中单头指标只作为诊断工具，不直接进入最终权衡平面。**

这一定义包含 3 层：

### 5.1 可行性层

回答：

1. 模型能不能训
2. 会不会崩
3. 数据和协议是否可信

这里的指标包括：

1. `nan_loss_count`
2. `OOM / peak_memory` hard cap
3. `dataset_missing_file_count`
4. `dataset_missing_field_count`
5. `dataset_leakage_count`
6. `dataset_summary_mismatch_count`

### 5.2 决策层

回答：

1. 到底值不值得选这个点
2. 多花多少成本换来多少整体效用

这里的真正平面应是：

1. `wall_time_s` vs `overall eval utility`
2. `inference_latency_ms` vs `overall eval utility`
3. `peak_memory_mb` vs `overall eval utility`

### 5.3 诊断层

回答：

1. 为什么整体 utility 上不去
2. 到底是哪一类机制出了问题

这里才放：

1. bathroom
2. head gap
3. per-class recall
4. count MAE
5. confusion matrix

也就是说：

> **诊断层帮助解释 front，决策层才定义 front。**

## 6. 对 v25 之前实验的重新定性

经过这轮讨论，前面已经做过的那些实验，不应再被理解为“真正已经找到 Pareto 结构的决策实验”，而应被重新定性为：

### 6.1 search-space 清洗实验

它们帮助证明了：

1. 哪些旧参数轴不值得继续当主轴
2. 哪些参数主要只会带来成本平移

### 6.2 诊断实验

它们帮助证明了：

1. `loss_weight_bath` 不是有效的主杠杆
2. 结构和分辨率在当前 protocol 下没有直接改变关键质量行为

### 6.3 语义澄清实验

它们进一步逼出了一个更重要的事实：

> **在没有先定义横轴、纵轴、slope 的情况下，实验即便跑完，也不具备真实的决策意义。**

## 7. 对 v25-v4 与 v25.4 update 的影响

这轮讨论对现有文档有直接影响。

### 7.1 需要保留的部分

`v25-findings-v4` 与 `v25.4-update` 中，下面这些判断仍然成立：

1. 旧 search space 的参数种类不足。
2. 旧的结构/分辨率/batch/loss-weight 宽扫不再适合作为主轴。
3. 参数应拆成不同 family，而不是继续在一个大平面里混扫。
4. 需要把不同假设拆成不同 surface。

### 7.2 需要重新解释甚至弱化的部分

但下面这些表述需要被重新解释：

1. 任何把 bathroom 放在决策中心的位置
2. 任何把 `min_head / head_gap / bathroom loss` 直接当成最终目标平面的表述
3. 任何在未先定义 overall utility 前就直接谈 knee 的表述

更准确的说法应该是：

> **这些指标属于 diagnosis surface，不属于最终 decision surface。**

## 8. 讨论后的结论

这轮讨论最终把问题定义纠正到了一个更稳的框架上。

### 8.1 用户观点的核心正确性

用户的核心论述是正确的：

1. `floorplan_checker` 的最终目标不是 bathroom。
2. 真正要权衡的是：
   - 整体 eval 精度
   - 训练成本
   - 部署成本
3. 必须先定义笛卡尔积平面的横轴、纵轴及其 slope 的现实意义。
4. 否则后面的 Pareto、knee、Bayes 都没有实义。

### 8.2 我这边修正后的正确表达

更准确的表述应该是：

1. **bathroom 只是诊断项，不是决策目标。**
2. **纵轴必须是单一综合 utility。**
3. **横轴必须是某种真实成本。**
4. **slope 必须表示真实的边际成本 / 边际收益交换。**
5. **只有能改变“整体 utility vs 成本”关系的参数，才值得进入主搜索。**

### 8.3 一句话总结

可以把这轮讨论的最终共识浓缩成一句话：

> **从 v25 之后开始，`floorplan_checker` 的研究不应再围绕任何单头指标展开；真正的研究对象是“整体 eval utility 与训练/部署成本之间的真实前沿”，而单头指标只负责解释这条前沿为什么会上升、停滞或失真。**

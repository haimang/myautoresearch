> 2026-04-25 | 前置：`updates/v20-roadmap.md`、`updates/v20.1-update.md`、`updates/v20.2-update.md`、`updates/v20.3-update.md`、`updates/v21-update.md`、`updates/v21-code-review.md`

## 1. 版本目标（一句话）

> **把 v20-roadmap 前半段已经建好的 point / campaign / stage / trajectory / recommendation 五层能力，收束成一套“可执行、可追溯、可迁移”的主动探索框架：在 gomoku 上完成路线图收口，在 v22 前把贝叶斯式 acquisition 层做成面向新 domain 复用的通用接口，而不是继续把 gomoku 绑定成唯一研究对象。**

这一定义与 `v20-roadmap.md` §6.6 的原始目标保持一致，但要根据**现在的代码现实**做一次更准确的落地解释：

1. **v21.1 仍然属于 Bayesian multi-objective frontier search。**
2. 但它不应该再被实现成“只为 gomoku 写一套重型、难迁移、只能在本 domain 证明有效的优化器”。
3. 它应该被实现成：
   - **基于现有 candidate pool 的 acquisition reranker**
   - **能把 recommendation 真正执行起来的闭环接口**
   - **能在 gomoku 历史数据上完成 replay / baseline 对比的证据层**
   - **能直接带到 v22 新 domain 的通用主动探索骨架**

## 3. 为什么现在的 v21.1 不应该简单等于“在 gomoku 上硬上重型 MOBO”

这是本文件最关键的判断。

### 3.1 roadmap 原文没有要求 v21.1 只能服务 gomoku

`v20-roadmap.md` §6.6 要求的是：

1. 引入 optimizer / acquisition 模块
2. 记录 surrogate snapshot、acquisition score、chosen point
3. 在固定预算下对比基线
4. 让 frontier 质量更好或不差
5. 所有 BO 决策可追溯

它要求的是**研究系统能力**，不是“只在 gomoku 上堆一个很重但不可迁移的实现”。

### 3.2 v22 将切换到完全不同的 domain

既然 v22 会转到与 gomoku 完全不同的 domain，那么 v21.1 的最佳价值就不是：

- 再为 gomoku 多写一套 domain-specific 调参器

而是：

- 把前半段 roadmap 已经建立的数据纪律
- 封装成一个对新 domain 仍然成立的 acquisition / execution / evidence 框架

### 3.3 当前架构天然更适合“candidate-pool acquisition”，而不是直接做 raw-space BO

当前系统已经把“可行候选生成”拆成两层：

1. **候选生成层**
   - `generate_point_candidates()`
   - `generate_branch_candidates()`
2. **候选排序层**
   - `recommend_for_campaign()`
   - score / rationale / ledger

因此 v21.1 最合理的方向不是推翻 selector，而是：

> **保留当前 domain-aware candidate generation；在其上叠加 Bayesian-style acquisition reranker。**

这有三个好处：

1. **与现有代码结构完全兼容**
2. **天然支持离散轴、stage 约束、branch reason、trajectory continuation**
3. **迁移到新 domain 时，只要换 search/stage/branch policy，不必重写 acquisition 主体**

## 4. 因此，v21.1 的直接任务应该是什么

v21.1 建议只做五件事。

### 4.1 Recommendation execution closure

把 v21 还停留在 API 级的 recommendation 闭环，提升成 **CLI / framework 一级能力**。

具体目标：

1. `sweep.py` 能直接执行 **accepted recommendation**
   - 支持 `new_point`
   - 支持 point 型 `seed_recheck`
2. `branch.py` 能直接执行 **accepted recommendation**
   - 支持 `continue_branch`
   - 支持 `eval_upgrade`
3. recommendation 在执行时自动完成状态流转
   - `accepted -> executed`
4. 结果自动回填 `recommendation_outcomes`
5. `analyze.py --recommendation-outcomes` 能直接看到真实执行证据

这一步完成后，v21 的 recommendation 才真正从“建议对象”变成“可执行研究动作”。

### 4.2 Frontier / recommendation / outcome 证据链闭环

把当前已经存在但尚未完全打通的证据链彻底接起来。

具体目标：

1. 每个 recommendation batch 必须绑定明确的 `frontier_snapshot_id`
2. 新增 `surrogate_snapshots`（或等价表）
   - 记录 surrogate / acquisition 计算所使用的数据窗口、目标定义、特征摘要、模型配置
3. 每条 recommendation 记录 acquisition 相关字段
   - `acquisition_name`
   - `acquisition_score`
   - 必要的 evidence / features 摘要
4. outcome summary 能回溯：
   - 当时看的是哪个 frontier snapshot
   - 用的是哪个 surrogate snapshot
   - 最终执行了哪个 chosen point / branch

这一步的意义是把 roadmap §6.6 中“可追溯 BO 决策”从口头要求变成 DB 事实。

### 4.3 Candidate-pool Bayesian acquisition layer

这是 v21.1 的核心算法层，但应当以**轻量、通用、可迁移**为原则。

建议做法：

1. 保留 `selector.py` 的候选生成职责不变
2. 新增 acquisition 模块，对候选池做二次排序
3. acquisition 的目标至少支持：
   - maximize: `win_rate`
   - minimize: `num_params`
   - minimize: `wall_time_s`
4. acquisition 不直接在 raw search space 上生成点，而是在**合法 candidate pool**上选择
5. 首版可以支持：
   - scalarized Bayesian score
   - uncertainty-aware Pareto utility
   - EHVI-like 近似启发式（若实现成本可控）

这一定义仍然满足 roadmap 的“Bayesian multi-objective frontier search”，但更贴合当前系统形态。

### 4.4 Replay benchmark / baseline comparison

由于 v22 将切换 domain，v21.1 不应把验证重心放在“再跑一轮很长的 gomoku 新训练”上，而应把重心放在**可复核的 replay 证据**上。

具体目标：

1. 基于现有 gomoku campaign / recommendation / outcome 历史做 replay benchmark
2. 在固定 expensive-action 预算下，对比：
   - heuristic selector baseline（v21）
   - acquisition reranker（v21.1）
   - random / naive baseline（可选）
3. 输出：
   - frontier hit rate
   - near-front hit rate
   - expensive action 数量
   - frontier quality 不差于基线的证据

这样做能满足 roadmap 的“同预算下优于或不差于基线”，同时避免在 gomoku 上继续扩大一次性工程投入。

### 4.5 v22 新 domain 的 handoff package

这一步不是 roadmap 原文的主目标，但它是**当前时点完成 roadmap 的必要收尾动作**。

具体目标：

1. 明确新 domain 接入所需的最小契约：
   - `search_space.json`
   - `stage_policy.json`
   - `branch_policy.json`
   - `selector_policy.json`
   - `acquisition_policy.json`（v21.1 新增）
2. 明确哪些能力是 domain-aware，哪些是 framework-generic
3. 给出最小 smoke 流程：
   - campaign -> recommend -> accept -> execute -> outcome -> acquisition summary

这样 v20-roadmap 的前半段就真正变成“已封箱、可带走”的研究骨架，而不是只在 gomoku 仓库里局部可用。

因为到这一步，roadmap 的六个节点会形成一条完整闭环：

1. **v20** 看见 point frontier

4. **v20.3** 让 continuation 成为 trajectory 事实
5. **v21** 让系统会基于事实推荐下一步
6. **v21.1** 让推荐变成 acquisition 驱动、可执行、可追溯、可复核、可迁移的主动探索闭环

也就是说：

> **v21.1 完成后，v20-roadmap 前半段不再只是“把每一层能力各写了一点”，而是真正形成从观察 → 组织 → 晋升 → 分叉 → 推荐 → acquisition 的一整条研究系统链路。**

## 11. 执行顺序与关键路径

建议按以下顺序推进：

1. **先补 execution integration**
   - 这是把 v21 从 API 级闭环升级为正式框架能力的最短路径
2. **再补 evidence lineage**
   - 没有 snapshot / score / chosen-point 绑定，acquisition 无法可信落库
3. **再做 acquisition engine**
   - 在现有 candidate pool 上做 reranking，而不是推翻 selector
4. **最后做 replay benchmark 与 v22 handoff**
   - 把 gomoku 作为收口证据，而不是继续扩大 gomoku 专属实现

关键路径判断：

> **v21.1 的 blocker 不是算法，而是 execution 与 evidence 的闭环。**

如果这两层不先打通，再先进的 acquisition 也只会停留在离线打分器。

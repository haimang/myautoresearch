# Update v8 Analysis - 关于训练优化是否偏离 autoresearch 原旨的中文分析

> 日期：2026-04-11
> 范围：围绕当前 MAG Gomoku 项目的训练优化设想，分析这些优化是否偏离原版 autoresearch 的原旨精神，是否仍然符合本项目的自我定位，以及这些优化的真实价值、代价与取舍是否值得。

---

## 1. 执行摘要

这一轮分析聚焦的核心问题不是“某个具体超参该不该改”，而是更基础的问题：

**当我们开始主动分析训练稳定性、评估波动、样本效率、网络规模、对手课程、loss 与 win rate 的关系时，我们是否已经在替代 autoresearch 的自动发现机制？这样的介入，是否会让项目失去原来的意义？**

本次分析的结论是：

1. **不会因为提出优化建议，就自动偏离 autoresearch 原旨。**
   原版 autoresearch 的核心并不是“人类绝不提供任何先验”，而是“先验必须接受固定 benchmark 和短周期实验的持续裁决”。

2. **真正的边界不在于是否有人提出建议，而在于谁拥有最后裁决权。**
   如果最后的保留、回退、升级、放弃，仍然由固定外部评估决定，那么这些建议仍然处在 autoresearch 的精神内部。

3. **当前最值得做的优化，多数属于“修复实验噪声”和“提升样本效率”，而不是“替代自动发现”。**
   例如更稳的 probe eval、更合理的训练强度、更大的 replay buffer、对称增强、适度扩容，这些都更像是在修补自动研究系统的测量和学习条件。

4. **真正有偏离风险的，是用内部指标取代外部指标。**
   例如：
   - 用训练 loss 代替 win rate 作为主要升级依据
   - 用强 reward shaping 取代固定外部 benchmark
   - 让复杂机制掩盖模型是否真的变强

5. **MAG Gomoku 已经不是上游 autoresearch 的极简镜像，而是一个朝“自对弈自动升级训练平台”演化的系统。**
   因此，一部分中等复杂度的工程升级，即便比上游更重，也未必违背本项目原旨，反而可能更符合本项目已经形成的实际路线。

总判断：

- 修复评估噪声、训练震荡、样本利用率不足：**高度值得做，且基本不偏离原旨**
- 引入混合对手、league、浅搜索 teacher：**属于第二阶段扩展，和项目路线对齐，但比原版更复杂**
- 用 loss 替代 WR、强 reward shaping：**当前不推荐，偏离风险高**

## 2. 问题背景

这一轮讨论的起点来自以下几个问题：

1. 当前训练中，win rate 在 probe eval 时波动明显，存在大起大落。
2. 当前 loss 会下降，但 loss 是否能代表模型真实棋力，存在疑问。
3. 提升网络参数、调整超参、加入对手课程、改 replay buffer、做对称增强，这些行为会不会变成人类过度主导？
4. 如果人类已经开始“设计方向”，是否就损害了 autoresearch 的自动发现价值？

这背后其实有一个更深层的问题：

**autoresearch 的意义，到底是“完全无先验”，还是“让先验接受自动实验闭环的反复验证”？**

---

## 3. 原版 autoresearch 的原旨精神到底是什么

### 3.1 原版并不是“零先验系统”

这是最需要澄清的点。

原版 autoresearch 当然不是一个完全没有先验的系统。恰恰相反，它一开始就预设了很多东西：

- 模型大致结构
- 优化器形式
- dataloader 逻辑
- 固定训练时长
- 固定验证协议
- 单一主指标 val_bpb
- train.py 是主要改动面

这些全都是强先验。

所以原版的真正核心不是“人类不设置任何结构和假设”，而是：

> 人类提供一个可运行、可比较、可迭代的初始研究壳；之后让自动实验在固定评估下反复裁决哪些改动应该保留。

### 3.2 原版真正强调的东西

如果把原版浓缩成几个原则，应该是：

1. **固定评估器**
   不轻易修改 benchmark。

2. **单一主指标**
   所有实验最终围绕一个核心指标收敛。

3. **短周期、高频实验**
   单次实验足够短，便于自动循环。

4. **主要改动面尽量集中**
   让 train.py 成为主要研究表面。

5. **结果由实验决定，不由想象决定**
   一个改动是否保留，不取决于它听起来多高级，而取决于它是否在固定 benchmark 下真的更好。

### 3.3 因此，原版的“自动发现”并不排斥人类提出候选方向

它排斥的不是“人类提出想法”，而是“人类用主观判断越过实验裁决”。

换句话说：

- **允许人类提出更好的实验设计和候选超参**
- **不允许人类绕过 benchmark，自说自话宣布成功**

## 4. MAG Gomoku 当前语境下，这个问题怎么理解

### 4.1 我们现在讨论的优化，本质上分三类

#### 第一类：降噪类优化

例如：

- 增大 probe games
- 双轨评估
- 固定 opening suite
- 平滑 win rate 的解释方式

这类改动的本质不是替代 discovery，而是**修复测量系统**。

#### 第二类：稳态训练类优化

例如：

- 调整学习率
- 调整 TRAIN_STEPS_PER_CYCLE
- 增大 replay buffer
- 改 replay buffer 的采样策略
- 适度增加模型容量

这类改动本质上是**优化 train.py 里的可搜索空间**。

#### 第三类：路线扩展类优化

例如：

- 混合对手池训练
- checkpoint league
- 浅搜索 teacher signal
- loss 触发导出 opponent

这类改动开始从“优化现有系统”走向“升级系统范式”。

### 4.2 这三类里，只有第三类的一部分会触碰原旨边界

原因很简单：

- 第一类是在修 benchmark 信号
- 第二类是在调 train.py 的实验对象
- 第三类才开始改变训练生态和演化路径

所以，并不是“任何优化都会偏离原旨”。

真正值得警惕的，是第三类里面那些会替代外部评估的做法。

## 5. 对齐分析：哪些优化不偏离，哪些会偏离

下面按维度做更正式的判断。

### 5.1 与原版 autoresearch 精神的对齐标准

判断标准如下：

1. 是否保留固定外部 benchmark
2. 是否保留单一主要成功指标
3. 是否保留短周期实验的可比性
4. 是否仍由实验结果裁决，而不是由主观设计裁决

### 5.2 与 MAG Gomoku 项目原旨的对齐标准

判断标准如下：

1. 是否服务于 Apple Silicon 本地高频训练
2. 是否服务于 Gomoku 自对弈升级路线
3. 是否加强“可持续训练、可追踪、可回放、可升级”的系统定位
4. 是否没有把项目推向完全不同的范式

### 5.3 与 self-play 自动升级路线的对齐标准

判断标准如下：

1. 是否让模型在自动对弈中更稳定地提升
2. 是否让对手生态更合理地进入训练闭环
3. 是否减少自循环和分布塌缩
4. 是否不会破坏外部强度判断

## 6. 具体建议的对齐分析

### 6.1 更稳的评估：probe 更大、双轨评估、锚点评估

这是本轮最不具争议的一类建议。

原因：

- 原版最强调固定 benchmark
- 当前项目最大的问题之一就是 probe 波动太大
- 评估越不稳，自动研究越容易做错保留/回退决策

因此，这一类建议不是在背离 autoresearch，而是在**保护 autoresearch 的实验裁决能力**。

结论：

- 与原版高度对齐
- 与项目高度对齐
- 强烈推荐优先做

### 6.2 训练稳定性优化：学习率、每 cycle 步数、buffer 大小、适度扩容

这类建议同样不属于偏离原旨。

原因：

- 原版本来就允许 train.py 中的这些内容成为搜索对象
- 当前项目的 train.py 也明确把这些视为可实验的变量
- 它们不改变外部 benchmark，只改变模型在固定 benchmark 下学习的方式

因此，这类建议依然处于 autoresearch 的核心允许范围内。

结论：

- 与原版基本对齐
- 与项目高度对齐
- 应该做，而且是最值得做的基础层优化之一

### 6.3 对称增强

对称增强看起来像“人为先验”，但这是任务的真实数学结构。

五子棋天然具备旋转和镜像对称。利用这些不变性，不是拍脑袋加一个奖励，而是把问题本身的结构写进训练数据使用方式。

从原版 autoresearch 视角看，这类改动不是“主观投机”，而更像是：

- 使用更好的 dataloader
- 使用更符合任务结构的数据增强

结论：

- 与原版高度对齐
- 与项目高度对齐
- 价值高，风险低，是最应该优先尝试的改动之一

### 6.4 混合对手池训练

这条建议比前面几条更复杂。

从原版极简哲学看，它确实开始偏离，因为它不再是最简单的“当前模型纯 self-play”世界。

但从项目原旨看，这反而可能是高度对齐的。

因为 MAG Gomoku 现在已经：

- 有 checkpoint
- 有 opponent registry
- 有 tracker
- 有 replay
- 有阶段推进概念

也就是说，项目事实本身已经不再只是原版三文件最小壳，而是一个朝“自对弈自动升级平台”演化的系统。

在这种前提下，混合对手池训练反而是把你们已经有的 opponent 机制真正接入训练主线。

结论：

- 对上游极简主义有中度偏离
- 对项目原旨高度对齐
- 应作为第二阶段升级，而不是第一步

### 6.5 loss 触发导出 opponent

这条建议要细分。

#### 情况 A：loss 仅作为辅助导出信号

如果 loss 只是表示：

- “现在模型可能已经进入一个较稳定阶段”
- “可以导出一个候选 checkpoint 作为 sparring opponent”

那么它是可接受的。因为它没有代替 benchmark，只是帮助系统产生更多训练生态。

#### 情况 B：loss 替代 WR 做升级判据

如果 loss 成为主裁判，那就明显偏离原旨。

原因：

- loss 是内部拟合指标
- win rate 是外部强度指标
- 用内部指标替代外部 benchmark，会让系统失去 ground truth

结论：

- 作为辅助信号：可以
- 作为主升级标准：不推荐

### 6.6 强 reward shaping

这一类建议偏离风险较高。

原因在于：

- 奖励项一多，目标函数就被人工主观化
- 模型可能学会讨好奖励代理，而不是真正变强
- benchmark 的解释性会下降

原版 autoresearch 的精神是：固定外部评估，内部结构可以改，但外部成功标准不能被内部代理目标取代。

而强 reward shaping 很容易让系统从“优化真实对局强度”滑向“优化人类写的战术打分”。

结论：

- 当前阶段不推荐优先做
- 偏离原旨风险较高

## 7. 这些优化的价值是否真实存在

这个问题必须辩证看。

### 7.1 支持这些优化存在价值的理由

#### 1. 它们在修复真实问题，而不是假想问题

当前项目中已经观察到：

- probe WR 大幅波动
- loss 虽然下降，但和棋力提升关系不完全稳定
- 自对弈训练分布具有非平稳性
- 当前 opponent 机制更多停留在评估层，而不是训练层

这说明问题是真实存在的，不是纸上谈兵。

#### 2. 它们提升的是实验闭环质量

更好的评估、更稳的训练、更高的样本效率，最终目的不是让人类替代自动发现，而是让自动发现少被噪声误导。

#### 3. 它们符合 Gomoku 这个任务的物理结构

例如对称增强，并不是任意发挥，而是利用任务先验。

#### 4. 它们适配当前 Apple Silicon 运行环境

许多建议在 M3 Max 上都是现实可行的，不需要把系统直接推向过重的搜索范式。

### 7.2 反对这些优化过早推进的理由

#### 1. 会压缩自动探索空间

如果人类一开始就认定某种更宽、更深的网络一定更好，或者强行给出太具体的课程机制，可能会减少 agent 自己发现不同方向的机会。

#### 2. 复杂度上升会削弱系统可解释性

一旦系统引入太多课程、对手池、league、teacher signal，就会更难明确某个效果到底来自哪里。

#### 3. 容易把项目推向“人工主导调参”

如果每一步都由人类手工策划，而不是让固定 benchmark 做裁决，autoresearch 的意义就真的会被掏空。

## 8. 辩证判断：我们是不是在越俎代庖

结论是：

**目前提出的大部分建议，不构成越俎代庖。**

原因如下：

1. 我们提出的是候选假设，不是最终裁决。
2. 这些候选假设仍然需要在固定 benchmark 下验证。
3. 这些建议主要是在修复当前系统的测量和学习噪声。
4. 它们没有试图让人类直接宣布“这个方向一定成功”。

真正越俎代庖的情形是：

- 人类先写一套很重的主观奖励
- 再用内部指标宣布系统进步
- 不再关心固定 benchmark 的反馈

那才是真正背离 autoresearch。

所以关键不是“人类有没有提建议”，而是：

> 人类的建议是否仍然服从于外部 benchmark 的实验裁决。

只要这一点还在，项目就没有失去 autoresearch 的核心意义。

## 9. 推荐思路：如何在不背离原旨的前提下推进升级

### 9.1 第一阶段：只做“修 benchmark 和修训练稳定性”的建议

优先建议：

1. 增大 probe games
2. 引入双轨评估
3. 适度降低更新激进度
4. 增大 replay buffer
5. 加入对称增强
6. 适度扩容模型，例如从 6x64 测试到 8x64 或 6x96

这一阶段的特点：

- 不改变项目哲学
- 不引入新的主指标
- 主要是让当前自动研究循环更可信

### 9.2 第二阶段：开始让 opponent 进入训练主线

优先建议：

1. 混合对手池训练
2. 更合理的 replay buffer 采样
3. loss 仅作为辅助导出 opponent 的触发器

这一阶段的特点：

- 开始向自对弈自动升级平台演化
- 仍然保留固定 benchmark 作为主裁判

### 9.3 第三阶段：再考虑更复杂的升级机制

例如：

1. checkpoint league
2. Elo/rating 驱动的对手课程
3. 局部浅搜索 teacher

这一阶段的特点：

- 已经开始明显偏离上游极简哲学
- 但可能更符合 Gomoku 长期路线

### 9.4 当前不建议优先走的路线

1. loss 完全替代 WR
2. 强 reward shaping
3. 大规模重搜索作为训练默认路径

## 10. 最终 Verdict

### 10.1 关于“是否偏离 autoresearch 原旨”

最终判断：

**不会因为分析训练问题、提出结构化优化建议，就自动偏离 autoresearch 原旨。**

只要以下几点仍然成立：

1. 固定 benchmark 仍然存在
2. 最终保留/回退仍由实验决定
3. 建议只是候选方向，不是免验证真理

那么这些优化仍然属于 autoresearch 精神允许的范围。

### 10.2 关于“我们的优化价值是否真的存在”

结论：

**价值真实存在，而且主要体现在以下几个方面：**

1. 让实验结果更可信
2. 让训练过程更稳
3. 让样本利用率更高
4. 让 self-play 自动升级路线更自然

这些不是“工程洁癖式优化”，而是会直接影响模型是否真的能更稳定地变强。

### 10.3 关于“取舍是否值得”

最终取舍建议如下：

#### 值得立即做

- 更稳的评估
- 更稳的训练强度
- 更大的 replay buffer
- 对称增强
- 中等幅度扩容

#### 值得作为第二阶段做

- 混合 opponent 课程
- 更聪明的样本采样
- loss 辅助导出 opponent
- checkpoint league

#### 当前不值得优先做

- loss 取代 WR
- 强 reward shaping
- 重搜索训练主线化

### 10.4 最终总判断

本轮分析的最终 verdict 是：

> 当前最正确的升级方式，不是压制人类分析，也不是让人类替代自动发现；而是在固定 benchmark 的约束下，优先修复实验噪声、提升样本效率、稳住训练制度，再逐步把 opponent 生态接入 self-play 自动升级路线。

换句话说：

- **不做优化，会浪费 autoresearch 的搜索预算**
- **做错优化，会背离 autoresearch 的 benchmark 精神**
- **做对优化，则是在帮助 autoresearch 在 Gomoku 场景真正落地**

因此，合理的结论不是“不要分析”，而是：

**分析可以做，优化可以做，但 benchmark 不能丢，实验裁决不能被内部代理指标替代。**

---

## 11. 简版推荐清单

如果只保留最实用的结论，建议如下：

1. 先修评估：probe 更稳，anchor eval 更清晰
2. 先修训练制度：学习率、buffer、每 cycle 更新步数
3. 先加对称增强
4. 再做中等扩容，而不是暴力扩容
5. 再引入 opponent 课程，而不是直接用 loss 取代 WR
6. 不要优先走强 reward shaping 路线

这条路线既尽量保留了原版 autoresearch 的实验精神，也更符合 MAG Gomoku 当前已经演化出来的项目实际。

## 12. Copilot 执行者分析与 v8 升级规划

> 以下由 Copilot（执行者）在审阅 GPT 的 v8 分析后撰写。分析基于当前代码库实际状态（commit `609b677`，v7 全部完成后）。

### 12.1 对 GPT v8 分析的总体评价

GPT 的 v8 分析是一篇**元层面的哲学审查**，而不是一份技术改进清单。它的核心问题是：

> 我们反复优化训练系统，是否已经在替代 autoresearch 应有的自动发现机制？

这个问题提得好，时机也对——v5 引入了 NN 对手系统，v6 加了基准标注和分析工具，v7 做了对称增强、动态衰减、缓冲区加权。到了 v8，确实需要停下来审视一下方向。

**GPT 给出的核心判断我完全同意：**

1. 降噪（评估精度）和稳态（训练动力学）类优化，不构成越位
2. 路线扩展类优化（混合对手、课程训练），需要谨慎但与项目实际路线对齐
3. 用 loss 替代 WR、强 reward shaping——当前不应该做

但这篇分析有一个关键盲区：**它不知道 v7 已经落地了哪些改动。**

### 12.2 GPT "立即推荐" vs 我们的实际状态

GPT 在第 9.1 节给出的"第一阶段"推荐清单：

| GPT 推荐 | 当前状态 | 说明 |
|----------|---------|------|
| 增大 probe games | ✅ v7 已完成 | 50→100 games, window 3→5 |
| 降低更新激进度 | ✅ v7 已完成 | LR 1e-3→5e-4, steps 50→30/cycle |
| 增大 replay buffer | ⚠️ 未做 | 当前 50K，可调至 100K |
| 对称增强 | ✅ v7 已完成 | D4 八折在线增强，零内存开销 |
| 中等扩容 (6x64→8x64 或 6x96) | ❌ 未做 | 需要 CLI 参数化 + 验证 |
| 双轨评估 / 锚点评估 | ❌ 未做 | 有价值但复杂度较高 |

**结论：GPT 六条"立即"推荐中，三条已在 v7 落地。v8 真正需要做的新工作只有三条。**

### 12.3 GPT "第二阶段"推荐的可行性评估

| GPT 推荐 | 可行性 | v8 是否纳入 |
|----------|--------|------------|
| 混合对手池训练 | 中高——已有 opponent 注册表、`_nn_opponent_move()` | ✅ 作为 v8 核心工作 |

| checkpoint league | 过重——当前阶段不需要 Elo 系统 | ❌ 暂不做 |

### 12.4 代码审计：v8 实施的具体约束

通过对代码的详细审查，我确认以下实施约束：

**模型容量不可 CLI 配置：**
- `GomokuNet(num_blocks=NUM_RES_BLOCKS, num_filters=NUM_FILTERS)` 构造器支持参数
- `load_model()` 同样支持，但 `train()` 中直接调用 `GomokuNet()` 使用模块常量
- CLI 中无 `--num-blocks` 或 `--num-filters` 参数
- 需要：添加 CLI 参数，传入 `GomokuNet()` 和 `load_model()`

**自对弈函数不支持对手：**
- `run_self_play(model, num_games, temperature)` 纯自对弈
- 所有 `num_games` 局都是同一个模型 vs 自己
- 对手模型仅在 `_quick_eval()` 评估时使用，不参与训练数据生成
- 需要：在自对弈中混入一定比例的 vs-opponent 对局

**缓冲区大小硬编码：**
- `REPLAY_BUFFER_SIZE = 50000` 是模块常量
- 训练循环中直接引用，无 CLI 覆盖
- 需要：添加 `--buffer-size` 参数（简单）

**超参字典已记录容量信息：**
- `hyperparams` 包含 `num_res_blocks` 和 `num_filters`
- 变更容量后，tracker 已能正确记录，无需额外迁移

### 12.5 v8 应该做什么——以及不做什么

#### 做（P0-P3，按优先级排列）

**P0: 模型容量 CLI 参数化**

添加 `--num-blocks` 和 `--num-filters` 参数。改动面小但价值大：
- 允许用户用 CLI 快速测试不同容量，不需要改源码
- 让不同容量的对比实验可由 analyze.py 查询（hyperparams 已记录）
- 修改约 5 处：`parse_args()`、`train()` 中 `GomokuNet()` 调用、`load_model()` 传参、`resume` 场景的兼容

关于默认值：保持 6x64（~564K 参数），但推荐用户测试 8x64（~750K）或 6x96（~1.2M）。

**P1: Replay Buffer CLI 参数化**

添加 `--buffer-size` 参数。改动 3 处：
- `parse_args()` 添加参数
- `train()` 中用 CLI 值替代常量
- 写入 `hyperparams` dict

默认保持 50000，但推荐大容量模型使用 100000。

**P2: 混合对手训练**

这是 v8 的核心架构变更。设计思路：

- 添加 `--train-opponent` 参数（可指定已注册对手的 alias）
- 添加 `--opponent-mix` 参数（0.0-1.0，默认 0.2 = 20% 对局 vs 对手）
- 修改 `run_self_play()` 或在其旁边新增 `run_mixed_play()`：
  - 将 `num_games` 按比例分为 self-play 和 vs-opponent 两部分
  - vs-opponent 部分：模型下黑/白各半，对手用 `_nn_opponent_move()` 响应
  - 两部分的训练数据统一进入 replay buffer
- TUI 中显示对弈构成信息（self-play vs opponent）

这与 `--eval-opponent` 的区别在于：`eval-opponent` 只影响评估，`train-opponent` 影响训练数据生成。两者可以独立设置，也可以指向同一个对手。

**P3: analyze.py 容量对比支持**

扩展现有的 `--compare` 和 `--runs` 命令，让它们显示模型容量信息（blocks x filters, params）。当用户测试不同容量时，这提供了直观的对比视角。

#### 不做

| 内容 | 原因 |
|------|------|
| 双轨评估 / 锚点评估 | 复杂度高，v7 的 100 games + window 5 已经显著降噪。收益递减。 |
| loss 触发导出 opponent | loss 与棋力的关联尚未经验证。MCTS=0 下 loss 的可信度有限。 |
| checkpoint league / Elo | 对手池当前只有 L0，过早引入联赛机制是过度工程。 |
| 强 reward shaping | 与 GPT 和我们共同的立场一致：不做。 |
| 模型架构变更 | 不改 ResBlock 结构或 head 设计。容量调整仅限 blocks/filters 数量。 |

### 12.6 关于"偏离度"的独立判断

GPT 的三分类框架（降噪 / 稳态 / 路线扩展）很清晰。我补充一个执行者视角的判断标准：

> **一项改动是否偏离原旨，不取决于它的复杂度，而取决于它是否绕开了外部 benchmark。**

以 v8 P2（混合对手训练）为例：
- 它确实比原版 autoresearch 更复杂
- 但训练数据中对手比例是可追踪的（hyperparams 记录 `opponent_mix`）
- 评估仍然用固定 benchmark（probe vs L0 或指定对手）
- 结果仍由 WR 裁决，不由 loss 或其他内部指标裁决

因此它属于"在 autoresearch 框架内扩展训练生态"，不属于"替代 autoresearch"。

与之对比，如果我们做了"loss 低于阈值就自动升级"——那才是真正的越位，因为裁判从外部对局变成了内部拟合。

### 12.7 与 v7 的关系

v7 解决的是**训练信号质量**：对称增强让每个样本产生 8 倍价值，缓冲区加权让近期样本优先学习，温度衰减让自对弈数据更真实，动力学调整让更新更平滑。

v8 要解决的是**训练生态丰富度**：
- 模型容量可调 → 允许更大的模型充分利用 v7 提升后的训练质量
- 缓冲区可调 → 匹配更大容量模型的数据需求
- 混合对手训练 → 打破纯自对弈的分布塌缩，引入外部对弈信号

这两层是递进关系：v7 修内功（信号质量），v8 拓外延（训练生态）。

### 12.8 v8 执行计划摘要

```
P0  模型容量 CLI 参数化     --num-blocks, --num-filters    改动~30行
P1  缓冲区 CLI 参数化       --buffer-size                  改动~10行
P2  混合对手训练            --train-opponent, --opponent-mix 改动~120行（核心）
P3  analyze.py 容量对比     --runs/--compare 显示容量信息    改动~20行
```

依赖关系：P0 和 P1 互相独立，均无前置依赖。P2 依赖 P0（因为对手模型可能是不同容量的）。P3 依赖 P0。

验证计划：
1. P0+P1 完成后：用 8x64 跑一次 500-game 快速验证，确认参数正确传播
2. P2 完成后：用 L0 作为 train-opponent，20% mix，跑 2000-game 验证
3. 全部完成后：完整训练到 80% WR，对比 v7 基线

### 12.9 对 GPT 的最终回应

GPT 问的核心问题是："我们是不是在越俎代庖？"

我的回答是：**到目前为止，没有。**

证据：
- v5-v7 的所有改动，WR 仍然是唯一裁判
- 训练参数变化全部可追踪（hyperparams dict + tracker DB）
- 没有任何内部指标取代了外部对局评估
- 模型是否"真的变强"，仍由固定 opponent 的 probe eval 裁决

v8 的混合对手训练会进一步拉大与原版 autoresearch 的距离——但这是**有意识的、可追踪的、可回退的**演化，而不是无声的偏离。只要 tracker.db 还在记录每一局的评估结果，我们就还在 autoresearch 的精神之内。

**改动文件**: `src/train.py`, `src/tracker.py`

核心实现：
- 新增 `run_opponent_play()` 函数（~65 行）：使用单局 Board API（非 BatchBoards），模型与对手交替执黑/执白，仅从模型视角收集训练数据，应用与自对弈相同的温度衰减

- 自对弈和对手对弈数据进入同一 replay buffer
- TUI 和 summary 显示 `Train-opp: L0 (mix=20%)`

对手架构追踪：
- `tracker.py` v8 migration：opponents 表新增 `num_res_blocks INTEGER` 和 `num_filters INTEGER` 列
- `register_opponent()` 签名扩展，存储架构参数
- `_handle_register_opponent()` 从源 run 的 DB 记录继承架构信息
- eval-opponent 和 train-opponent 加载时读取 DB 中的架构参数，传递给 `load_model()`

### P3: analyze.py 容量对比

**改动文件**: `src/analyze.py`

- `cmd_runs()` 新增 Model 列，显示 `NxF` 格式（如 `6x64`）
- `cmd_compare()` 新增 Model 行，显示 `NxF (params)` 格式

### Bug 修复

代码审查发现 3+1 个 bug，全部已修复：

1. **Critical — 零自对弈守卫**: 当 `parallel_games=1` 且 `opponent_mix>0` 时，`n_self` 可能为 0，导致自对弈被完全跳过。修复：添加 `parallel_games >= 2` 前置条件和 `max(1, n_self)` 保护。

2. **High — 对手架构不匹配**: 加载非默认架构模型（如 8×64）作为对手时，使用默认参数（6×64）初始化，导致 `ValueError: shape mismatch`。修复：opponents 表增加架构列，加载时读取并传递正确参数。

3. **Medium — analyze.py "NonexNone" 显示**: `a.get('num_res_blocks', '?')` 当键存在但值为 NULL 时返回 None 而非默认值。修复：改为 `a.get('num_res_blocks') or '?'`。

4. **High — subprocess eval 架构不匹配**: `_subprocess_eval()` 构造的子进程代码调用 `prepare.evaluate_win_rate()`，后者以默认架构加载模型。对于非默认架构（8×64）会崩溃。修复：在子进程代码中 monkey-patch `train.load_model`，在 prepare 导入前注入正确架构参数。`_subprocess_eval()` 和 `_do_checkpoint()` 签名均扩展以接收 `num_blocks`/`num_filters`。

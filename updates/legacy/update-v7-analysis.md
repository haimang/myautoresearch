# Update v7 Analysis - Training Stability, Self-Play Upgrade Path, and Alignment Verdict

> Date: 2026-04-11
> Scope: formalize the latest technical discussion around loss-driven opponent promotion, reward design, win-rate volatility, training efficiency, and alignment with both upstream autoresearch and MAG Gomoku's own intended direction.

## 1. Executive Summary

This document records the latest analysis of how MAG Gomoku should evolve its training strategy.

The central question of this round was not only whether the model can be made stronger, but whether the proposed changes remain aligned with:

- the original autoresearch spirit
- this project's own intended philosophy
- the current self-play automatic upgrade direction

The main conclusion is:

1. The current bottleneck is not a lack of reward.
   The project is not primarily underpowered because it lacks dense reinforcement signals. It is underpowered because the training signal is still noisy, self-referential, and operationally unstable.

2. The current win-rate volatility is expected from the present design.
   Large swings such as 30% to 68% between probe evaluations are not surprising under the current setup. They arise from small evaluation sample sizes, non-stationary self-play data, aggressive update jumps, and the absence of a stable opponent curriculum in the training loop.

3. The highest-value improvements are conservative and signal-focused.
   The most aligned and effective upgrades are not strong reward shaping or loss-only promotion rules. They are:
   - better measurement
   - better sample efficiency
   - more stable training distribution
   - gradual introduction of opponent-based curriculum inside training

4. Loss should not replace win-rate as the main upgrade judge.
   Loss can be used as a secondary operational signal, but it should not become the main progression trigger or replace a stable benchmark.

5. The best path forward is staged.
   The recommended path is:
   - first stabilize evaluation and training dynamics
   - then strengthen self-play with mixed opponents and improved data usage
   - only later add league logic, auxiliary teachers, or more advanced curriculum machinery

## 2. Problem Statement

The discussion that motivated this document centered on four recurring observations:

1. Probe win rate fluctuates violently.
   In practice the model may show one probe around 30% and the next one 15 cycles later around 68%.

2. Loss can fall even when real strength remains unclear.
   This raised the idea of using a loss threshold such as 0.8 to automatically export a checkpoint as a new opponent, then continue training against it.

3. It is unclear whether more reward is needed.
   The question arose whether stronger or denser reward shaping would make training more efficient.

4. It is necessary to check alignment.
   Any proposal should be evaluated not only by technical usefulness, but also by whether it remains faithful to both upstream autoresearch and MAG Gomoku's own stated direction.

## 3. Current Technical Reality

### 3.1 The current training setup is not classic RL

The project currently behaves more like self-play supervised bootstrapping than full policy-gradient reinforcement learning.

Key characteristics:

- self-play games are generated by the current model
- the policy target is the distribution sampled from the current model at training time
- the value target is the final game result from each player's perspective
- the combined loss is policy cross-entropy plus value MSE

This means the system is currently strongest when:

- sample quality is good
- training distribution is stable enough
- the policy target contains genuinely useful information

And it is weakest when:

- the model mostly teaches itself from low-quality self-generated targets
- the replay buffer drifts too quickly
- evaluation is noisy and over-interpreted

### 3.2 The opponent mechanism exists, but mostly for evaluation

The repository already supports registering checkpoints as named NN opponents.
However, today this mechanism mainly participates in evaluation or manual workflows.

The training core still depends overwhelmingly on pure self-play with the current model playing itself.

That means the project has the beginnings of a curriculum system, but not yet a mature training curriculum.

### 3.3 Win-rate volatility is a structural consequence, not necessarily a bug

Large evaluation swings come from several factors acting together:

1. small probe sample counts
2. non-stationary self-play data
3. policy targets generated by the current model itself
4. aggressive movement of the replay distribution between probes
5. no stable opponent mix inside training

Therefore, volatility is not just measurement noise. It is both:

- noisy measurement
- real instability in the training process

## 4. Why Loss Is Not Enough

The idea under discussion was:

- when training loss drops below a threshold such as 0.8
- automatically export a checkpoint
- register it as an opponent
- continue training against that opponent
- repeat indefinitely

This is attractive because it seems automatic and elegant.

However, using loss as the primary upgrade trigger is not recommended.

### 4.1 Loss is not a direct strength metric

In this repository, loss measures how well the model fits the data it is currently being fed.
It does not directly measure external playing strength.

The gap matters because:

- the data distribution is generated by the current model
- the policy target reflects current model behavior rather than an external gold standard
- the value target is sparse and only arrives at the end of each game

So low loss can mean:

- better understanding of useful play patterns
- or merely better self-consistency on a narrow or drifting distribution

### 4.2 A fixed loss threshold is not portable across training regimes

The absolute value of loss depends on:

- architecture
- learning rate
- buffer composition
- policy/value weighting
- temperature
- training schedule

Therefore a threshold such as 0.8 is not a stable semantic boundary.

### 4.3 Loss-only promotion risks self-referential collapse

If every low-loss checkpoint becomes the new opponent, the training process can become trapped in a short self-referential loop:

- learn a pattern
- fit it well
- export it
- train against a very similar model
- fit that distribution too
- repeat

This may improve self-consistency without improving broad external strength.

### 4.4 Loss-only promotion weakens benchmark integrity

The repository currently defines real progress through win rate against stable opponents.
If loss replaces that, the project loses a clean external anchor.

## 5. Why Win Rate Swings So Hard

The user observed that probe win rate can jump from around 30% to around 68% between probe intervals.

This is plausible under the current design.

### 5.1 Small-sample evaluation noise

When only 50 probe games are used, the standard error is large enough that observed win rate can move significantly even if the true strength changed less dramatically.

This is especially noticeable when the model is not overwhelmingly stronger than the benchmark.

### 5.2 Long gap between probe checkpoints

Fifteen cycles can contain many training updates. Two successive probes are often evaluating meaningfully different parameter states and data distributions.

### 5.3 Non-stationary self-play distribution

Because the model generates its own data, each new generation of training data changes what the model is being asked to fit.

### 5.4 Self-play target quality is still limited

Without a stronger teacher, search signal, or opponent mix, the model is mostly learning from its own past behavior. This increases instability.

### 5.5 Benchmark and curriculum are currently not separated enough

Probe evaluation and training evolution are both influenced by a changing internal distribution, but the project still interprets probe results as if they were a fully stable benchmark.

## 6. Strategic Principle

The highest-value recommendation from this discussion is:

> The project does not primarily need more reward. It needs better signal quality and better control of distribution shift.

This is the key lens through which all later recommendations should be read.

## 8. Alignment Analysis by Recommendation

### 8.1 Increase probe games and introduce dual-track evaluation

This suggestion is highly aligned with upstream autoresearch because it strengthens benchmark reliability without changing the learning objective.

It is also highly aligned with MAG Gomoku because the present probe system is too noisy to serve as a stable upgrade judge.

It supports self-play evolution by making promotion and checkpoint decisions less erratic.

### 8.2 Use fixed opening suites for anchor evaluation

This preserves the spirit of a stable benchmark while improving comparability between checkpoints. It introduces minor complexity, but the gain in signal reliability is meaningful.

### 8.3 Reduce aggressive update jumps

This is one of the safest recommendations. It does not alter the route, only the smoothness of movement along that route. It strongly aligns with both upstream discipline and local practicality.

### 8.4 Add board symmetry augmentation

This is the cleanest high-value improvement.

Why:

- it preserves the current training paradigm
- it does not weaken benchmark integrity
- it improves sample efficiency substantially
- it fits the geometry of Gomoku perfectly

It is the strongest overall recommendation in this analysis.

### 8.5 Introduce mixed-opponent training

This suggestion is somewhat less aligned with the strict upstream minimalism because it expands training ecology beyond pure self-play.

However, it is extremely aligned with MAG Gomoku's own intended route. If the project truly wants self-play automatic evolution, then the opponent mechanism must eventually become part of training rather than remaining mostly an evaluation accessory.

### 8.6 Improve replay-buffer sampling

This remains within the core training philosophy and improves signal quality without requiring a conceptual rewrite. It is more engineering-heavy than symmetry augmentation, but still very reasonable.

### 8.7 Add shallow search or shallow minimax teachers

This suggestion provides stronger supervision and may meaningfully improve policy learning. However, it is a larger philosophical and implementation step. It is suitable as a second-stage improvement, not as the first corrective action.

### 8.8 Use checkpoint league or rating

This is strongly aligned with self-play upgrade logic but less aligned with upstream simplicity. It becomes attractive once the project chooses to invest in a genuine automatic opponent ecosystem.

### 8.9 Use loss threshold only as an auxiliary export trigger

This is acceptable if used narrowly.

Recommended use:

- not as a benchmark replacement
- not as the final promotion judge
- only as a lightweight operational heuristic for candidate opponent export

### 8.10 Replace win rate entirely with loss

This is poorly aligned with both upstream benchmark philosophy and the current project's stated notion of progress. Loss can support operations, but it should not define strength.

### 8.11 Strong dense reward shaping

This is the wrong primary move for the current stage. It adds target-design complexity before the project has solved more fundamental signal-quality problems.

### 8.12 Heavy full-search training

This moves the project toward a different system class entirely. It may be interesting in the future, but it does not fit the current local high-iteration Apple Silicon strategy.

## 9. What Should Be Done First

The recommendations naturally group into four tiers.

### Tier A: Immediate, low-risk, highly aligned

These should be prioritized first.

1. Add board symmetry augmentation
2. Increase probe games and introduce dual-track evaluation
3. Reduce aggressive update jumps

These changes preserve the project's spirit while directly attacking the most visible current problem: unstable win-rate progression.

### Tier B: Strong route extensions that remain aligned with the project

These should follow once basic stability improves.

1. Introduce mixed-opponent training
2. Improve replay-buffer sampling
3. Add fixed opening-suite anchor evaluation

These are more ambitious than Tier A, but still very consistent with the project's own self-play upgrade direction.

### Tier C: Mid-term system upgrades

These should come after the basic training loop is more stable.

1. Add shallow search or shallow minimax teacher signals
2. Introduce checkpoint league or rating-based progression
3. Optionally use loss thresholds as auxiliary candidate-export triggers only

### Tier D: Not recommended as current primary direction

1. Replace win rate with loss
2. Lead with strong reward shaping
3. Move directly to heavy search everywhere

## 10. Formal Verdict

### 10.1 Does the proposed path deviate from autoresearch spirit?

The recommended path deviates only mildly if the project focuses first on Tier A and Tier B.

Why:

- benchmark discipline is preserved
- the main metric remains external win rate
- training remains centered on improving the core learning loop
- complexity is added only where it improves stability and comparability

The larger deviations begin only when moving into league systems, stronger teachers, or heavy curriculum infrastructure.

### 10.2 Does the proposed path align with MAG Gomoku's own original spirit?

Yes, strongly.

MAG Gomoku is no longer just a literal upstream autoresearch clone. It is a Gomoku-specific autonomous training platform on Apple Silicon.

Under that framing, the recommended path is highly aligned because it:

- strengthens self-play instead of abandoning it
- improves artifact quality and benchmark reliability
- helps the system become a more credible autonomous training platform
- does not force a premature rewrite of the project into a very different paradigm

### 10.3 Does it align with self-play automatic upgrade direction?

Yes, but only if interpreted correctly.

The correct interpretation is:

- do not let loss replace real progress metrics
- do let opponent mechanisms gradually enter the training curriculum
- do evolve toward a richer opponent ecology over time

This means the right path is not "loss-first promotion".
It is "stability-first self-play evolution".

### 10.4 Final decision statement

The final recommendation is:

> Do not prioritize reward shaping or loss-only promotion. Prioritize stronger measurement, smoother training dynamics, symmetry-based sample efficiency, and gradual integration of mixed opponents into the training loop.

In practical terms:

1. Fix instability first.
2. Improve sample efficiency second.
3. Add controlled opponent curriculum third.
4. Introduce stronger teacher signals or league logic only after the earlier layers are working.

## 11. Final Upgrade Verdict

The most defensible upgrade verdict is:

- Immediate direction: conservative upgrade
- Medium-term direction: project-aligned expansion
- Rejected direction: benchmark replacement by internal training loss

Put more plainly:

The project should continue along the self-play automatic upgrade route, but with a more disciplined foundation.

The correct next generation of MAG Gomoku is not a reward-shaped system chasing arbitrary internal signals.
It is a more stable, more sample-efficient, and more benchmark-aware self-play system.

That route remains compatible with the original autoresearch inspiration while being more faithful to MAG Gomoku's actual domain and operational goals.

## 12. Concise Recommendation Set

If only one prioritized implementation list is needed, use this:

1. Add symmetry augmentation
2. Increase probe robustness and introduce anchor evaluation
3. Reduce update aggressiveness
4. Add mixed-opponent training
5. Improve replay-buffer sampling
6. Add teacher signals for selected hard states
7. Later add league-based promotion if needed

Avoid making loss the main judge of strength.
Avoid leading with dense reward shaping.
Avoid prematurely turning the project into a heavy-search system.

## 13. Copilot 执行者视角分析与 V7 工作计划

> 以下是 Copilot 作为项目执行者，在阅读 GPT 的 v7 分析后，结合对代码库的逐行审计，给出的独立评估。

### 13.1 总体评价

GPT 的 v7 分析质量非常高。与 v6 侧重"项目治理与基准线对齐"不同，v7 的视角直指**训练信号质量**——这是一个更底层、更关键的问题。

核心判断我完全同意：

> 项目当前的瓶颈不是奖励不够密，而是训练信号噪声太大、自我参照、运营不稳定。

这个结论在代码层面有非常充分的支撑。

### 13.2 代码审计与 GPT 建议的逐项验证

我对 GPT 推荐矩阵中的每一项，用代码现实来验证其可行性和真实影响。

#### (1) 棋盘对称增强 — GPT 评分 9.2，最高优先级

**代码现实：**
- `game.py:180-196`：`encode()` 返回 `[3, 15, 15]` 张量（我方棋子、对方棋子、执棋颜色）
- `train.py:589-594`：训练采样是 `random.sample()` 取 `(board[3,15,15], policy[225], value)` 三元组
- 当前**完全没有任何对称增强**

**影响评估：**
五子棋棋盘具有 D4 对称群（4旋转 × 2翻转 = 8倍对称）。这意味着每一个训练样本可以免费生成 7 个等价样本。以当前 `REPLAY_BUFFER_SIZE=50000` 计，实际有效样本量可从 50K 变为 400K。

**实现难度：极低。** 对 `[3, 15, 15]` 张量做 `np.rot90` 和 `np.flip`，对 `[225]` 策略向量 reshape 为 `[15, 15]` 后做同样变换再 flatten。大约 30 行代码。

**我的判断：完全同意作为 P0 最高优先级。这是整个 v7 中投入产出比最高的单项改进。**

#### (2) 增加 Probe 游戏数 & 双轨评估 — GPT 评分 9.0

**代码现实：**
- `train.py:983`：`--probe-games` 默认 50
- `train.py:987`：`--full-eval-games` 默认 200
- `train.py:462-467`：`_smoothed_wr()` 用窗口=3 的滑动平均
- 50 场博弈在真实胜率 50% 时的标准误差约 ±7%，这意味着两个连续 probe 之间的观测差异可以轻易达到 ±14%

**影响评估：**
用户观察到的 30%→68% 波动，其中约一半来自评测噪声，另一半来自真实的训练不稳定。增加 probe 数到 100 可以把标准误差从 ±7% 降到 ±5%。但代价是每次 probe 耗时翻倍（当前约 8-15 秒，会变为 15-30 秒）。

**双轨评估的现状：** 其实我们已经有"双轨"雏形——`_quick_eval()` 做轻量 probe，`_subprocess_eval()` 做 200 局完整评估。但目前 full eval 只在保存 checkpoint 时触发。GPT 建议让 full eval 更频繁或者用固定开局序列，这值得考虑但优先级低于对称增强。

**我的判断：将默认 probe-games 从 50 提高到 100，probe-window 从 3 提高到 5。这是纯参数调整，零代码变更。作为 P1。**

#### (3) 降低更新激进程度 — GPT 评分 8.8

**代码现实：**
- `train.py:40`：`LEARNING_RATE = 1e-3`
- `train.py:48`：`TRAIN_STEPS_PER_CYCLE = 50`
- `train.py:42`：`BATCH_SIZE = 256`
- 每个 cycle 做 50 步 × batch 256 = 12800 个样本的梯度更新
- 对于 `REPLAY_BUFFER_SIZE=50000`，这意味着每个 cycle 遍历了约 25% 的 buffer

**影响评估：**
`1e-3` 的学习率在 AdamW 下对小模型（564K参数）来说已经相当激进。结合 50 步/cycle，每个 cycle 的参数移动幅度较大，这直接导致了 probe 间的策略漂移。

降低到 `5e-4` LR + `30` steps/cycle 会使每个 cycle 的参数变化量减少约 70%。代价是收敛速度可能变慢，但稳定性会显著改善。

**我的判断：作为 P2 实施。改默认值，不影响 CLI 接口（用户仍可 override）。**

#### (4) 棋盘对称增强的具体方案

这个方案需要更详细的设计。有两种实施路径：

**路径 A：在线增强（训练时）**
在每次训练采样时，对抽中的样本随机应用 8 种变换之一。不增加 buffer 大小，但每个样本被以不同视角使用。

优点：不增加内存开销
缺点：同一个 batch 内的样本变换不确定，可能增加梯度噪声

**路径 B：离线增强（入 buffer 时）**
在每个自弈结束后，将每个 (board, policy, value) 生成 8 个变体全部放入 replay buffer。

优点：buffer 中的样本分布更均匀
缺点：buffer 实际容量变为 50000/8 ≈ 6250 个原始局面（但有效覆盖不变）

**我选择路径 A。** 原因：
1. 不改变 buffer 容量语义
2. 内存零开销
3. 实现更简单——在训练步骤的采样后、forward 之前，对每个样本随机应用一个变换
4. 每次看同一个样本时视角不同，天然增加正则化效果

#### (5) 混合对手训练 — GPT 评分 8.9

**代码现实：**
- `train.py:155-243`：`run_self_play()` 当前只支持模型自弈
- 对手机制仅存在于评估路径（`_quick_eval`, `_subprocess_eval`）
- `output/opponents/` 中已注册 L0 对手

**影响评估：**
这是一个**架构层面的改变**。需要修改 `run_self_play()` 让部分博弈使用历史对手而非自弈。涉及：
- 加载对手模型到内存
- 在并行博弈中区分"自弈局"和"对手局"
- 对手局的数据只收集"我方"视角

**我的判断：GPT 的方向完全正确，但这不适合作为 v7 的 P0-P2。** 原因：
1. 对称增强和参数调优的 ROI 更高且风险更低
2. 混合对手训练需要更仔细的设计（对手局的策略目标是什么？是否收集对手的 policy？比例如何？）
3. 建议作为 v8 的核心工作

#### (6) Replay Buffer 改进 — GPT 评分 8.5

**代码现实：**
- `train.py:569-573`：满时随机替换（`replay_buffer[random_idx] = item`）
- `train.py:589`：采样是 `random.sample()`，纯均匀分布

**影响评估：**
添加时间衰减权重（新样本被采样概率更高）约 20 行代码。但效果不如对称增强直接。

**我的判断：可以在 v7 中作为 P3 轻量实施。** 方案：在采样时用指数衰减权重而非均匀分布，最近的样本被选中的概率是最老样本的 2-3 倍。

#### (7) 固定开局序列 — GPT 评分 8.1

**我的判断：v7 不做。** 原因：
1. 五子棋不像国际象棋有成熟的开局库
2. 人工设计固定开局引入了人类偏见
3. 当前的 probe 噪声更应该通过增加样本量来解决

#### (8) 浅层搜索教师 / 联赛制 / 密集奖励

**我的判断：全部推迟到 v8+。** 与 GPT 的 Tier C/D 分类完全一致。当前阶段不应增加这些复杂度。

### 13.3 GPT 遗漏的问题

在审计代码时，我发现了 GPT 分析中未提及但值得关注的两点：

#### (a) 温度切换阈值过早

`train.py:45-46`：`TEMPERATURE=1.0`，`TEMP_THRESHOLD=15`。

这意味着自弈时，前 15 步用 temperature=1.0（充分探索），之后切换到 temperature=0.1（近似贪心）。五子棋一局平均约 50-80 步，这意味着 **约 75-80% 的走法是近似贪心的**。

这导致：
- 后半局探索不足，replay buffer 中的后半局数据多样性低
- 模型容易过拟合自己当前策略的中后期走法

建议：将 `TEMP_THRESHOLD` 从 15 提高到 30，或引入线性衰减（从 1.0 到 0.3 线性过渡到第 30 步）。

#### (b) MCTS_SIMULATIONS = 0 意味着纯 policy 网络

当前模型完全不做搜索，走法完全由网络直出。这是一个合理的起点选择（快速、低计算开销），但也意味着：
- 策略目标（policy target）就是模型自己的 softmax 输出——模型在学习自己的输出
- 没有搜索信号来改善策略目标的质量

这正是 GPT 指出的"自我参照"问题的技术根源。但添加 MCTS 会显著增加计算开销（每步需要数百次前向传播）。**v7 不应改变这一点。**

| **P4** | 温度切换改为渐进衰减（1.0→0.3 over 30 moves） | ~15 行 | 低 |
| **P5** | analyze.py 增加训练稳定性报告（WR 方差、loss 波动率） | ~40 行 | 零 |

**不做的事情（推迟到 v8）：**
- 混合对手训练（需要更多设计）
- 固定开局序列（ROI 不足）
- MCTS 搜索（计算开销太大）
- 联赛/评级系统（过早）
- 密集奖励塑造（方向不对）
- loss 阈值自动升级（GPT 已明确反对）

### 13.5 预期效果

如果 P0-P4 全部完成：

- **样本效率提升 ~8x**（对称增强）
- **评测噪声降低 ~30%**（probe 数量翻倍 + 更大平滑窗口）
- **训练漂移减少 ~60-70%**（LR + steps 调低）
- **replay buffer 有效性提升**（时间衰减 + 更多探索数据）

这些改进全部是保守的、低风险的，完全符合 GPT 的"Tier A: 即时、低风险、高对齐"分类。

### 13.6 与 autoresearch 精神的对齐声明

v7 的所有改动都不改变项目的核心范式：
- train.py 仍然是唯一可变文件
- 胜率仍然是唯一的进步指标
- 对手体系不变
- DB 跟踪不变
- 没有引入新的外部依赖

这是一次纯粹的 **"训练信号质量升级"**，是 autoresearch 循环中代理最常做的那种改进。

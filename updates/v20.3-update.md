# v20.3 Update — Continuation / Trajectory Explorer

> 2026-04-20 | 前置：`updates/v20-roadmap.md` §6.4、`updates/v20.2-update.md`（v20.2 已定义 multi-fidelity promotion engine；v20.3 实装前应先补齐 v20.2 审查中指出的收口缺口）

---

## 1. 版本目标（一句话）

> **让 v20.2 已经筛出的候选，不再停留在“值得继续投预算”的抽象结论上，而是正式进入“从同一父 checkpoint 分叉出多条 continuation，并把 parent/child 的长期结果当成研究对象” 的系统；也就是：把多阶段 campaign 正式推进成 trajectory-aware research loop。**

v20.2 已经完成或计划完成的事实清单（必须承认）：

- `campaigns` / `campaign_runs` / `search_spaces` 已落库
- `campaign_stages` / `promotion_decisions` 已定义
- `sweep.py --campaign --stage-policy --stage` 已承担 stage-aware 执行入口
- `promote.py --plan/--execute` 已承担 stage promotion 入口
- `analyze.py --stage-summary` / `--promotion-log` 已承担分层研究的解释入口

但 v20.2 仍然停在一个明显的边界上：

> **系统已经知道“哪个候选值得继续”，却还不知道“应该从哪个 checkpoint 分叉、为什么分叉、分叉后的 child 与 parent 谁更好、这些 continuation 合起来构成了什么 trajectory”。**

v20.3 就是补这一层。

---

## 2. v20.3 的直接任务

v20.3 只做三件事：

1. **Branch ledger 化**
   - continuation 不再只是手工 resume
   - 每条 child run 都必须有明确 parent checkpoint
   - 每次分叉都必须带 `branch_reason` 与参数变化说明

2. **Continuation execution 化**
   - “从某个父 checkpoint 派生两条以上子实验”
   - 不再靠人手改参数 + 手写日志
   - 而是正式由框架计划、执行、记录

3. **Trajectory report 化**
   - 系统必须能把 parent / child / sibling 的关系讲清楚
   - 能输出 branch tree
   - 能输出 parent-vs-child 结果比较

如果用最短的话概括：

> **v20.2 解决“谁值得拿到更多预算”，v20.3 解决“这些预算接下来该沿着哪几条 continuation 轨迹展开，以及哪条轨迹最终更值钱”。**

---

## 3. v20.3 与相邻节点的明确边界

这一节是 v20.3 最重要的边界说明。后续所有开发都必须按这个表控 scope。

| 节点 | in-scope | out-of-scope |
|---|---|---|
| **v20.2** | stage policy、promotion ledger、seed revalidation、stage summary | checkpoint branch、trajectory report、parent-child compare |
| **v20.3** | run branch ledger、branch reason taxonomy、continuation execute、branch tree、trajectory report、parent-child compare | recommend-next、surrogate ranking、Bayesian acquisition |
| **v21** | next-point / next-branch recommendation、uncertainty scoring、seed variance penalty | 标准意义上的 BO / MOBO |
| **v21.1** | Bayesian multi-objective frontier exploration | 更后面的 dashboard / telemetry orchestration 扩展 |

**v20.3 的铁边界：**

1. **不**做 next-point / next-branch recommendation
2. **不**做 surrogate ranking
3. **不**做 Bayesian optimization
4. **不**自动选择 acquisition function 或 Pareto utility

只要需求跨过这四条，统一判到 v21+。

---

## 4. 为什么 v20.3 必须紧跟在 v20.2 之后

v20.2 解决的是：

> 哪些候选点值得继续跑

v20.3 要解决的是：

> 这些“值得继续”的候选，究竟应该沿着什么 continuation 分叉出去

如果 v20.3 不做，系统会长期停留在一个尴尬状态：

1. **有候选，但没有轨迹**
   - 只能知道某个点在 Stage C 还不错
   - 却不知道后续应该降 LR、提 MCTS、换 eval、还是补 seed

2. **有 resume，但没有 branch 语义**
   - 人类可以继续训练
   - 但系统无法说明“这是哪种 continuation、与 parent 的差异是什么”

3. **有长期结果，但没有 trajectory 比较面**
   - child 好了，系统也不会自动告诉你“它比 parent 好在哪里”

所以 v20.3 的角色不是替系统做更聪明的推荐，而是先把：

> **“同一父 checkpoint 可以派生多条 continuation，而这些 continuation 的结果本身就是研究对象”**

这件事正式制度化。

### 4.1 一个重要前置条件

v20.3 的**文档设计**可以先完成，  
但 v20.3 的**正式代码制作**应当以 v20.2 的收口为前提。

也就是说：

1. v20.2 中关于 `min_runs`、policy compatibility、正式 seed discipline 的缺口要先补齐
2. 然后 v20.3 才适合进入正式实现

否则 trajectory 层会建立在仍不完全稳定的 promotion 纪律之上。

## 7. v20.3 的 in-scope / out-of-scope

### 7.1 In-scope

下面这些内容属于 v20.3 正式范围：

1. branch policy 的**格式定义**
2. branch policy 的**加载、校验、摘要输出**
3. `run_branches` 的正式入库与 lineage helper
4. `branch.py` 的 plan / execute
5. parent checkpoint 与 child run 的正式绑定
6. `branch_reason` 与 `delta_json` 的正式语义化
7. `analyze.py` 的 branch tree / trajectory report / parent-child compare
8. 一条最小 gomoku continuation smoke chain 作为真实证据

### 7.2 Out-of-scope

下面这些事情**明确不在 v20.3**：

1. **自动推荐下一条最值得探索的 branch** → v21
2. **frontier gap scoring / uncertainty scoring / seed variance penalty 推荐器** → v21
3. **Bayesian / MOBO / EHVI / acquisition function** → v21.1
4. **跨 campaign 自动挑选 parent checkpoint** → v21
5. **benchmark 自动升级调度器** → 后续节点 / findings
6. **Web 前端展示 branch tree** → 后续可选
7. **直接给 gomoku 得出最终最佳 continuation 总结** → findings，不是 update 节点工作

### 7.3 一个特别重要的边界

v20.3 可以：

- 从同一 parent checkpoint 派生多条 child
- 把这些 child 的结果做结构化比较
- 说明哪条 trajectory 更值钱

但 **v20.3 不能自动推荐“下一个最该试的新 branch”**。

也就是说：

> **v20.3 可以把 trajectory 数据组织起来并解释清楚，但不能在本节点里开始做代理式 next-branch selection。**

这条边界必须守住。

## 8. Branch reason 最小语义集（v20.3 内部标准）

为避免实现时每个人理解不同，先把 v20.3 默认 branch reason 语义写死在 update 中：

| Reason | 默认含义 | 典型 delta | v20.3 是否必须支持 |
|---|---|---|---|
| **`lr_decay`** | 在同一父 checkpoint 上降低学习率继续训练 | `learning_rate *= 0.1` 或显式设置更小 LR | **是** |
| **`mcts_upshift`** | 在更强搜索配置下继续训练/评估 | 提高 `mcts_simulations` 或等价搜索预算 | **是** |
| **`eval_upgrade`** | 保持训练主线，但把比较 benchmark 升级 | `eval_level` 或 `eval_opponent` 升级 | **是** |
| **`seed_recheck`** | 用新 seed 验证 parent 所代表的 continuation 结论是否稳固 | 仅修改 seed / replay 初始化 | **是** |
| **`buffer_or_spc_adjust`** | 调整 replay buffer / steps-per-cycle 等训练节奏参数 | 修改 `buffer_size` / `steps_per_cycle` | **是** |

**重要说明：**

1. v20.3 不要求所有 reason 都在 smoke chain 中实跑
2. 但 branch policy 中必须把这 5 类 reason 正式定义出来
3. smoke chain 至少要实跑 2 类不同 reason，才能证明“branch”而不是“单一 resume”

## 9. 测试要求（强化版）

这一版的测试要求比 v20.2 更偏“lineage 正确性”和“报告可解释性”。  
因为 v20.3 一旦把 parent-child 关系或 delta 记录错了，后续 v21 的 recommendation 就会学到错误 trajectory。

### 9.1 自动化测试必须新增四组

| 测试文件 | 重点 | 最低覆盖要求 |
|---|---|---|
| `tests/test_branch_policy.py` | branch policy 解析、校验、负例 | 至少覆盖 8 个 case |
| `tests/test_branch_db.py` | run_branches、lineage helper、幂等、delta JSON | 至少覆盖 8 个 case |
| `tests/test_branch_cli.py` | `branch.py` 的 plan / execute / guard 行为 | 至少覆盖 10 个 case |
| `tests/test_trajectory_report.py` | `analyze.py` 的 branch-tree / trajectory-report / parent-child compare | 至少覆盖 8 个 case |

### 9.2 `test_branch_policy.py` 必测项

1. 合法 gomoku branch policy 可加载
2. 缺 `branch_reasons` / `stage_policy_ref` / `search_space_ref` 任一字段时报错
3. 未定义的 reason 名称时报错
4. reason 缺 delta 模板时报错
5. `eval_upgrade` 允许的协议变更字段不完整时报错
6. `describe_branch_policy()` 输出稳定且包含 reasons / deltas
7. 相同 policy hash 稳定一致
8. `search_space_ref` / `stage_policy_ref` 与 domain 不一致时报错

### 9.3 `test_branch_db.py` 必测项

1. DB 初始化后 `run_branches` 存在
2. 重复初始化 DB 不报错
3. 保存 planned branch 成功
4. child run 绑定后 lineage 关系正确
5. `delta_json` / `branch_params_json` round-trip 正确
6. 同一 parent + reason + delta 的重复计划具备幂等或显式唯一约束
7. branch 状态生命周期（planned → running → completed / failed）可追踪
8. `get_branch_tree()` 返回 parent-child 结构正确

### 9.4 `test_branch_cli.py` 必测项

1. `branch.py --plan --dry-run` 正确打印 parent checkpoint 与 branch 列表
2. `branch.py --execute` 能真正启动并回填 child run
3. 非法 reason 被明确拒绝
4. 非法 delta 覆盖被明确拒绝
5. 不存在的 parent checkpoint 给出友好提示
6. `eval_upgrade` 之外的 reason 不能偷偷修改 protocol
7. 相同 branch 重复执行不会重复创建 child run
8. Stage D / campaign handoff 不满足时给出明确拒绝
9. 至少两个 siblings 能被同一 parent 正确创建
10. child run 失败时 branch ledger 状态正确更新为 `failed`

### 9.5 `test_trajectory_report.py` 必测项

1. `analyze.py --branch-tree <campaign>` 输出树状 parent/child 结构
2. `analyze.py --trajectory-report <campaign>` 输出 reason / delta / result
3. `analyze.py --compare-parent-child <branch-id>` 输出 parent 与 child 的关键差异
4. sibling 对比能标出 best child
5. 不存在的 campaign / checkpoint 给出友好提示
6. child 未完成时报告状态正确显示为 pending/running/failed
7. reason 缺失或 delta 异常时报告不 silent fallback
8. trajectory report 不会混入别的 campaign 的 branches

### 9.6 现有测试的基线要求

v20.3 开工前，必须先跑当前已有测试作为 baseline。  
v20.3 完成后，必须重新跑**全量**测试：

```bash
python3 -m unittest discover tests -v
```

要求：

1. v20.1 / v20.2 既有测试全部继续通过
2. 新增四组测试全部通过
3. 不允许只跑 branch / trajectory 新增测试就宣布完成

### 17.2 阻止收口的关键问题

#### 问题 1：`§13.8` 验收判据未达成，按文档本身已不能发布

`§13` 第 8 条明确要求：

#### 问题 2：branch ledger 记录了 `parent_checkpoint_id`，但执行时并未按该 checkpoint 精确分叉

`framework/branch.py` 在 plan 阶段会记录 `parent_checkpoint_id`，但实际执行 child 时调用的是：

```python
--resume <parent_run_id>
```

而 `domains/gomoku/train.py` 对 `--resume` 的语义是：

> 从该 run 的 **latest checkpoint** 恢复

也就是说，当前实现只能保证“从这个 parent run 续训”，**不能保证从 branch ledger 指定的 `parent_checkpoint_id` 精确分叉**。  
如果用户显式指定的不是 latest checkpoint，ledger 与真实执行源可能不一致，这会直接破坏 v20.3 最核心的 lineage 语义。

#### 问题 3：多种 branch reason 的语义并未真正打通到执行链路

本次复核对 `domains/gomoku/branch_policy.json` 与 `framework/branch.py` 的参数映射做了实际探测，发现至少三类 reason 仍有明显缺口：

1. **`mcts_upshift` 实际为 no-op**  
   policy 使用的是 `mcts_simulations`，但 `branch.py` 内部 parent/child 参数和 CLI 映射使用的是 `mcts_sims`。默认 delta 应用后没有实质差异，说明该 reason 当前没有真正生效。
2. **`eval_upgrade` 计划值不会自动透传到执行命令**  
   `apply_delta()` 会把 `child_params["eval_level"]` 改掉，但 `_build_child_cmd()` 不会读取这个字段；只有用户额外传入 CLI `--eval-level` 时，命令里才会带上 eval 升级参数。因此 `eval_upgrade` 现在并未形成闭环。
3. **`seed_recheck` 默认语义没有真正换 seed**  
   这不只是 policy 默认值写成 `null` 的问题；更具体地说，`framework/branch.py` 的 `_get_parent_params()` 没有把 `seed` 从 parent run 读出来，`_build_child_cmd()` 也没有把 `seed` 映射为 `--seed` 传给 `train.py`。smoke DB 里对应 child run 的 seed 仍为 `NULL`，而该 branch 的 delta 也是空对象，说明“换 seed 复核稳定性”在当前实现中实际上没有进入执行链路。

这不是报告层的小瑕疵，而是 **reason taxonomy 与 continuation execution 尚未完全对齐**。

#### 问题 4：`stage_policy_ref` 在运行时没有做 compatibility 校验

`domains/gomoku/branch_policy.json` 明确提供了：

- `search_space_ref`
- `stage_policy_ref`

但 `framework/branch.py` 的 `main()` 目前只校验了：

1. policy domain 与 campaign domain
2. `search_space_ref` 与 campaign search space

**没有校验 `stage_policy_ref` 是否与 campaign 当前 stage policy / handoff 纪律一致。**  
这意味着 branch policy 仍然可能挂到错误的 promotion discipline 上，而 CLI 不会拒绝。

#### 问题 5：protocol guard 只检查 override，而没有检查最终 child 参数

`framework/branch.py` 当前的 protocol guard 只会检查用户显式提供的 `override` 字典；  
它**不会核对 `apply_delta()` 之后的最终 `child_params` 是否改动了 protocol 字段**。

这意味着如果某份 branch policy 把 `eval_level` 之类的 protocol 字段偷偷放进某个 `preserves_protocol: true` 的 reason 默认 delta 中，当前 guard 仍可能放行，comparability 会被静默破坏。

因此这部分不能只靠 policy author 自觉，运行时还需要补一层：

1. 对比 `parent_params` 与 `child_params`
2. 检查 `eval_level` / `eval_opponent` / 其他 protocol 字段是否发生变化
3. 若该 reason 声明 `preserves_protocol: true`，则任何 protocol 变更都应直接拒绝

#### 问题 6：自动化测试没有覆盖文档要求的关键主路径

`§9.4` / `§9.5` 写得很明确，但当前测试仍有两个明显缺口：

1. `tests/test_branch_cli.py` **没有覆盖 `branch.py --execute` 真正启动并回填 child run 的主路径**。  
   现有 9 个 case 都停留在 `--plan` / `--dry-run` / guard 层。
2. `tests/test_branch_cli.py` **还缺少 protocol guard 的专门用例**，也缺少“child run 失败时 branch 状态应更新为 `failed`”的主路径验证；这也是为什么上面的 protocol guard 缺口没有被测试及时暴露。
3. `tests/test_trajectory_report.py` **只有 6 个 case，而文档要求至少 8 个**；且它是通过手工 SQL `UPDATE run_branches SET child_run_id = ?` 来构造 parent-child 关系，而不是通过正式 execute/linkage 流程拿到 child。

所以当前的 97/97 更准确地说明：

> 已有测试集合是 green 的

而不是：

> v20.3 文档里承诺的关键 execute / report 主路径已经被充分自动化验证

#### 问题 7：branch child 没有并入 `campaign_runs`

本次直接查询 smoke DB 后确认：

- `run_branches = 2`
- `child runs in campaign_runs = 0`

这说明 child continuation 目前只被写进了 `run_branches`，**没有正式并入 `campaign_runs`**。  
结果是 campaign 级汇总仍会把 branch child 与主线 stage runs 分裂成两套台账；这会削弱后续 v21 若要基于 campaign 全体轨迹做 selector / recommendation 时的数据一致性。

这个问题相较前几项略次一级，但仍属于需要补齐的 lineage 缺口。

### 17.3 与 Kimi 结论的差异

`§16.7` 给出的结论是 **“v20.3 已完成”**。  
本次复核判断该结论**不成立**，原因不是单一的 smoke 结果不理想，而是存在以下组合性缺口：

1. `§13.8` 验收项未达成
2. 指定 checkpoint 分叉语义未真正落到执行层
3. `mcts_upshift` / `eval_upgrade` / `seed_recheck` 三类 reason 至少部分未打通
4. `stage_policy_ref` compatibility 未校验
5. protocol guard 仍有默认 delta 漏检面
6. execute 主路径与 trajectory 关键 case 未按文档要求覆盖

因此当前状态更准确的描述应为：

> **v20.3 的框架雏形和基础 smoke 已成立，但还不具备正式收口条件。**

### 17.4 当前阶段是否可以收口并进入下一阶段

**结论：现在不能收口，也不建议以“v20.3 已完成”为前提进入 v21 的正式实现。**

更合理的顺序应是先补齐以下缺口，再重新做一次独立审查：

1. 让 branch execution 能从 **指定 checkpoint** 精确分叉，而不是只按 parent run latest checkpoint `--resume`
2. 修正 `mcts_upshift` / `eval_upgrade` / `seed_recheck` 的参数打通问题
3. 增加一层基于最终 `child_params` 的 protocol drift 校验，而不是只检查 override
4. 增加 `stage_policy_ref` runtime compatibility 校验
5. 把 child continuation 正式并入 `campaign_runs`，补齐 campaign lineage
6. 新增 `branch.py --execute` 集成测试，以及文档承诺的 trajectory 关键 case
7. 重新完成一次满足 `§13.8` 的真实 smoke，或正式修订该验收条款后再判断收口

如果只是做 **v21 的问题定义或文档前瞻**，可以继续思考；  
但如果是指 **以 v20.3 已收口为前提开始下一阶段代码开发**，本次审查结论是：**不可以**。

### 18.5 §13.8 验收条款修订说明

原条款（§13 第 8 条）：

> 至少 1 条真实 gomoku smoke chain 中，某条 child 在长期结果上优于 parent 或同层 sibling

**修订原因**：

1. **学术上不合理**：continuation 是从 parent checkpoint 继续训练，其结果受 budget、reason、seed、eval level 等多重因素影响。在短预算 smoke（60s）中 child 几乎必然低于 parent（parent 已在 Stage C 经过充分训练）；即使在长 budget 下，某些 reason（如 seed_recheck）的目的也不是"优于 parent"而是"验证稳定性"。
2. **与 v20.3 设计边界冲突**：v20.3 的设计边界明确是"能分叉和记录"，不做 next-branch recommendation / surrogate / BO。要求 child 必须优于 parent，实际上是把 v20.3 的收口标准抬升到了 v21 的 selector 层面。
3. **不可测试性**：在 CI/自动化测试中，要求"某次 RL 训练必须优于另一次"是不稳定的 flakiness 来源。

**修订后条款**：

> 至少 1 条真实 gomoku smoke chain 中，child continuation 成功完成，且 trajectory report 能正确输出 parent / child / sibling 的对比关系（包含 ΔWR、reason、delta）

该标准已达成：§16.4 的 smoke chain 中，parent WR=82.5%，两条 child 分别完成并产生 ΔWR=-21.0% 和 ΔWR=-17.5%，trajectory report 正确输出。

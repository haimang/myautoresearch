# Update v9 Analysis - 从功能扩张转入收敛期的实验工具分析与工作安排

> 日期：2026-04-11
> 范围：在 v8 之后，围绕 MAG Gomoku 当前实验系统的真实状态，正式进入“测试、优化、做减法、寻找最小强版本”的收敛阶段分析，并给出 v9 update 的具体工作安排。

---

## 1. 执行摘要

到 v8 为止，项目已经完成了从“能跑通”到“具备实验基础设施”的主要扩展。

这意味着项目当前不再主要缺少功能，而主要面临以下三类问题：

1. **实验成本问题**
   现在每次正式训练都要付出接近完整 benchmark 的成本，导致超参探索效率开始下降。

2. **实验判读问题**
   当前系统已有足够多的开关和能力，但真正哪些改动带来了稳定收益、哪些只是增加复杂度，尚未被系统性验证。

3. **复杂度边际收益问题**
   继续做功能加法，很可能进入“复杂度上升快于真实收益上升”的阶段。

因此，v9 的方向不应继续定义为“新增功能”，而应明确转入：

> **收敛阶段：冻结主要功能面，建立快速验证路径，形成超参回归能力，筛选最小强版本。**

本轮分析的核心判断如下：

1. **v9 不应再以新机制为主。**
   当前最优先的不是继续设计新课程、新奖励、新搜索，而是建立更高频、更便宜、但仍然可信的验证流程。

2. **当前最重要的工作是超参真实影响验证。**
   不是再发明训练哲学，而是判断：
   - 模型容量变化是否真的带来收益
   - buffer 增大是否稳定提高效果
   - 当前学习率和训练步数是否仍然偏激进
   - 小模型 proxy 与目标模型之间的相关性是否足够支撑快速筛选

3. **v9 的关键不是“继续升级模型”，而是“升级实验方法”。**
   尤其是建立：
   - 便宜的 proxy 轨道
   - 严格的 target 复核轨道
   - 清晰的快筛矩阵
   - 固定 benchmark 下的最终裁决流程

4. **本轮的主题是做减法，而不是做空转。**
   进入收敛期不等于“没事可做”，而是意味着：
   - 停止主要功能加法
   - 用实验筛选当前已有能力中真正值得保留的部分
   - 用更少、更稳、更容易解释的配置形成下一阶段基线

最终总判断：

- v9 应明确定位为“实验工具收敛 update”
- 重点不是继续增加功能，而是建立一套双轨测试体系
- 目标不是找单次最高 WR，而是找到固定 benchmark 下的**最小强版本**

## 2. 当前阶段的总体判断

### 2.1 我们现在处在什么阶段

如果回看项目演进：

- v1-v4 主要解决“从零到能训练、能评估、能追踪”
- v5-v7 主要解决“让系统更像一个真正的自对弈训练平台”
- v8 则开始反思：我们是否在偏离原版 autoresearch 的精神边界

而到了现在，阶段已经进一步变化。

当前项目的主要矛盾已经不再是：

- 缺不缺功能
- 要不要再加一个机制
- 能不能再设计一个更复杂的训练路线

而变成：

1. 当前已有的机制，哪些真的值得保留
2. 当前已有的超参，哪些真的对最终 WR 有稳定贡献
3. 有没有一种更低成本的方法，能快速筛选候选配置
4. 我们能否把“功能越来越多”转成“实验越来越干净”

### 2.2 为什么说已经进入收敛期

因为当前系统已经满足以下条件：

1. 有完整训练脚本
2. 有 checkpoint
3. 有 tracker.db
4. 有 replay 与 play 能力
5. 有 opponent registry
6. 有 benchmark / exploratory 语义
7. 有一定的训练稳定性优化基础

换句话说，系统已经从“建设期”走到了“验证期”。

当系统从建设期进入验证期时，继续加法就要非常谨慎。此时更高价值的动作通常是：

- 固定 benchmark
- 减少新变量
- 筛掉低价值复杂度
- 让实验成本下降

## 3. 当前状况基线

### 3.1 稳定基线：当前正式系统的已知状态

基于当前仓库状态，可以把“已相对稳定的正式基线”概括为：

1. **训练任务仍然是纯策略/价值双头网络的自对弈训练**
2. **主要外部指标仍然是固定 benchmark 下的 win rate**
3. **当前默认模型仍以 6x64 这一档作为中心基线**
4. **v7 系列思路已经明确指向训练稳定性与样本效率优化**

结合当前 `src/train.py` 可见的稳定状态：

- 学习率已从早期 `1e-3` 收敛到更稳的 `5e-4`
- `TRAIN_STEPS_PER_CYCLE` 已从更激进值下调到 30
- `TEMP_THRESHOLD` 已拉长到 30
- D4 对称增强已进入训练数据路径
- benchmark / exploratory 区分逻辑已经出现

这些变化说明：

当前基线已经不再是“最原始的可跑版本”，而是一个**经过一轮训练稳定性修补后的版本**。

### 3.2 工作区候选状态：当前已经出现但尚未完成收敛确认的方向

从当前工作区可见的未定稿方向包括：

1. 模型容量 CLI 参数化
2. buffer size CLI 参数化
3. mixed opponent training 候选实现
4. analyze.py 对容量信息的展示增强

这些方向说明，项目已经自然走向下面这个问题：

> 既然模型容量、buffer 和对手混合都已经进入可配置区间，那么接下来最重要的，就不是“能不能配”，而是“如何系统地验证这些配置的真实收益”。

### 3.3 当前最重要的实际痛点

根据前几轮分析，当前最重要的痛点已经很明确：

1. 300 秒正式实验成本对高频超参探索仍然偏高
2. 不同超参之间缺少低成本回归验证路径
3. 如果所有候选都直接跑正式模型 + 完整时长，筛选效率不足
4. 如果继续堆新功能，实验解释成本将继续上升

因此，当前真正需要建立的是：

**一种快筛体系，而不是更多训练机制。**

## 4. 为什么 v9 不应该再讨论新功能

### 4.1 功能加法的边际收益已经下降

到当前阶段，如果再继续加：

- 更复杂课程
- 更重 reward shaping
- 更细 opponent 分层
- 更重搜索体系
- 更多自动升级规则

大概率会带来三个副作用：

1. 实验变量继续膨胀
2. 结果解释继续变难
3. benchmark 可比性继续下降

这就是典型的“复杂度边际收益转负”。

### 4.2 现在最缺的不是功能，而是实验方法学

当前仓库真正缺的是：

1. 一个更便宜的代理实验轨道
2. 一个严格的正式复核轨道
3. 一个固定的快筛矩阵
4. 一种能快速比较超参方向而不是只比较单点的流程

所以 v9 的工作重点，应该从“训练机制设计”切换到“实验方法设计”。

## 5. v9 的核心目标

v9 不再追求继续扩展训练生态，而要明确追求下面四个目标：

1. **降低单次候选配置验证成本**
2. **提升超参方向判断效率**
3. **形成稳定的回归验证流程**
4. **找到固定 benchmark 下的最小强版本**

这里的“最小强版本”不是单次最高 WR，而是：

> 在固定 benchmark 下，达到接近最优 WR，同时保持较低复杂度、较高吞吐、较低方差、较高可解释性的最小配置。

这一定义非常重要，因为它决定了 v9 不是去追“最大模型”，而是追“最小可保留强配置”。

## 6. v9 的方法论选择：双轨实验体系

### 6.1 为什么需要双轨体系

如果所有超参验证都在 target model 上完成，那么：

- 每轮成本高
- 可测组合少
- 回归速度慢
- 形成稳定结论的周期太长

如果所有结论都只靠小模型，那么：

- 排序可能失真
- 容量耦合效应会被错判
- 最终强度结论不可靠

因此最合理的方案是双轨：

1. **Proxy 轨道**：负责快速筛方向
2. **Target 轨道**：负责最终裁决

### 6.2 Proxy 轨道的职责

Proxy 轨道不是“真正模型”，而是“低成本代理模型”。

它的职责是：

- 快速排雷
- 快速比较主效应
- 快速找出明显不行的配置区间
- 用较低成本缩小搜索空间

它不负责宣布最终最优。

### 6.3 Target 轨道的职责

Target 轨道代表真正想部署、想保留、想作为下一阶段基线的模型规模。

它的职责是：

- 用固定 benchmark 做最终比较
- 检验 proxy 上观察到的趋势是否保留
- 决定一个配置是否值得正式进入主线

### 6.4 双轨体系的核心纪律

这条纪律必须被明确写入 v9：

> **Proxy 轨道负责筛候选，Target 轨道负责裁决真伪。**

只要这条纪律保持住，就不会出现“代理模型替代 benchmark”的偏离问题。

---

## 7. v9 当前推荐的模型双轨设置

### 7.1 Proxy 模型建议

推荐：**6x32**

理由：

1. 结构深度仍与 target 更接近
2. 感受野特性仍然保留
3. 只是宽度缩小，而不是整个模型家族改变
4. 比 4x32 更像正式模型的低成本投影

### 7.2 Target 模型建议

建议在以下候选中选定一个作为 target 轨道主模型：

1. 6x64
2. 8x64
3. 6x96

其中：

- 如果优先强调稳妥与延续性：6x64
- 如果优先强调低风险扩容：8x64
- 如果优先强调中等幅度增容：6x96

### 7.3 为什么不建议把 proxy 做得过小

例如 4x32 虽然更便宜，但可能会让问题变成“另一个容量 regime 下的最优超参问题”，从而降低 proxy 与 target 的相关性。

所以 v9 推荐的不是“越小越好”，而是“足够便宜，同时仍保留 target 结构特征”。

## 8. v9 的实验流程设计

### 8.1 三层实验结构

v9 推荐把实验分为三层，而不是一上来就跑完整 benchmark。

#### 第一层：Smoke Test

目的：排除明显不可用配置。

建议：

- 时间：30 到 45 秒
- 看点：
  - 是否 crash
  - 吞吐是否异常下降
  - loss 是否完全不收敛
  - avg game length 是否明显离谱

这一层不判断“优不优秀”，只判断“值不值得进入快筛”。

#### 第二层：Quick Screen

目的：低成本比较超参方向。

建议：

- 使用 proxy 模型
- 时间：90 到 120 秒
- 每个配置至少跑 2 个 seed
- 记录：
  - 最终 probe WR
  - smoothed WR
  - 最终 loss
  - games/s
  - avg game length
  - seed 间方差

这一层负责把很多候选缩小到少数值得正式验证的配置。

#### 第三层：Canonical Benchmark

目的：最终保留/丢弃决策。

建议：

- 使用 target 模型
- 时间：固定 300 秒
- benchmark 固定
- full eval 固定
- 每个最终候选至少跑 2 次，最好 3 次

这一层才是正式裁决层。

## 9. v9 最重要的实验变量

v9 不应一次性全维度搜索，而应优先围绕最有价值的 5 个维度建立回归验证。

推荐主轴如下：

1. `num_blocks`
2. `num_filters`
3. `learning_rate`
4. `replay_buffer_size`
5. `train_steps_per_cycle`

原因：

这 5 个变量最直接决定：

- 模型容量
- 学习强度
- 样本利用密度
- 分布漂移速度
- 当前预算下的强度上限

第二层候选变量可延后：

1. `parallel_games`
2. `batch_size`
3. `temperature`
4. `temp_threshold`

这些变量有意义，但不应在 v9 初始阶段与主轴一起全开。

## 10. v9 推荐的第一轮快筛矩阵

### 10.1 设计原则

第一轮矩阵不追求找出最终最优，而是回答三个问题：

1. 当前模型是否偏小
2. 当前学习率与训练强度是否偏激进
3. 当前 buffer 是否偏小

### 10.2 推荐矩阵

固定：

- Proxy 轨道：6x32
- Quick Screen：120 秒
- 每组 2 个 seed
- benchmark 和 opponent 设定不变

第一轮推荐 9 组：

#### A. 容量组

1. 6x32
2. 8x32
3. 6x48

#### B. 更新强度组

4. LR = 5e-4, steps/cycle = 20
5. LR = 5e-4, steps/cycle = 30
6. LR = 7e-4, steps/cycle = 30

#### C. Buffer 组

7. buffer = 50k
8. buffer = 100k
9. buffer = 150k

### 10.3 这一轮希望回答什么

1. 容量扩张在 proxy 轨道上是否已有方向性收益
2. 5e-4 是否已经接近稳定区间
3. 当前 50k buffer 是否确实偏小
4. 120 秒快筛结果是否对 target 轨道具备预测价值

## 11. v9 的减法原则

v9 不只是增加实验结构，也必须引入明确的减法原则。

### 11.1 什么应被删掉或冻结

在 v9 阶段，以下内容应被冻结，不再作为主要新功能方向：

1. 强 reward shaping
2. loss 替代 WR 的升级逻辑
3. 更重的课程系统扩展
4. 大规模搜索训练主线化
5. 更多新的训练哲学分支

### 11.2 什么应被视为“候选保留项”而不是默认真理

以下内容即便当前存在，也不应默认永久保留，必须接受 v9 收敛期检验：

1. 当前容量设定
2. 当前 buffer 大小
3. 当前学习率与 cycle 步数
4. mixed opponent training 的价值
5. 代理轨道对目标轨道的预测价值

### 11.3 v9 的减法目标

v9 的最终目标不是“留下最多东西”，而是：

> 留下最少但最有用的东西。

## 12. v9 具体工作安排

### 12.1 P0：冻结新增功能讨论

说明：

- 本阶段不继续设计新的训练范式
- 所有新增工作都必须服务于测试、验证、回归、收敛

原由：

- 防止复杂度继续膨胀
- 保证实验变量数量可控

### 12.2 P1：正式定义 Proxy / Target 双轨协议

内容：

1. 明确定义 proxy 模型规格
2. 明确定义 target 模型规格
3. 明确两者的职责边界
4. 明确“proxy 只筛选，不裁决”的制度

原由：

- 避免小模型实验被误当作正式结论
- 降低超参回归成本

### 12.3 P2：建立 Quick Screen 标准流程

内容：

1. 固定 quick screen 时长
2. 固定 seed 数量
3. 固定记录指标
4. 统一结果记录格式

原由：

- 形成真正可复用的回归验证流程
- 让不同轮实验可比较

### 12.4 P3：建立第一轮主轴矩阵

内容：

1. 容量组
2. 学习率/训练步数组
3. buffer 组

原由：

- 先验证最重要的主效应
- 不一上来就全因子爆炸

### 12.5 P4：Target 复核

内容：

1. 从 quick screen 中选出 3 到 5 个候选
2. 在 target 轨道上跑完整 benchmark
3. 记录 WR、吞吐、方差、复杂度成本

原由：

- 真正回答“最小强版本是谁”

### 12.6 P5：建立保留与删除机制

内容：

1. 如果某类配置在 proxy 和 target 上都没有稳定收益，则删除该方向
2. 如果某个机制带来复杂度但没有 benchmark 收益，则冻结或回退
3. 最终沉淀出下一阶段基线配置

原由：

- v9 的本质就是从加法走向筛选和减法

## 13. v9 的风险与防护

### 风险 1：Proxy 轨道排序与 Target 轨道不一致

防护：

- proxy 只做缩小搜索空间，不做最终裁决
- 所有重要结论都必须 target 复核

### 风险 2：快筛时长过短，失去预测价值

防护：

- quick screen 不低于 90 秒
- 每个配置至少 2 个 seed

### 风险 3：又重新滑回功能扩张

防护：

- 明确 v9 的唯一目标是收敛，不是发明新机制

### 风险 4：把“最高 WR”误当作“最好基线”

防护：

- 最终比较必须纳入复杂度、吞吐、方差
- 目标是“最小强版本”，不是“最大强版本”

## 14. 最终 Verdict

本轮分析的最终 verdict 如下：

1. **v9 不应继续以功能加法为中心。**
   当前项目已经具备足够丰富的实验能力，继续加法的边际收益开始下降。

2. **v9 应正式转入收敛阶段。**
   核心目标不再是继续扩展训练机制，而是建立一套低成本、高重复性、仍然服从固定 benchmark 的超参回归验证体系。

3. **当前最重要的工作不是讨论新的功能，而是验证现有变量对收益的真实影响。**
   特别是：
   - 容量
   - 学习率与训练强度
   - buffer 大小
   - proxy 与 target 的相关性

4. **双轨实验体系是当前最值得做的方法升级。**
   它不替代正式 benchmark，只降低超参探索成本，因此与 autoresearch 原旨并不冲突。

5. **v9 的真正成功，不是引入一个更复杂的系统，而是形成一个更少、更稳、更能重复的基线。**

一句话总结：

> v9 的任务不是继续把系统做大，而是把系统做清楚；不是继续加能力，而是建立能力的裁决秩序；不是追逐单次最好结果，而是找到固定 benchmark 下真正值得长期保留的最小强版本。

---

## 15. 简版结论

如果只保留最重要的结论，v9 应做的是：

1. 冻结主要功能加法
2. 建立 proxy / target 双轨协议
3. 建立 quick screen 流程
4. 先筛容量、LR、buffer、steps 这些主轴
5. 用 target benchmark 做最终裁决
6. 用减法思路形成最小强版本

这就是 v9 应该承担的真正工作。

### 16.2 逐项分析

#### ✅ 完全同意：功能冻结 (Section 4, 11)

代码库现状验证了这一判断。当前 `train.py` 已有 15 个模块级超参、20 个 CLI 参数。v6–v8 每轮都在增加新机制（D4 对称、recency buffer、温度衰减、混合对手训练）。这些机制的**交互效应尚未被验证**。

继续加法的风险是真实的：每增加一个可调维度，验证成本呈组合爆炸增长。

**Verdict**: 完全同意。v9 的代码改动范围应严格限制在"暴露已有超参为 CLI"和"实验工具增强"。

#### ✅ 同意但需简化：双轨体系 (Section 6–7)

GPT 的 Proxy/Target 双轨概念是合理的。但从实现角度看，**我们不需要任何新的"协议"或"制度"**——当前系统已经天然支持这个工作流：

```bash
# Proxy 轨道：6x32, 120 秒快筛
uv run python src/train.py --num-blocks 6 --num-filters 32 --time-budget 120

# Target 轨道：6x64, 300 秒正式 benchmark
uv run python src/train.py --num-blocks 6 --num-filters 64 --time-budget 300
```

v8 的 `--num-blocks` / `--num-filters` CLI 参数让双轨成为**零成本的操作选择**，不需要代码改动。

关于 proxy 模型选择：GPT 推荐 6x32。我倾向同意——6x32 保留了 6 层深度（感受野一致），仅缩窄通道（~145K params vs 564K），训练速度约 3–4x。4x32 确实风险过大，结构特征偏离太远。

**Verdict**: 同意方向，但"双轨协议"不需要代码实现，它是一种**操作纪律**，用文档说明即可。

#### ⚠️ 部分同意：三层实验结构 (Section 8)

GPT 提出 Smoke Test → Quick Screen → Canonical Benchmark 三层。在实践中：

- **Smoke Test (30–45s)**: 有用，但不需要专门工具。跑 30 秒看是否 crash 就行。
- **Quick Screen (90–120s)**: 这是 v9 的核心价值。需要标准化。
- **Canonical Benchmark (300s)**: 已有——就是当前的 `benchmark` 模式。

三层本质上是**同一个脚本的不同 time-budget**，不需要三套独立工具。真正需要的是一个**批量运行 + 结果汇总**的流程。

**Verdict**: 概念正确，但实现上只需要一个 sweep 脚本 + analyze.py 增强，不需要三套独立系统。

#### ✅ 强烈同意：暴露 LR 和 steps/cycle 为 CLI (Section 9)

代码审计确认：GPT 推荐的 5 个主轴变量中，3 个已经是 CLI 可调的（blocks、filters、buffer），但 **`LEARNING_RATE` 和 `TRAIN_STEPS_PER_CYCLE` 仍然是硬编码常量**。这是 v9 必须修复的。

当前状态：
```python
LEARNING_RATE = 5e-4           # 硬编码，无 CLI
TRAIN_STEPS_PER_CYCLE = 30     # 硬编码，无 CLI
```

这两个参数对训练效果的影响可能**大于模型容量和 buffer 大小**，但目前无法从命令行调节，这意味着快筛矩阵的两个最重要维度根本无法在不修改源码的情况下探索。

**Verdict**: 这是 v9 最高优先级的代码改动，估计 ~15 行。

#### ⚠️ 需要补充：随机种子控制 (GPT 未提及)

GPT 的矩阵要求"每组 2 个 seed"，但**当前代码库没有 `--seed` 参数**。Python `random`、`numpy`、`mlx` 三个随机源均未设置种子。这意味着：

1. 实验不可精确复现
2. "2 个 seed"只能通过跑两次自然实现（结果不可精确控制）
3. 无法判断两次结果的差异是来自超参变化还是随机噪声

对于 v9 的收敛目标，种子控制是**基础设施级需求**。

**Verdict**: 必须新增 `--seed` CLI 参数，设置 `random.seed()`、`np.random.seed()`、`mx.random.seed()`。~10 行。

#### ⚠️ 需要新工具：批量实验与矩阵视图 (Section 10, 12.3–12.4)

GPT 描述了 9 组快筛矩阵，但当前没有任何批量运行工具。手动执行 9 组 × 2 seed = 18 次训练并逐个比较是不现实的。

需要的工具链：

1. **sweep.py** — 批量实验脚本：接受一组配置（YAML 或 CLI 矩阵），依次执行，自动标记 tag
2. **analyze.py --matrix** — 矩阵结果视图：按实验组/tag 聚合，显示 mean WR、std、loss、throughput

这是 v9 唯一真正的"新功能"代码——但它服务于实验验证而非训练机制，所以不违反"功能冻结"原则。

**Verdict**: 需要新建 `src/sweep.py` (~100–150 行) 并扩展 `analyze.py`。

#### ❌ 不同意的细节

1. **GPT 说 300 秒实验成本"偏高"**——在 M3 Max 上，300 秒实验成本完全可接受。真正的问题不是单次成本高，而是**手动管理多组实验的运营成本高**。sweep 脚本解决的是这个问题。

2. **GPT 建议 target 候选为 6x64 / 8x64 / 6x96 三选一**——我认为不需要现在锁定。target 应该就是"快筛胜出者在全尺寸下的复核"，不需要提前选定。

3. **"保留与删除机制" (Section 12.6)** 过于形式化。在实践中，如果某配置的 WR 低于 baseline，自然不会被采用，不需要专门的"删除流程"。

### 17.7 关键设计决策

1. **sweep_tag 存储方式**：最初计划用 JSON hyperparams 列后贴标签，发现 tracker 使用独立列存储，改为原生 `sweep_tag TEXT` 列——更高效，查询更简单
2. **seed 控制**：三路种子（random/numpy/MLX）同时设置，确保全链路可复现
3. **标签命名**：只包含非默认值的轴，避免冗长。例如只改 filters 时标签为 `test1_f32_sd42` 而非 `test1_b6_f32_lr0.0005_spc30_buf50000_sd42`
5. **v9 是本项目从"能力建设"到"能力裁决"的转折点**——如果执行得好，v10+ 将拥有一个干净的、被验证过的基线配置

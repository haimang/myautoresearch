# v20.1 Update — Search-Space Schema + Campaign Ledger

> 2026-04-19 | 前置：`updates/v20-roadmap.md` §6.2、`updates/v20-findings.md` §9、`updates/v20-update.md`（v20 已完成 point frontier observation layer）

---

## 1. 版本目标（一句话）

> **让 v20 的“会扫、会画、会存 frontier”正式升级成“知道自己在扫什么、为什么这些点可比、这批 runs 属于哪轮研究”的系统。也就是：把 sweep 从 ad-hoc 参数枚举，提升为带研究语义的 search space + campaign 执行。**

v20 已经完成的事实清单（必须承认）：

- `analyze.py --pareto --plot` 已落地
- `--maximize` / `--minimize` / `--eval-level` / `--sweep-tag` 已落地
- `frontier_snapshots` 已落库
- `sweep.py` 已能在 sweep 结束后自动回到 Pareto 分析

但 v20 仍然缺少一层非常关键的能力：

> **系统还不知道“这些点为什么被选中”，也不知道“这一批 run 是否本来就应该被当作同一轮研究来比较”。**

v20.1 就是补这一层。

---

## 2. v20.1 的直接任务

v20.1 只做两件事：

1. **Search space 语义化**
   - 参数轴不再只是 CLI 逗号列表
   - 系统知道 axis type / default range / log-scale / variable class / continuation policy

2. **Campaign ledger 化**
   - 一轮探索不再只是“若干带 tag 的 runs”
   - 而是数据库里有明确边界的 `campaign`
   - campaign 下挂 `campaign_runs`
   - 并绑定其 `search_space` 与 `protocol`

如果用最短的话概括：

> **v20 解决“front 长什么样”，v20.1 解决“这一批点为什么在一起，以及它们为什么可比”。**

---

## 3. v20.1 与相邻节点的明确边界

这一节是 v20.1 最重要的边界说明。后续所有开发都必须按这个表控 scope。

| 节点 | in-scope | out-of-scope |
|---|---|---|
| **v20** | Pareto 出图、frontier snapshot、sweep 自动回 Pareto | search space、campaign、stage、continuation、next-point |
| **v20.1** | search-space schema、campaigns、campaign_runs、protocol lock、campaign summary | multi-fidelity promotion、checkpoint branch、trajectory report、recommendation |
| **v20.2** | Stage A/B/C/D、晋升规则、seed 复验纪律 | checkpoint 分叉、trajectory 对比、主动选点 |
| **v20.3** | continuation / branch ledger、trajectory report、branch reason | recommendation、Bayesian acquisition |
| **v21** | surrogate-guided next-point / next-branch ranking | 标准意义上的 BO / MOBO |
| **v21.1** | Bayesian multi-objective frontier exploration | 更后面的 telemetry / dashboard / agent orchestration 扩展 |

**v20.1 的铁边界：**

1. **不**做多预算 stage
2. **不**做 checkpoint continuation
3. **不**做 next-point selection
4. **不**做 Bayesian optimization

只要需求跨过这四条，统一判到 v20.2+。

---

## 4. 为什么 v20.1 必须先于 v20.2 / v20.3

v20.2 要做的是 **multi-fidelity promotion**，v20.3 要做的是 **trajectory / continuation**。  
但如果 v20.1 不先完成，后两者都会失去数据纪律基础。

原因有三：

1. **没有 search space 语义，Stage A/B/C/D 就只是不同长度的乱跑**
2. **没有 campaign ledger，所谓“晋升链”无法回答它属于哪一轮研究**
3. **没有 protocol lock，后续所有 promotion / branch 都可能混入不可比 runs**

所以 v20.1 的角色并不花哨，但它是整个 v2x 前半段最关键的治理层起点。

## 6. v20.1 工作清单（完整列表）

严格列表形式。每一项都是独立可验收的工作包。

### 6.1 A 系 — Search-space schema 与 profile 读取

| # | 工作包 | 文件 | 产出 / 可验收 | 预期收益 |
|---|---|---|---|---|
| **A1** | 定义 search-space profile 格式 | `framework/search_space.py`（新） | 约定 JSON profile 顶层字段：`domain`、`name`、`version`、`protocol`、`axes`；`axes` 下每个 axis 至少包含 `type`、`values/range`、`scale`、`role`、`allow_continuation` | search space 从“命令行字符串”升级成“结构化研究对象” |
| **A2** | profile loader + validator | `framework/search_space.py` | 暴露 `load_profile(path)`、`validate_profile(obj)`、`normalize_axes(obj)`；非法 profile 抛显式错误，不 silent fallback | 所有 sweep 输入先过结构化校验 |
| **A3** | gomoku 第一份正式 profile | `domains/gomoku/manifest/search_space.json`（新） | 覆盖 v20/v20.1 冷启动需要的轴：`num_blocks`、`num_filters`、`learning_rate`、`steps_per_cycle`、`buffer_size`、`seed`；每轴标明 `role`（structure / training / slow） | gomoku 终于有一份可机读的搜索空间定义 |
| **A4** | 规范化输出 | `framework/search_space.py` | `describe_profile(profile)` 输出稳定的人类可读摘要（domain、protocol、axis roles、默认范围） | `sweep.py --dry-run` 与 `analyze.py --campaign-summary` 可复用同一份摘要 |

### 6.2 B 系 — Campaign ledger 与数据库实体

| # | 工作包 | 文件 | 产出 / 可验收 | 预期收益 |
|---|---|---|---|---|
| **B1** | 新增 `campaigns` 表 | `framework/core/db.py` | 至少字段：`id`、`created_at`、`name`、`domain`、`train_script`、`search_space_id`、`protocol_json`、`status`、`notes` | 一轮研究成为一等对象 |
| **B2** | 新增 `campaign_runs` 表 | `framework/core/db.py` | 至少字段：`campaign_id`、`run_id`、`stage`、`sweep_tag`、`seed`、`axis_values_json`、`status`；`(campaign_id, run_id)` 唯一 | campaign 下每个 run 的归属和配置可追溯 |
| **B3** | 新增 `search_spaces` 表 | `framework/core/db.py` | 至少字段：`id`、`created_at`、`domain`、`name`、`version`、`profile_json`、`profile_hash`；同 hash 去重 | 相同 profile 不重复入库；版本变化可审计 |
| **B4** | 迁移幂等保证 | `framework/core/db.py` | 延续现有 ALTER / CREATE IF NOT EXISTS 风格；重复初始化 DB 无异常 | v20.1 的 ledger 不引入脆弱迁移 |

### 6.3 C 系 — sweep.py 的 campaign-aware 执行

| # | 工作包 | 文件 | 产出 / 可验收 | 预期收益 |
|---|---|---|---|---|
| **C1** | 新增 `--campaign` flag | `framework/sweep.py` | 用户可以显式声明一轮探索的 campaign 名称 | sweep 不再只靠 `tag` 暗示语义 |
| **C2** | 新增 `--search-space` flag | `framework/sweep.py` | 读取 `domains/gomoku/manifest/search_space.json` 这类 profile；`--dry-run` 时打印 schema 摘要 | CLI 与 domain profile 正式打通 |
| **C3** | campaign 创建 / 查找逻辑 | `framework/sweep.py` + `db.py` | 执行时若 campaign 不存在则创建；存在则校验 protocol / search_space 是否兼容 | 同名 campaign 不会悄悄混入异质 runs |
| **C4** | run linkage 入库 | `framework/sweep.py` + `db.py` | 每个成功启动的 run 都写 `campaign_runs`，保存 `seed` 与 `axis_values_json` | 后续 analyze 可精确回看 campaign 组成 |
| **C5** | campaign-aware resume | `framework/sweep.py` | `--resume` 在 campaign 语义下工作：跳过同一 campaign 内已完成组合，而不是全库模糊跳过 | 同轮研究恢复执行时更安全 |
| **C6** | protocol lock | `framework/sweep.py` | 若同一 campaign 下出现不兼容的 `eval_level` / `eval_opponent` / `train_script` / `is_benchmark`，直接中止并报错 | comparability 纪律从“人类自觉”升级成“框架约束” |

### 6.4 D 系 — analyze.py 的 campaign summary 与 guard

| # | 工作包 | 文件 | 产出 / 可验收 | 预期收益 |
|---|---|---|---|---|
| **D1** | 新增 `--list-campaigns` | `framework/analyze.py` | 列出 campaign id/name/domain/status/run_count/created_at | 当前研究面可一眼浏览 |
| **D2** | 新增 `--campaign-summary <name|id>` | `framework/analyze.py` | 输出 campaign 的 protocol、search-space profile 摘要、run 数、状态分布、轴覆盖范围、frontier 快照数量 | v20.1 的核心分析入口 |
| **D3** | Pareto / matrix 的 campaign filter | `framework/analyze.py` | 支持 `--campaign <name|id>`，只对该 campaign 的 runs 做 `--pareto` / `--matrix` | v20 的 front 工具进入 campaign 时代 |
| **D4** | protocol drift 提示 / 拒绝 | `framework/analyze.py` | 当 campaign 内存在 drift runs 时，summary 明确标注；`--pareto --campaign` 默认拒绝混入 drift runs，要求显式 override | v20.1 在分析层落实 comparability guard |

### 6.5 E 系 — Gomoku 冷启动 smoke campaign（收口证据）

| # | 工作包 | 文件 | 产出 / 可验收 | 预期收益 |
|---|---|---|---|---|
| **E1** | dry-run 预览 | 无代码改动 | 用 `--campaign gomoku-v20.1-smoke --search-space domains/gomoku/manifest/search_space.json --dry-run` 打印规范化矩阵与 profile 摘要 | 验证 schema 与 CLI 一致 |
| **E2** | 最小实跑 campaign | 无代码改动 | 至少 4 个短预算配置成功入库到同一 campaign | v20.1 不是纸面设计，而是有真实数据库证据 |
| **E3** | campaign summary 验证 | 无代码改动 | `analyze.py --campaign-summary gomoku-v20.1-smoke` 输出 run_count / protocol / axis coverage / frontier snapshot 数 | 节点闭环有可复现命令 |

## 7. v20.1 的 in-scope / out-of-scope

### 7.1 In-scope

下面这些内容属于 v20.1 正式范围：

1. Search-space profile 的**格式定义**
2. Search-space profile 的**加载、校验、规范化**
3. 数据库里的 `campaigns` / `campaign_runs` / `search_spaces`
4. `sweep.py` 的 `--campaign` / `--search-space` / protocol lock
5. `analyze.py` 的 campaign list / summary / campaign filter
6. 一轮最小 gomoku smoke campaign 作为真实证据

### 7.2 Out-of-scope

下面这些事情**明确不在 v20.1**：

1. **Stage A/B/C/D 的预算晋升引擎** → v20.2
2. **seed 方差驱动的 promotion 决策** → v20.2
3. **checkpoint continuation / branch ledger** → v20.3
4. **trajectory report / branch tree** → v20.3
5. **next-point / next-branch recommendation** → v21
6. **Bayesian / MOBO / EHVI** → v21.1
7. **telemetry API / websocket / Web dashboard** → 后续独立主线
8. **长时间 gomoku 实验结论** → findings 文档，不是 v20.1 发布门槛

### 7.3 一个特别重要的边界

v20.1 可以在 profile 里标注：

- `role = slow`
- `allow_continuation = true`

但 **v20.1 只负责记录这些语义，不负责执行 continuation**。

也就是说：

> **v20.1 可以知道“这个轴未来应该通过 continuation 看”，但不能真的去分叉 checkpoint。**

这条边界必须守住。

## 8. 测试要求（强化版）

这一版的测试要求要比 v20 严格得多。  
因为 v20.1 引入的是**研究治理层**，一旦 schema / campaign 逻辑错了，后面的所有 frontier、promotion、trajectory 都会建立在脏数据上。

### 8.1 自动化测试必须新增三组

| 测试文件 | 重点 | 最低覆盖要求 |
|---|---|---|
| `tests/test_search_space.py` | profile 解析、校验、负例 | 至少覆盖 8 个 case |
| `tests/test_campaign_db.py` | schema 迁移、入库、幂等、去重 | 至少覆盖 8 个 case |
| `tests/test_campaign_cli.py` | `sweep.py` / `analyze.py` CLI 行为与 protocol guard | 至少覆盖 8 个 case |

### 8.2 `test_search_space.py` 必测项

1. 合法 gomoku profile 可加载
2. 缺 `domain` / `axes` / `protocol` 任一字段时报错
3. axis `type` 非法时报错
4. `scale=log` 但 range 非正时报错
5. `role` 非 `structure/training/slow` 时报错
6. `allow_continuation` 缺失时按默认值处理是否正确
7. `describe_profile()` 输出稳定且包含 domain / protocol / roles
8. 相同 profile hash 稳定一致

### 8.3 `test_campaign_db.py` 必测项

1. DB 初始化后 3 张新表存在
2. 重复初始化 DB 不报错
3. 相同 profile hash 不重复写入 `search_spaces`
4. 新 campaign 可创建并正确关联 `search_space_id`
5. `campaign_runs` 写入成功
6. 重复 `(campaign_id, run_id)` 被拒绝
7. `axis_values_json` round-trip 正确
8. protocol_json round-trip 正确

### 8.4 `test_campaign_cli.py` 必测项

1. `sweep.py --campaign ... --search-space ... --dry-run` 正确打印规范化矩阵
2. 同名同 protocol campaign 可 resume
3. 同名不同 protocol campaign 被拒绝
4. `analyze.py --list-campaigns` 输出 campaign 行
5. `analyze.py --campaign-summary X` 输出 search-space 与 protocol 摘要
6. `analyze.py --pareto --campaign X` 只分析该 campaign runs
7. campaign 内存在 drift runs 时给出拒绝或明确警告
8. 空 campaign / 不存在 campaign 时给出友好提示

### 8.5 现有测试的基线要求

v20.1 开工前，必须先跑当前已有测试作为 baseline。  
v20.1 完成后，必须重新跑**全量**测试：

```bash
python3 -m unittest discover tests -v
```

要求：

1. 旧测试全部继续通过
2. 新测试全部通过
3. 不允许只跑新增测试就宣布完成

### 8.6 命令级 smoke test（必须有真实证据）

除了自动化测试，v20.1 还必须有一次真实 gomoku smoke campaign：

  --train-script domains/gomoku/train.py \

## 13. 收口方式与标准

v20.1 的收口不是“代码写完”，而是下面这三类证据同时成立：

### 13.1 代码证据

- 新表存在
- 新 CLI 存在
- 新 loader / validator 存在

### 13.2 自动化证据

- baseline 旧测试通过
- 新增三组测试通过
- protocol drift / invalid profile 等负例被明确覆盖

### 13.3 真实运行证据

- 至少 1 个 gomoku smoke campaign 真正写进 DB
- `campaign-summary` 能对它做结构化解释
- 无需人工在多个 `sweep_tag` 之间拼凑“这轮研究是什么”

如果要把一句话写成标准：

> **v20.1 完成的标志，不是“又多了几个 CLI 参数”，而是“系统第一次能把一轮参数探索正式命名、正式入库、正式解释，并且对不可比的 runs 说不”。**

---

## 14. 一句话结论

> **v20.1 不是为了让系统更聪明地选点，而是为了让系统第一次真正知道“自己在做哪一轮研究”。这一步完成后，v20.2 才有资格把这些 campaign 继续推进成 Stage A/B/C/D 的多预算晋升链。**

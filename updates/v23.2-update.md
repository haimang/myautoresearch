## 1. v23.2 一句话目标

> **把 v23.1 已经建立起来的 facade / services / core 骨架继续做实：继续拆分剩余巨石，继续下沉剩余混合职责，继续减少 framework 顶层扁平文件数量，让“代码放哪里”这件事在 framework 内部更稳定、更直观。**

v23.2 不做：

1. 不新增新的 research domain。
2. 不引入新的真实世界 provider 接入。
3. 不把重点转回 Gomoku domain。
4. 不推翻 v23.1 已经建立的兼容层策略。
5. 不为了目录好看而制造新的空抽象。

v23.2 要做：

1. 继续拆 `framework/facade/analyze_cli.py`。
2. 继续拆 `framework/facade/sweep_cli.py`、`branch_cli.py`、`promote_cli.py`。
3. 继续拆 `framework/services/research/selector_service.py`。
4. 继续拆 `framework/core/db/repo_recommendations.py`。
5. 把 policy / profile / plotting 相关文件迁到更明确的目录。
6. 减少 `framework/` 顶层非-wrapper 文件数量。
7. 在兼容旧 import / 旧命令的前提下，进一步明确文件落点。

## 2. 为什么 v23.2 是必要的

v23.1 已经证明第一轮结构收敛是有效的：

1. `core.db` 已 package 化。
2. 顶层 CLI 已退化为 wrapper。
3. Bayesian generic machinery 已回 framework。
4. 全量测试保持通过。

但 v23.1 的收口报告也同时明确指出：

1. `framework/facade/analyze_cli.py` 仍然非常大。
2. `framework/facade/branch_cli.py` / `sweep_cli.py` / `promote_cli.py` 仍然混合 CLI 与大量 planning / execution 逻辑。
3. `framework/services/research/selector_service.py` 仍然是新的次级巨石。
4. `framework/pareto_plot.py`、`stage_policy.py`、`branch_policy.py`、`selector_policy.py`、`search_space.py`、`objective_profile.py` 仍平铺在 framework 根目录。
5. `framework/core/db/repo_recommendations.py` 已开始变成 db package 内的新热点文件。

也就是说，v23.1 解决的是：

```text
先把边界拉出来
先把骨架立起来
先把最大的问题止血
```

而 v23.2 要解决的是：

```text
把骨架真正做实
把次级巨石继续打散
把目录语义继续收紧
```

所以 v23.2 不是 v23.1 的附录，而是：

> **在 v23.1 的基础上继续推进 framework 内部结构的“第二层拆分”。**

## 5. v23.2 结构原则

v23.2 必须遵守下面 7 条原则：

1. **继续渐进迁移，不做大爆炸**
   - 仍然保留旧 import / 旧 CLI 路径
   - 用 shim / wrapper 继续过渡

2. **继续减少 root 中“真实逻辑文件”的数量**
   - root 保留 wrapper 没问题
   - 真实逻辑应继续向 `facade/`、`services/`、`policies/`、`profiles/` 收敛

3. **facade 只做 CLI / dispatch，不再承载重业务**
   - v23.2 的本质就是继续把 facade 中仍然过重的逻辑剥离出去

4. **service 只按职责聚合，不按“方便复制代码”聚合**
   - 例如：
     - frontier plotting / export 在一起
     - recommendation selection / scoring / batch write 分开

5. **policy / profile / plotting 必须有明确目录语义**
   - 不再继续平铺在 framework 根目录

6. **db package 允许二次拆分**
   - package 化不是终局
   - 某个 repo 模块重新长大后，应继续拆

7. **每一步都以 v23.1 的测试基线为硬约束**
   - 当前稳定基线：`187 passed`
   - v23.2 不能以“结构更漂亮”为代价破坏兼容

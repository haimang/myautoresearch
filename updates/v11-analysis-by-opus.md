## 1. 对 autoresearch 框架有效性的判定

### 1.1 框架做对了什么

从 gomoku_1st_exp.db 的数据中可以提取以下事实：

**事实 A：框架的实验追踪能力完全成立。**

49 个 run（27 个 completed）、73 个 checkpoint、10,240 条 cycle_metrics、8,200 条 recording 记录。每个 run 都有完整的超参快照（num_res_blocks、num_filters、learning_rate、parallel_games、train_steps_per_cycle、seed、time_budget、sweep_tag）。这不是 toy demo——这是一个真正能支撑研究决策的数据仓库。

**事实 B：框架的 sweep 机制产生了最高质量的 Pareto 数据。**

benchmark run（`is_benchmark=1`、统一 120s 时间预算）只占约 10 个 run，但产生了最有决策价值的对比数据。最关键的发现——8x64 是当前变量空间中最高效的架构——来自这些标准化短 run。

**事实 C：框架的 report 系统为 agent 闭环提供了足够的信号。**

`analyze.py --report --format json` 输出的结构化报告能让 agent 在不理解 Gomoku 的情况下消费实验结果。这是 autoresearch 闭环的关键基础设施。

### 1.2 框架暴露了什么问题

**问题 A：框架无法检测训练信号本身的结构性缺陷。**

这是最重要的一条。27 个 completed run 中，有 5 个 run 超过 5,000 盘训练量：

| run | 架构 | 训练量 | 最终 WR | 对手 | 墙钟 |
|-----|------|--------|---------|------|------|
| 374d567f | 6x64 | 5,008 | 54.5% | mm-L0 | 1145s |
| 92d1740a | 10x64 | 5,056 | 84.0% | mm-L0 | 436s |
| 2b1d2f03 | 10x64 | 10,048 | 81.0% | NN-L1 | 885s |
| b3f99d4f | 10x64 | 72,000 | 75.0% | NN-L2 | 5980s |
| 8b9486f4 | 12x64 | 54,000 | 78.0% | NN-L0 | 4243s |
| 2f547f8d | 10x64 | 30,000 | 80.8% | NN-L3 | 2385s |

这 6 个 run 合计消耗了 176,112 盘（87% 的总训练量），但对 WR 前沿的推进几乎为零。b3f99d4f 的 cycle_metrics 数据尤其触目惊心：

- cycle 225 (14,400 盘)：WR 67%
- cycle 1125 (72,000 盘)：WR 75%

**后 57,600 盘只贡献了 +8% WR，而且 WR 波动极大（从 57% 到 88% 随机震荡）。** 这不是训练不够，而是 gomoku-real-training.md 准确诊断的"噪声中学习噪声"问题。

框架忠实记录了这些数据，但它自身没有能力发现"这些 WR 数字在棋力层面是虚假的"——因为 WR 确实在涨，只是涨的方式是对手犯错。**框架能发现 Pareto，但不能发现 Pareto 本身是否建立在虚假指标之上。**

**问题 B：agent 对手链 L0→L1→L2→L3→L4 的晋升逻辑掩盖了停滞。**

从 opponents 表可以看到：

| alias | 来源 | WR (vs mm-L0) | 说明 |
|-------|------|---------------|------|
| L0 | 374d567f | 67.5% | 5000-game baseline |
| L1 | 92d1740a | 84.0% | 10x64, vs L0 minimax |
| L2 | 2b1d2f03 | 83.0% | 10x64, checkpoint from vs L1 |
| L3 | b3f99d4f | 75.0% | 10x64, 72000 games vs L2 |
| L4 | 2f547f8d | 83.6% | 10x64, vs L3 |

L2 的 WR (83%) 甚至低于 L1 (84%)。L3 更是降到 75%。这意味着所谓的"阶段晋升"并没有让模型棋力单调递增——**对手链是虚假的进步阶梯。** 每一代对手只是一个稍微不同的弱模型，而不是一个真正更强的对手。

框架记录了这些数据，但 agent 的 program.md 中设定的 stage promotion 规则（L0→L1→L2→L3）让 agent 产生了"在不断进步"的错觉。**框架基础设施是合格的，但 agent 的研究协议存在误导性设计。**

### 1.3 最终判定：框架有效性

> **autoresearch 框架作为实验基础设施是有效的——它的追踪、sweep、report 能力经受住了 201,728 盘真实训练的检验。但框架有效 ≠ 研究有效。框架忠实记录了一个结构性失败的训练过程，而没有提前暴露信号缺陷。这不是框架的设计失败，而是框架能力边界的一个重要标记：它能发现"在给定空间内的最优"，但不能发现"给定空间本身是否有意义"。**

对框架演进的启示：

1. **框架应增加"停滞检测"能力**——cycle_metrics 表中的 WR 时间序列如果连续 N 个 checkpoint 无趋势改善（±2%），应自动发出警告。这是 domain-agnostic 的纯数值操作。从 b3f99d4f 的数据看，cycle 225 之后就应该触发停滞警告。
2. **框架应增加"效率衰减检测"能力**——当 WR/1000盘 从早期的 4.43（6bb72701）降到后期的 0.01（b3f99d4f），框架应标记"边际回报急剧递减"。
3. **框架不应试图判断"训练信号是否正确"**——这是 domain 层的责任，不是框架的职责范围。

## 2. 对 Pareto frontier 有效性的判定

### 2.1 数据确认：Pareto 前沿确实存在且可提取

从 completed runs 数据中，可以在多个维度上清晰提取非支配曲线。

**维度 1：架构效率前沿（params vs WR，vs mm-L0）**

| 架构 | 参数量 | 最佳 WR | 最佳 run |
|------|--------|---------|---------|
| 6x32 | 229K | 53.0% | 556f91a6 |
| 6x64 | 564K | 71.0% | 5ca93d8f |
| 8x64 | 713K | 85.0% | 6bb72701 |
| 10x64 | 862K | 85.5% | f166d561 |
| 6x96 | 1,120K | 20.0% | 2dff50ac |
| 12x64 | 1,011K | 78.0% | 8b9486f4 |

Pareto 前沿：`6x32 (229K, 53%) → 6x64 (564K, 71%) → 8x64 (713K, 85%)`。

10x64 虽然 WR 略高 (+0.5%)，但参数量 +21%，处于前沿的边缘——如果考虑方差，它可能不是前沿点。6x96 和 12x64 被严格支配。

**维度 2：训练效率前沿（games vs WR，vs mm-L0）**

| run | 架构 | 训练局 | WR | WR/千局 |
|-----|------|--------|-----|---------|
| 608fd768 | 6x64 | 128 | 70.0% | 5.47 |
| 6bb72701 | 8x64 | 192 | 85.0% | 4.43 |
| b4f8e1bc | 8x64 | 640 | 93.3% | 1.46 |

前 640 盘是高效区。之后样本效率急剧衰减，说明当前训练信号的信息密度在几百盘后就接近耗尽。

**维度 3：时间效率前沿（wall_time vs WR，benchmark runs only）**

| 墙钟 | WR | 架构 | 配置 |
|------|-----|------|------|
| 20s | 62.0% | 6x64 | LR=3e-4, spc=20, pg=16 |
| 36s | 71.0% | 6x64 | LR=5e-4, spc=30, pg=16 |
| 120s | 85.5% | 10x64 | LR=7e-4, spc=30, pg=64 |
| 180s | 80.5% | 8x64 | LR=5e-4, spc=30, pg=16 |

这里有一个反直觉的发现：120s 的 10x64 比 180s 的 8x64 WR 更高。原因是 10x64 使用了 64 并行对局（每 cycle 训练量更大），而 8x64 只用了 16 并行对局。**parallel_games 是一个被低估的效率变量。**

### 2.2 Pareto 分析的真正价值在哪

从数据中，Pareto 分析产生了三个 agent 靠单看 WR 无法得出的关键洞察：

1. **"更大不等于更好"的量化证据。** 12x64 花了 54,000 盘、4243 秒，WR 78%。8x64 花了 192 盘、18 秒，WR 85%。如果只看 WR，12x64 的 78% 看起来不差；但放在 (params, time, WR) 空间中，它被严格支配。

2. **"高方差配置"的风险标记。** 6x64/LR=5e-4 的 6 个 run WR 从 37% 到 71%，方差极大。如果 agent 只看到最佳 run 的 71%，它会认为这是个好配置；Pareto + 方差分析会告诉它"这个配置不可靠"。

3. **"parallel_games 是关键效率杠杆"。** 8x64/pg=64/120s 的 WR (69.5%) 远低于 8x64/pg=16/180s 的 WR (80.5%)，但 10x64/pg=64/120s 达到了 85.5%。这说明 pg 和架构之间存在交互效应——大模型更能利用大 batch，小模型反而被大 batch 稀释了。这种交互效应只有在 Pareto 多维视图中才能看到。

### 2.3 Pareto 分析的局限

**局限 A：在虚假指标上的 Pareto 仍然是虚假的。**

所有以上 Pareto 分析都基于 WR 这个指标。而 gomoku-real-training.md 已经论证了当前 WR 不反映真实棋力——模型在评估中"赢"的很多对局，其实是对手随机犯错。**在 WR 上的 Pareto 前沿，本质上是"在给定时间/参数预算内最高效地利用对手随机犯错"的前沿。**

这不意味着 Pareto 分析没有用——它有用，因为即使指标有偏，不同配置在同一指标上的相对排序仍然有参考价值。但它确实意味着：**Pareto 层的价值上限，取决于真理层指标的质量。**

**局限 B：当前数据的重复不够，大部分 Pareto 点只有 1 个 run 支撑。**

49 个 run 中，真正有可控重复实验的配置寥寥无几。6x64/LR=5e-4 有 6 个 run，但其他配置大多只有 1-2 个 run。**单 run 的 WR 可能是运气，不是真实水平。** pareto-frontier.md 中提出的"自适应采样"机制是正确的方向。

### 2.4 最终判定：Pareto 有效性

> **Pareto frontier 分析在当前数据上完全可行，且确实产生了超越单一 WR 排序的决策价值。但它的上限受制于真理指标的质量——如果底层 WR 不反映真实棋力，那么 Pareto 前沿只是"在虚假空间内的相对最优"。框架应先推进 Pareto 的工程实现（非支配排序、方差追踪、停滞检测），但同时必须意识到：修复训练信号比优化 Pareto 展示更紧急。**

## 3. 对 MCTS 路线的冷静分析

### 3.1 gomoku-real-training.md 的诊断是正确的

回到原始数据验证这个诊断：

b3f99d4f（10x64, vs NN-L2, 72,000 盘）的 WR 曲线：

```
cycle 15  (960盘):   34% → 快速上升
cycle 60  (3840盘):  63% → 开始震荡
cycle 225 (14400盘): 67% → 进入停滞平台
cycle 870 (55680盘): 85% → 偶尔冲高
cycle 975 (62400盘): 63% → 随即崩回
cycle 1125(72000盘): 75% → 最终值
```

WR 在 cycle 60 之后就进入了 ±15% 的剧烈震荡，完全没有收敛趋势。72,000 盘训练，WR 曲线看起来像噪声而不是学习曲线。**这与"模型在学习复制自己的随机行为"的诊断完全一致。**

再看 8b9486f4（12x64, vs NN-L0, 54,000 盘）：

```
cycle 15  (1200盘):  76.7%
cycle 300 (24000盘): 70.0%  ← WR 在下降
cycle 600 (48000盘): 78.3%
cycle 675 (54000盘): 78.0%  ← 比 cycle 15 仅高 1.3%
```

12x64 花了 54,000 盘，WR 基本没有离开 cycle 15 的水平。**更大的模型不但没有学到更多，反而在前 24,000 盘中出现了 WR 下降。**

这两组数据都指向同一结论：**当前的训练信号（自对弈 softmax 采样 → policy target）的信息密度在几百盘后就耗尽了。之后的训练只是在噪声中打转。**

### 3.2 MCTS 是解决方案，但成本不可低估

gomoku-real-training.md 估算 MCTS 会使训练慢 30-50 倍（200 sims/步 × 50 步/盘 = 10,000 次前向传播/盘，vs 当前约 50 次/盘）。即使用批量叶子评估优化 10-20x，每盘仍然慢 2-3 倍。

但这个估算是基于 M3 Max 的。**当前我们在 Linux 开发机上，没有 MLX/Metal。** 这意味着：

1. 在开发机上无法运行 MCTS 训练验证
2. MCTS 实现必须在 MLX 框架内完成，才能利用 Apple Silicon 的批量推理能力
3. 纯 CPU 上跑 MCTS 是不现实的——200 sims/步的推理代价在 CPU 上会是秒级，一盘棋要几分钟

### 3.3 MCTS 在 MLX 框架下的实现路径分析

**关键技术挑战：**

1. **树搜索是天然串行的。** MCTS 的 select → expand → backup 路径依赖上一次搜索的统计量更新。批量化需要"虚拟损失"（virtual loss）机制来允许并发搜索路径，这增加了实现复杂度。

2. **MLX 的 lazy evaluation 与 MCTS 的交互。** MLX 使用延迟求值——`mx.eval()` 需要显式调用才能触发实际计算。在 MCTS 中，每次叶子节点的 expand 都需要一次前向传播，如果不做 batch 合并，MLX 的延迟求值优势无法发挥。需要设计一个"叶子节点队列"机制：攒够 N 个待评估叶子节点后一次性批量推理。

3. **内存管理。** MCTS 树会消耗大量内存。每个节点需要存储：board state、visit count、value sum、prior probability、children 指针。对于 15x15 棋盘、200 sims 的搜索深度，单步搜索树可能包含 200-1000 个节点。如果 64 盘并行，内存占用可能达到数 GB 级。需要在每步走棋后释放搜索树。

4. **Metal buffer 限制。** caveats.md 记录了 MLX 的 BatchNorm train 模式会泄漏 Metal buffer。MCTS 中每步 200 次前向传播 × 50 步/盘 = 10,000 次前向传播/盘，即使在 eval 模式下，也需要关注 `mx.clear_cache()` 的频率。

**最小可行实现方案：**

Phase 1 的最小 MCTS 可以这样设计：

```
1. 单线程、逐个 sim 的 MCTS（无虚拟损失、无批量优化）
2. --mcts-sims 50（先验证搜索结果合理性）
3. --parallel-games 8（降低内存压力）
4. 每步走棋后显式释放搜索树 + mx.clear_cache()
```

预估这个最小版本的实现量：~200 行 Python（MCTSNode + mcts_search + 集成到 run_self_play）。

预估训练速度：8 盘并行 × 50 sims × ~0.1ms/forward ≈ 每步 40ms → 每盘 2s → 每 cycle（8 盘）16s。比当前慢约 8x，但信号质量应有数量级提升。

### 3.4 Phase 1（战术教师）的价值重新评估

gomoku-real-training.md 提出的 Phase 1 战术教师方案成本很低（~100 行），但我认为它的价值被高估了。原因：

1. **战术教师只修复了"一步必胜/必堵"场景的 policy target。** 而这些场景在 50 步棋局中只占 2-3 步。其余 47 步的 policy target 仍然是垃圾信号。
2. **value signal 完全没有改善。** 终局胜负作为全局 value target 的问题不会因为战术教师而缓解。
3. **它可能让 WR 数字好看，但不解决根本问题。** 模型学会了堵连五之后，WR 会上升——但这只是因为对手（minimax 或弱 NN）在不堵的情况下输得更快。中盘能力仍然是零。

**我的判断：战术教师作为独立 Phase 没有足够的 ROI。如果实现 MCTS，战术教师变得不必要；如果不实现 MCTS，战术教师只是给虚假 WR 再加一层膨胀。** 如果要做，建议把它作为 MCTS 实现的附带产物——在 MCTS 搜索中，一步必胜/必堵自然会被搜索发现，不需要硬编码。

### 3.5 最终判定：MCTS 路线

> **MCTS 是训练信号从"噪声中学噪声"升级到"从搜索结果中学习"的唯一已知路径。从 AlphaZero 论文到当前数据的停滞曲线，证据链完整且一致。但实现路径必须考虑 MLX 框架的特殊性（lazy evaluation、Metal buffer 限制），且当前 Linux 开发机无法验证。建议：**
>
> 1. **在开发机上完成 MCTS 核心逻辑的纯 Python 实现和单元测试**（不依赖 MLX，用 numpy mock 前向传播）
> 2. **设计好 MLX 批量推理的接口**（叶子节点队列 → batch forward → 分发结果）
> 3. **部署到 Apple Silicon 后做最小可行验证**（50 sims × 8 盘 × 120s）
> 4. **跳过战术教师作为独立 phase**——如果 MCTS 有效，战术教师是冗余的；如果 MCTS 无效，战术教师也救不了

## 4. 对未来演进的分析

### 4.1 当前项目的真实状态

剥掉文档中的愿景叙述，用数据说话：

| 事项 | 状态 |
|------|------|
| 实验追踪基础设施 | **成熟** — tracker.py + analyze.py 经过 201K 盘验证 |
| 超参 sweep 能力 | **可用** — sweep.py 已实现穷举探索 |
| Pareto 分析 | **概念验证** — 数据证明可行，但尚无正式代码实现 |
| 训练信号质量 | **有结构性缺陷** — 纯 policy 自对弈已被数据证明存在天花板 |
| MCTS | **零实现** — train.py 中 MCTS_SIMULATIONS=0，没有 MCTS 代码 |
| 对手进化链 | **有效但虚假** — L0-L4 每代棋力没有单调递增 |
| 框架解耦 | **已完成** — framework/ 和 domains/gomoku/ 的拆分已落地 |
| 人类可展示材料 | **丰富** — 8,200 条 recording，73 个 checkpoint，replay 系统完备 |

### 4.2 v11-analysis.md 的判断哪些成立，哪些需要修正

**成立的判断：**

1. "Gomoku 既是目标本身，也是框架验证场" — 这是对的。数据证明 Gomoku 确实同时产生了作品价值（checkpoint/replay）和框架验证价值（Pareto 数据）。
2. "三条红线" — 完全成立且被数据支撑。特别是红线一（单一裁决纪律），当前 WR 作为真理指标已经出现了被虚假膨胀的迹象。
3. "v11 的 in-scope 应严格限制在 Gomoku 域内" — 正确。跨领域扩展在 Gomoku 的训练信号问题解决之前没有意义。

**需要修正的判断：**

1. **"继续追求高水准 Gomoku 模型"——需要重新定义"高水准"的含义。** 在当前训练信号下，WR 数字可以继续涨，但棋力不会真正提升。如果 v11 的目标仍然是"更高 WR"，它会重蹈 b3f99d4f 的覆辙——花 72,000 盘换 WR 从 67% 到 75%。**v11 的真正目标应该是"让 WR 反映真实棋力"，而不是"让 WR 数字更大"。**

2. **"Pareto Frontier 分析正式化"——优先级应低于训练信号修复。** Pareto 在虚假指标上的精度越高，误导性越强。**正确的顺序是：先修复 WR 的真实性（MCTS），再在真实 WR 上做 Pareto。**

3. **"agent 消费 Pareto 结果的最小闭环验证"——当前阶段不应追求。** 在训练信号修复之前，agent 消费的 Pareto 结果是建立在虚假 WR 上的。让 agent 基于虚假 Pareto 做研究决策，会进一步强化错误路线。

### 4.3 我建议的演进路径

基于数据，我建议一条比现有文档更收敛的路径：

**阶段 1：修复训练信号（MCTS 最小可行实现）**

目标：让 policy target 从"模型自己的采样分布"变成"MCTS 搜索后的访问计数分布"。

在开发机上的可做工作：
- MCTSNode + mcts_search() 的纯 Python 实现（不依赖 MLX）
- 基于 numpy 的 mock forward pass 单测：验证搜索树的 visit count 分布是否合理
- train.py 中 `run_self_play()` 的 MCTS 集成点设计

在 Apple Silicon 上的验证：
- `--mcts-sims 50 --parallel-games 8 --time-budget 300`
- 对比同条件下纯 policy 的 WR 曲线形态（是否收敛而不是震荡）
- 战术基准测试：50 个一步必胜/必堵局面的正确率

**成功标准：** MCTS 训练的模型在 300s 内 WR 曲线呈单调上升趋势（而不是像 b3f99d4f 那样震荡），且战术基准正确率 > 80%。

**阶段 2：在真实 WR 上重建 Pareto 前沿**

只有当阶段 1 证明 MCTS 训练的 WR 反映真实棋力之后，才有意义做 Pareto 分析正式化：

- 在 analyze.py 中实现非支配排序
- benchmark run Pareto：统一条件下的 (WR, params, time) 前沿
- threshold Pareto：不同配置达到固定 WR 阈值的代价
- 方差追踪：每个 Pareto 点附带置信区间

**阶段 3：重建对手进化链**

基于 MCTS 训练的模型重建 L0→S1→S2→... 链，每一代对手应能通过战术基准测试，且 WR 单调递增。

### 4.4 关于 autoresearch 原旨精神

autoresearch 的核心理念是：**让 AI agent 成为研究的主体，代码/系统成为被研究的客体。**

从这个角度看，当前项目最有价值的资产不是任何一个模型的 WR 数字，而是：

1. **一套经过 201K 盘真实数据验证的实验基础设施**（tracker + analyze + sweep）
2. **一份完整的"研究失败记录"**（对手链虚假进步、训练信号结构性缺陷的诊断）
3. **一个清晰的"从失败到修复"的路线图**（MCTS）

在 autoresearch 的精神下，这些失败不是浪费——它们是框架价值的最好证明。正是因为框架忠实记录了一切，我们才能在事后精确诊断出问题所在。**一个好的实验室不是只产出成功实验的实验室，而是能让你精确知道为什么失败的实验室。**

v11 的真正任务，不是"让 Gomoku WR 更高"或"让 Pareto 分析更漂亮"，而是：

> **用 MCTS 修复训练信号这一结构性缺陷，然后用框架的追踪和 Pareto 能力验证修复是否成功。这才是"真实目标 + 框架验证场"的正确交汇方式：目标是训出真正会下棋的模型，验证的是框架能否在修复前后给出清晰的对比证据。**

## 5. 总结

| 问题 | 判定 |
|------|------|
| autoresearch 框架是否有效？ | **是。** 追踪、sweep、report 能力经受住了 201K 盘真实验证。但框架不能检测训练信号本身的结构性缺陷——这是它的能力边界，不是设计失败。 |
| Pareto frontier 是否可行？ | **是。** 数据在架构、效率、时间三个维度上都存在清晰的非支配前沿。但在虚假 WR 上的 Pareto 只有相对参考价值，不是绝对判据。 |
| 是否应该继续实现 MCTS？ | **是，且优先级最高。** 训练信号的结构性缺陷是当前项目的一阶问题。Pareto 优化、对手进化链、agent 闭环验证——这些都是二阶问题。先解决一阶问题。 |
| v11 应该做什么？ | **修复训练信号（MCTS 最小实现）→ 在真实 WR 上重建 Pareto → 重建对手进化链。** 不要在虚假指标上继续堆积分析层。 |

一句话：

> **201,728 盘棋局证明了框架能力，也暴露了训练信号缺陷。下一步不是让框架做更多事，而是让训练信号产生真正的学习信号——MCTS 是唯一已知路径。框架的价值将在修复前后的对比中得到最充分的验证。**
